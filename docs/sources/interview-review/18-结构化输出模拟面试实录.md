## 面试实录

### 第 1 轮 · 编译缓存的容量口径（FIFO/100 vs 按字节）

**面试官：** 你好，我是负责推理引擎方向的面试官，今天想把你 MindIE 的结构化输出这条线挖到底。第一个问题：你的编译缓存用 SHA-256 做 key——代码里默认容量多少、淘汰是 LRU 还是 FIFO？为什么是"按条数"而不是像 vLLM 那样按字节上限？单条编译产物大小量级？最坏内存上界多少？

**面试者：** 先对齐代码真相，避免被翻仓：默认容量 **100**（`grammar_cache_size`，`_DEFAULT_GRAMMAR_CACHE_SIZE=100`），淘汰是 **FIFO**——普通 `dict`，超限时 `next(iter())` 删最早插入项，**命中不会 move-to-end**，所以严格说不是 LRU。简历若曾写 LRU/128，那是早期口述口径，以代码为准。

为什么选 FIFO：目标场景是 tool-calling／Agent，schema 种类通常个位数到几十个、集合稳定近「全热」，FIFO 与 LRU 命中率差异小，实现最简（不用 OrderedDict / 链表）。100 条相对常见 schema 种类留了几倍余量，经验命中率约 85%–95%。key 用规范化 schema 串的 SHA-256，二次请求零编译。

单条产物量级：PDA + adaptive mask cache，扁平 schema 几百 KB，深嵌套多 anyOf 可到几 MB。

内存上界隐患在「按条数」：100 × 几百 KB ≈ 几十 MB 可控；若 100 条全是 ~5MB 深嵌套，≈500MB，可能逼近甚至超过 vLLM 默认 512MB 字节上限。根因是条数不约束单条大小方差。自我迭代：改成 vLLM 式字节上限，或 **条数 + 字节双门限**（例如 `max_bytes=512MB AND max_entries=256`）。

### 第 2 轮 · 后端选型：xgrammar vs Outlines vs llguidance

**面试官：** 回答合格——你主动把 FIFO/100 和代码对齐了，也算清了按条数的失控上界，这点我满意。但"双门限兜底"先记下阈值。下一个问题，别跟我说"xgrammar 快"：为什么选 xgrammar 而不是 Outlines 或 llguidance？请给三组可量化对比——编译期耗时、每步 μs 级开销、对 JSON 递归嵌套的表达能力，然后告诉我：在 MindIE 跑 NPU、要靠 C++ 内核可移植性的这个具体环境里，压死骆驼、让你排除掉 Outlines 和 llguidance 的那个**决定性**因素到底是哪一个？

**面试者：** 先给三组硬对比。

编译期：xgrammar 复杂 schema 100–200ms、简单的 5–15ms；Outlines 正则→FSM，复杂 schema 曾到分钟级（core 用 Rust 重写后改善），是它最大的痛点；llguidance 走 Earley 惰性解析，没有重编译负担但把成本摊到每步。

每步 μs：xgrammar 20–80μs（99% 查表＋<1% 现场检查）；Outlines FSM 纯查表 10–20μs，最快；llguidance ~50μs。单看每步 Outlines 略胜，但差距在同一量级、会被 GPU/NPU 前向淹没。

递归表达：这是分水岭。xgrammar 字节级 PDA 带栈，原生表达任意嵌套 JSON Schema；Outlines 本质是 FSM，递归结构要展开近似，深层嵌套或无界递归表达不了，对"任意 JSON Schema"这个刚需是硬伤；llguidance 是完整 CFG，表达最灵活，这项和 xgrammar 打平。

决定性因素——落到 MindIE 跑 NPU、要 C++ 内核可移植这个具体环境：是**架构解耦与可移植性**。xgrammar 核心是 C++、bitmask 与后端算子解耦，我只需在 NPU 侧接一条 `masked_fill_(-inf)` 路径就能落地；Outlines/llguidance 早期跟 CUDA/Python 生态耦合更紧，迁到 NPU 成本高。Outlines 还叠加了递归要展开近似这条硬伤，直接出局；llguidance 表达够但可移植性不占优。

坦白讲，这也是跟随 vLLM/SGLang 主流选型的结果——它们默认 xgrammar，我在 NPU 上复用其 C++ 内核，风险最低。

### 第 3 轮 · 费米估算：mask 开销占 TPOT 的比例，batch 放大后是否成立

**面试官：** 数据摆得不错，但"跟随 vLLM/SGLang 主流"这半句划掉——那是背书不是论证，别在我这儿用；可移植性也别当万能挡箭牌，回头我会追你 NPU 上 masked_fill_ 到底怎么落的。现在当场算笔账：batch=64、词表 12.8 万、TPOT≈30ms 的 NPU 场景。把两块分开：mask 生成能和前向 overlap，那它占关键路径几乎为零——你要证明"能被藏住"的前提；bitmask apply 是关键路径上躲不掉的，64×15.6KB 的位图展开做 masked_fill_，你估它多少 μs、占 30ms 的百分之几？再回答我：batch 从 64 拉到 512，你那个"<1%~3%"还成立吗，为什么？

**面试者：** 分两块算。

先说 mask 生成"能被藏住"的前提：它跑在 CPU/host 侧，与 NPU 前向是两条流。单请求填 bitmask 20–80μs，batch 64 用线程池并行填，墙钟时间远小于 64× 串行。只要这个 host 耗时 < 一次前向的 30ms，就能被 overlap 完全藏掉，关键路径占比≈0。但前提是两条：一是 host 侧别成瓶颈（CPU 核够、并行真的展开），二是调度上真的把 mask 生成排在前向的 non-blocking 窗口里——否则就暴露到关键路径，这正是我们异步调度叠约束踩时序坑的地方。

再算 bitmask apply（躲不掉的）：batch 64 时位图 64×15.6KB≈1MB，读进来展开；真正主导的是对 logits 的写——64×12.8万×4B(float32)≈32MB。masked_fill_ 是纯 element-wise、访存主导。按 NPU HBM ~1TB/s 带宽，32MB 读＋写≈几十 μs，算上 kernel 启动和位图展开，给 50–100μs 量级。换算 100μs/30ms≈0.33%，落在 0.1%~0.5%，确实 <1%。

batch 拉到 512：apply 绝对开销随 batch 近线性涨约 8×，到几百 μs~1ms 量级。但——TPOT 本身在 512 batch 下前向计算量也显著变大（访存/算力都涨），分母同步放大，所以占比大致守恒，不会简单线性恶化。真正的风险不在 apply，在两处：一是 mask 生成的 host 串行部分若没并行化，512 请求累积可能"藏不住"暴露到关键路径；二是 context-dependent token 的现场检查在大 batch 下线性累积。所以我的辩证结论是：apply 占比大致守恒仍 <1%~3%，但大 batch 下的真实风险是 host 侧 mask 生成的瓶颈，那才是会让占比抬头的地方。

### 第 4 轮 · 异步调度 + 约束解码的时序 bug（off-by-one）

**面试官：** 32MB÷带宽这个算法我认，但你计账重复了——argmax/softmax 本来就要遍历整行 logits，masked_fill 的访存很大程度被采样 kernel 摊掉甚至可融合，你那 50–100μs 是高估；顺带记着，回头告诉我 mask 能不能跟采样融合成一个 kernel。现在别打太极，把你反复提的那个异步调度时序 bug 讲透：bitmask 是基于哪一步的 grammar 状态生成的、又 apply 到哪一步的 logits？异步把 sampling 结果回传延后一拍后，grammar 状态和 mask 具体怎么错位、错的是哪个 token？你当时靠什么现象定位到的、最后怎么修的——是加屏障、状态回滚，还是延迟推进？给我真实的调用栈级细节。

**面试者：** 先认批评。50–100μs 是高估——argmax/softmax 本来就要遍历整行 logits，masked_fill 的访存基本被采样 kernel 摊掉甚至融合，不该单独计账。融合问题直接答：mask apply 原则上能和采样融进一个 kernel，vLLM 就是用 Triton kernel 做 apply，理论上可与 argmax/softmax 融合、省一次全行读写；工程上是否真融看实现。下修到几十 μs 甚至更低，占 30ms 远不到 0.1%。

再讲那个 off-by-one。grammar 状态演进依赖"上一步真正采样出的 token"：正常同步流程是 step N 基于"已 accept 到 token_{N-1} 的状态"生成 mask → apply 到第 N 步 logits → 采样出 token_N → `accept_token(token_N)` 推进 → 再算 step N+1 的 mask。

异步/流水线下采样结果回传延后一拍：step N 的 token_N 到 step N+1 才拿到。如果状态推进没跟上，step N+1 就用"还没 accept token_N 的旧状态"去生成 mask——mask 比实际输出滞后一个 token，错的正是当前正在解码那个 token 的合法集合。表现为偶发吐非法字段名、括号提前/延迟闭合、JSON 校验挂。

定位：低并发同步跑死活不复现，一开异步调度或高并发才偶发，且和 batch 内位置相关。我在每步打了两条日志——grammar 内部的状态指针/已处理 token 数，和该请求实际 accepted token 序列，逐步比对，发现两者差一个，就是 off-by-one 坐实。

修法诚实说：核心是让 `accept_token` 推进与真实 token 回传严格对齐——延迟推进，拿到确定的 token_N 之后再推进状态、再生成下一步 mask，绝不用未对齐的旧状态填 mask；对约束请求必要时加个轻量屏障，不让它进入会导致状态错位的那级异步流水线，牺牲这一个请求的一点 overlap 换正确性。本质上这和投机解码"前瞻推进、拒绝后 rollback"是同构问题——都是 grammar 状态和真实被接受 token 必须严格对齐，差一个都不行。

### 第 5 轮 · 投机解码 × 约束解码：前瞻校验与 rollback 的正确性

**面试官：** off-by-one 的因果链和延迟推进我认，屏障也确实是最稳的兜底。但挑两点：加屏障就把你辛苦攒的 overlap 又吐回去了，有没有不牺牲并发的做法？延迟推进会不会让约束请求整体慢一拍，你量过没有？先记着。顺你自己的类比往下：投机解码一次抛 k 个 draft token，约束校验到底发生在草稿阶段还是验证阶段？grammar 要一次前进 k 步再按接受数回滚——这个 rollback 靠 xgrammar 的 persistent execution stack 和 max_rollback_tokens 怎么落地？关键场景：第 j 个 draft 被 grammar 判非法、却被目标模型接受，或反之，你怎么保证最终采纳的 token 序列既合法又符合投机解码的分布正确性？

**面试者：** 先接两个挑刺。(1) 不牺牲并发：不该给整条请求加屏障，只对"grammar 状态推进"这一步做细粒度依赖——拿到真实 token 才推进，而 mask 生成对大多数请求照样 overlap；再激进点可以"推测性推进+事后校正"，先按最可能 token 推进，回传不一致再回滚，把屏障粒度压到最低。(2) 延迟推进影响的是"下一步 mask 何时能算"，只要 mask 本来就藏在前向后面，延迟一拍通常仍落在前向时间窗内，对约束请求 TPOT 影响小于一个 step 量级——但我诚实说，这只是定性观察，没跑严格 benchmark，欠你一个数。

投机解码。校验时机：**草稿和验证两阶段都要用同一个 grammar mask**。草稿阶段每步都要用 mask 约束候选 token，否则草稿会吐一堆必被拒的非法 token，接受率暴跌；验证阶段目标模型对每个位置的分布也被同一 mask 约束。

rollback 落地（**三段式，勿说成「verify 后再从 k 回滚到 m」**）：

1. **试探**：沿 k 个 draft 逐个 `accept_token` 临时推进，边推进边填每个 speculative 位置的 bitmask——这时**假设 draft 全被接受**，但不是正式提交；靠建 matcher 时声明的 `max_rollback_tokens=k` + xgrammar persistent execution stack。
2. **立刻整体 rollback**：mask 算完马上 `rollback(state_advancements)`，grammar 回到窗口前真实状态——**发生在目标模型 verify 之前**。
3. **verify 后正式 accept**：目标模型验证后真实路径可能与 draft 分歧（不只是截断）。例：Draft=`A,B,C`，验证后=`A,X`（A 接受、B 拒绝、分歧改采 X，C 丢弃）。若没先 rollback，grammar 会假停在 `A→B→C`，而真实应是 `A→X`；从假状态无法靠「少回滚几步」直接得到 `A→X`。所以 verify 之后做的是**按真实采纳序列正式 `accept`**，不是 `rollback(k→m)`。

为什么 rollback 必须在 verify 前：① 分歧可能换 token 而不只是截断前缀；② GPU 验证耗时长，越早撤假状态，依赖 grammar 的逻辑看到假路径 `A→B→C` 的窗口越短。

正确性关键正是"同一 mask 作用于草稿和验证"：非法 token 在两阶段都被置 -inf，目标模型根本不可能采样到它，所以你说的"grammar 判非法却被目标接受"这个矛盾从机制上不存在。分布正确性：mask 只是把非法 token 概率清零，不改变草稿分布 q 和目标分布 p 在**合法子集**上的相对比例；在 mask 后归一化的分布上做标准 rejection sampling（按 min(1, p/q) 接受、拒绝则从残差分布重采样），依然无偏——投机解码的分布保证不被约束破坏。

诚实边界：MindIE 当前是 **Serving 入口硬互斥**（`InferParam::ValidateMtpConstraints`），插件层仍未联调；我更有把握讲清的是 vLLM `grammar_bitmask` 的 `(1+max_spec_tokens)` 多位置 mask ＋「试探推进 → 立刻 rollback → verify 后正式 accept」三段式。深文见 `03` §3.5.5、本附录 A.3/A.4。

### 第 6 轮 · NPU 落地：bitmask 的 H2D 传输、kernel 选择与压缩比

**面试官：** "同一份 mask 同时作用于草稿和验证、rejection sampling 在合法子集上依旧无偏"——这个洞察是今天最亮的一点，说明你没把约束和投机当两张皮，认可。现在收网，把我憋了几轮的 NPU 落地问题一次问透：(1) host 侧算好的 int32 压缩 bitmask 怎么过到 NPU——H2D 拷贝走没走 pin memory、和前向是不是异步双流？(2) 位图展开加 masked_fill_ 到 logits，你是接 xgrammar 官方 kernel 还是自己写 NPU 算子，为什么？(3) batch×15.6KB≈1MB/步 的 H2D 传输，在总线上占多少带宽、会不会成瓶颈，你凭什么判断"传压缩位图"比"传完整 mask"划算？给我数。

**面试者：** 三个都给数和取舍。

(1) 传输：bitmask 在 host 算好后走 H2D 到 NPU。要点是锁页内存（pin memory）让 DMA 直传、省掉一次从可分页内存到锁页的额外拷贝，并且放在**独立拷贝流**上异步 H2D，与前向计算流分开，让传输和计算 overlap。vLLM 就是 pin memory + `non_blocking=True` 的做法；MindIE NPU 侧对应用 ACL 的异步内存拷贝流（aclrtMemcpyAsync + 独立 stream），拷完再让计算流 wait 一下这个流。

(2) kernel：优先复用 xgrammar 思路，在 NPU 上用 `masked_fill_`（torch_npu 的 masked_fill_ 或等价 NPU 算子）落地，不自研。原因是这类算子 element-wise、访存主导，框架算子已接近带宽上限，自己写一个高性能 NPU kernel 成本高、收益有限。工程节奏是先用框架算子快速落地保证正确性，性能不够再谈把 apply 融进采样 kernel——vLLM GPU 侧就有自研 Triton kernel 和直接调 xgrammar 官方 kernel 两条路并存，验证了"能融、也能直接用现成算子"。

(3) 带宽账。batch=64、压缩位图 1MB/步、一步 30ms：1MB/30ms≈33MB/s。相对 PCIe Gen4 ~32GB/s（或 NPU 片上/总线带宽更高），占比 0.1% 量级，九牛一毛，根本不是瓶颈。

为什么传压缩位图划算：int32 位图每 bit 表示一个 token 合法性，压缩比正好 32×。若改传"完整 mask"，per-token 用 float32 就是 64×12.8万×4B≈32MB/步，是压缩位图的 32 倍，32MB/30ms≈1GB/s——虽仍没打满总线，但白白多占 32× 传输量，而且到了 device 还是要展开。所以传 bit-packed 位图省 32× 传输，代价只是 device 侧多一步位展开，那是 element-wise、几乎免费。结论：压缩位图在传输量和总线压力上都明显优，是正确取舍。

诚实边界：MindIE NPU 侧算子的具体实现以当时版本为准，我这里讲的是设计取舍逻辑，不是逐行背实现。

### 第 7 轮 · 输出质量：约束何时有害、何时该关掉

**面试官：** pin memory + ACL 异步流、复用框架算子不自研、32× 压缩比带宽占比 0.1%——这套账清清楚楚，NPU 落地这块我没什么可挑的了，过。倒数第二题，换个不那么好躲的角度：有人质疑"强约束会把模型逼进低概率路径、产出合法但语义变差的输出"，你怎么接？给我三样东西，别和稀泥：(1) 一个真会因约束而质量下降的具体例子；(2) 从概率分布角度说清为什么变差——是不是等于对合法子集做了重归一化、丢了什么；(3) 什么场景你会**主动建议关掉约束**、改走 prompt+few-shot 或后校验？要有明确立场，别两头讨好。

**面试者：** 三样都给，不和稀泥。

(1) 具体例子：schema 要求模型输出推理密集任务的答案，但只留了 `answer` 字段、没留 reasoning 字段——模型被 mask 逼着跳过 chain-of-thought 直接闭合到 answer，等于砍掉了它算对答案所依赖的中间步骤，准确率明显掉。同类还有：枚举字段模型本想输出一个语义更贴切但不在枚举集里的值，被逼选次优枚举；数字字段模型想输出 "3.14159"，schema 限定 integer，被截成 "3"；或深层嵌套里被逼提前闭合括号。

(2) 分布角度说清：约束 = 把非法 token 概率置 0、再在合法子集上重归一化。数学上它不改变"给定合法前缀下、合法 token 之间的相对比例"，这也是投机解码那套 rejection sampling 还能无偏的原因。但它丢的是**概率质量的位置**：如果模型此刻的高概率质量大量压在非法 token 上（说明模型本来强烈不想走这个格式），重归一化就把采样强行搬到长尾的低概率合法 token 上。这些尾部 token 对应的是模型没对齐好、没怎么见过的路径，一步走偏、后续 conditioning 在这个坏前缀上继续偏，产生 compounding error。一句话：合法子集只占模型概率质量一小部分时，约束等于强采分布尾部，质量必然下降——**格式和模型意图冲突越大，损失越大**。

(3) 我的明确立场，什么时候主动关：
a) 模型经过 function-calling SFT、加清晰 prompt 本就能高概率产出合法格式——此时约束是保险丝不是主力，可以只做后校验+失败重试，甚至只在偏离时兜底，没必要全程 mask。
b) 开放式/推理密集任务（长 CoT、创意写作），全程格式约束会腰斩质量，应改 prompt+few-shot 引导 + 输出后 parser 校验、失败重采。
c) schema 与模型能力严重不匹配（罕见格式、模型没见过），硬约束只会批量产语义垃圾，不如先放宽 schema。

反过来，高并发、要求 100% 合法、schema 稳定的 tool-calling 生产链路，约束解码是刚需，光靠 prompt 兜不住。

总立场：**约束解码解决"合法性"，不解决"质量"**。合法性和模型意图对齐时它近乎免费；两者冲突时它是有害的。工程上正解不是加大约束力度，而是用宽松 schema 设计 + few-shot 让模型"本来就想"输出合法格式，把冲突降到最低——约束只做最后那道硬保证。

### 第 8 轮 · 收尾：补齐两笔欠账，宣布结束

**面试官：** 最后收两笔欠账，简短给数就行：(a) 你那个"条数+字节双门限"到底卡多少——比如多少 MB 封顶、多少条封顶，说个你敢写进配置默认值的数；(b) 延迟推进对约束请求 TPOT 的实测多几个百分点、还是多一个 step 的绝对毫秒数。整场评价：通过。亮点是量化功底扎实、账目自洽，"同一份 mask 让投机解码仍无偏"和 off-by-one 因果链是真懂不是背的；还欠的是遇到"跟随主流""双门限"这类地方容易滑成口号，落地数字偶有高估——把这些逼成硬数字，你就是很强的候选人了。

**面试者：** 两笔账直接给数。

(a) 双门限我敢写进默认配置的是：`max_bytes=512MB` AND `max_entries=256`，谁先触发谁先 LRU 淘汰。512MB 对齐 vLLM 默认、保证内存硬上界；256 条覆盖绝大多数 tool-calling schema 种类，防止条数爆的同时也挡住"256 条全是大产物撑爆内存"——字节门限兜住上界，条数门限兜住碎片。

(b) 延迟推进对约束请求 TPOT：典型场景增量 <1%，绝对值几百 μs 量级，远小于一个 step——因为它只是把下一步 mask 生成推后到 token 回传后，只要 mask 仍藏在 30ms 前向窗内就几乎无感。只有 host 侧已成瓶颈、mask 藏不住时才退化到最坏情况：多一个完整 step，即 +30ms、该步近翻倍。坦白讲我有把握的是这个量级判断，精确曲线欠一次严格 benchmark。

## 复盘总结（Review）

### 一、本场覆盖的 8 个考点与结论

| # | 考点方向 | 核心结论（面试者站住脚的论证） |
|---|---|---|
| 1 | 缓存容量口径：**FIFO/100 vs 按字节** | 代码真相是默认 **100 条 FIFO**（非 LRU/128）；最坏 100×~5MB≈500MB 会失控，按字节（512MB）更稳。迭代目标：**双门限** `max_bytes=512MB AND max_entries=256` |
| 2 | 后端选型：**xgrammar vs Outlines vs llguidance** | 三组量化对比（编译期 / 每步 μs / 递归表达）；决定性因素是 NPU 落地下的 **C++ 内核可移植性 + bitmask 与算子解耦**；Outlines 递归需展开近似出局 |
| 3 | **费米估算**：mask 开销占 TPOT | apply 走 HBM 带宽账 ≈几十 μs、占 30ms 约 0.1%~0.5%；batch 放大后占比**大致守恒**，真实风险在 host 侧 mask 生成瓶颈 |
| 4 | **异步调度 off-by-one** 时序 bug | grammar 状态推进必须与真实回传 token 严格对齐；用**延迟推进 + 细粒度屏障**修复；与投机解码 rollback 同构 |
| 5 | **投机解码 × 约束解码** | 同一份 mask 作用于草稿+验证两阶段 → 非法 token 两阶段均置 -inf；三段式：试探填 mask → **立刻** rollback → verify 后正式 accept；**rejection sampling 在合法子集上仍无偏** |
| 6 | **NPU 落地**：H2D / kernel / 压缩比 | pin memory + ACL 异步拷贝流；复用框架 `masked_fill_` 不自研；int32 位图 **32× 压缩**、带宽占比 0.1% 非瓶颈 |
| 7 | **输出质量**：约束何时有害 | 约束=对合法子集重归一化，丢"概率质量位置"→ 冲突大时强采分布尾部、compounding error；**约束解决合法性不解决质量**，给出三类主动关约束场景 |
| 8 | 收尾补数 | 双门限具体阈值 + 延迟推进 TPOT 影响（典型 <1%，最坏 +1 step） |

### 二、面试官认可的亮点

- **量化功底扎实、账目自洽**：全程用费米估算（内存上界、HBM 带宽、32× 压缩比、占比守恒）支撑结论，且数字前后一致。
- **两个"真懂而非背诵"的信号**：① "同一份 mask 同时约束草稿与验证 ⇒ 投机解码分布仍无偏"这一洞察，把约束解码与投机解码打通；② 异步 off-by-one 的完整因果链 + 真实定位手段（对比 grammar 状态指针与 accepted token 序列）。
- **诚实的取舍意识**：主动承认按条数缓存的缺陷、50–100μs apply 是高估、延迟推进只有定性观察等，不吹牛、划得清能力边界。

### 三、被面试官抓到的短板（改进清单）

1. **口号化倾向**：出现过"跟随 vLLM/SGLang 主流""双门限兜底"这类未展开的背书式表达 —— 教训：任何选型理由都要落到**技术因果**或**具体数字**，别用"主流/更优"糊弄。

3. **欠账 benchmark**：延迟推进对 TPOT 的影响、mask 生成能否被 overlap 藏住，均只有定性判断，缺严格实测曲线 —— 后续应补基准测试把量级判断钉成硬数据。

### 四、可复用的"降维打击"话术锚点

- 缓存："默认 100 条 FIFO；最坏按条数可到数百 MB，正解是 `512MB AND 256 条` 双门限，字节兜上界、条数兜碎片。"

- 性能："apply 走 HBM 带宽账占 TPOT 0.1%~0.5%，batch 放大占比守恒；真正的风险是 host 侧 mask 生成藏不住。"

- 质量边界："约束解决合法性、不解决质量；冲突越大损失越大，正解是宽松 schema + few-shot，而不是加大约束力度。"

## 附录 A：投机解码 × 结构化输出——当前缺口与可行实现方案

> 第 5 轮面试者留了一句诚实边界：*"MindIE 当时投机+约束是否完整联调上线要看版本。"* 本附录基于**当前仓库真实代码**把这句话钉死：先如实盘点两特性各自的实现与"是否打通"，再参照 vLLM 的成熟做法给出一个落地导向、分阶段的实现方案。
>
> **阅读约定**：凡标注「代码依据」的是从本仓库读到的**事实**（给出文件/类/方法名）；凡标注「设计建议」「推测」「需进一步确认」的是**方案侧**内容，非现状。性能数字沿用全文口径并标注为量级估计。

### A.0 结论先行（一句话）

**插件层未打通 + Serving 入口已硬拦（分层真相）**：投机解码（MTP 插件）与结构化输出（xgrammar 约束）在引擎插件侧**各自能独立工作、但两者没有打通（未联调）**——grammar 没有 rollback 接口、matcher 建立时没设 `max_rollback_tokens`、bitmask 只按单位置生成、FSM 状态每步只推进 1 个 token，而 MTP 的 propose/verify 路径完全不碰 grammar。  
**产品侧**：`InferParam::ValidateMtpConstraints` 已对 `mtp + response_format` 做**入口硬互斥**（报错 `"structured output (response_format) cannot be used with mtp"`；UT 有断言）——属于「契约 fail-fast 盖住工程缺口」。  
**勿再说**「完全无互斥 / 配置上可同时开启且无报错」——那是只 grep 了 Python 插件的旧口径；C++ Serving 入口已拦。插件层「无联动 raise」仍是事实，但那是能力缺口，不是「产品允许同开」。

### A.1 现状：两特性各自长什么样（代码事实）

**结构化输出（约束解码）**——`mindie_llm/text_generator/plugins/structured_output/`

- `XgrammarGrammar`（`structured_output_grammar.py`）暴露的方法只有：`num_processed_tokens`、`num_tried_tokens`、`accept_tokens`、`fill_bitmask`、`is_terminated`。`accept_tokens` 内部对每个 token 调 `matcher.accept_token(token)` **只前进、不回退**。
- `GrammarBackend.create_grammar`（`structured_output_manager.py`）里 `matcher = compiled.xgr_module.GrammarMatcher(compiled.ctx)`——**构造 matcher 时没有传 `max_rollback_tokens`**。
- `fill_bitmask` 调 `matcher.fill_next_token_bitmask(bitmask, idx)`，**一次只填 1 个位置**；bitmask 缓冲区 `_init_bitmask_buffer` 形状是 `[bitmask_prealloc_batch, vocab_size//32]`，即 **batch × 1 个位置 × (vocab/32)**。
- `update_states_after_sampling` 把 `token_ids` reshape 成一维后，对第 i 个请求取 `flattened_token_ids[i]`、只 `accept_tokens(..., [token])` **一个** token——**隐含"每步每请求恰好 1 个 token"的假设**。

**投机解码（MTP）**——`mindie_llm/text_generator/plugins/mtp/`

- `MtpPlugin` + `DecodingPolicy`：草稿模型一次 propose `num_speculative_tokens`（记为 k）个 draft token，主模型在 k+1 个位置上验证。
- 验证是**贪婪逐位精确匹配**：`verify_greedy_one_batch`（draft==target 才继续，`plugin_verify_exp` 用 `torch.cumprod(matches)` 数接受长度 `indices_counts`）。当前**不是 rejection sampling**，也不涉及分布残差重采。
- `handle_sampling` 把采样参数复制成 `batch*(k+1)` 份，主模型 logits 也变成 `batch*(k+1)` 行。

**关键事实：`mtp_plugin.py` / `decoding_policy.py` 全文没有出现 `grammar` / `bitmask` / `structured` 任何引用**——投机路径与约束路径在代码上零交集。

### A.3 参照蓝本：vLLM 是怎么做对的（代码事实）

- **matcher 预留回滚窗口**：`backend_xgrammar.py::XgrammarBackend.compile_grammar` 里 `xgr.GrammarMatcher(ctx, max_rollback_tokens=self.num_speculative_tokens)`。
- **grammar 暴露回滚/试探**：`XgrammarGrammar.rollback(n)`（`matcher.rollback(n)` + 同步 `num_processed_tokens`）、`validate_tokens()`（accept 一串后立即 rollback，不真正推进；用于草稿过滤）。
- **多位置 bitmask + 三段式时序**（`__init__.py::grammar_bitmask`）：
  1. **试探填 mask**：分配 `max_batch_size*(1+max_num_spec_tokens)` 行；沿 draft 窗边 `fill_bitmask` 边 `accept_tokens`——**假设 draft 全被接受**，但这是临时推进、非正式提交；
  2. **立刻整体 rollback**：`grammar.rollback(state_advancements)` 退回窗口前——**发生在目标模型 verify 之前**；
  3. **verify 后正式 accept**：scheduler 拿到真实 `new_token_ids` 后再 `accept_tokens`（路径可能与 draft 分歧，见下）。
- **reasoning/think 跨窗口**：`should_advance` / `trim_reasoning_for_advance` / `reasoning_end_token_index` 处理"思考段 token 不能喂给 grammar、投机窗口跨越 think→answer 边界"。

#### 为何 rollback 必须在 verify 之前（正确性 + 暴露窗口）

反例（draft 与验证路径分歧，**不只是截断前缀**）：

```
Draft:           A → B → C
目标模型验证后:   A → X      （A 接受，B 拒绝，分歧位改采 X；C 丢弃）
```

| 若…… | grammar 会怎样 | 问题 |
|---|---|---|
| 试探后**不** rollback，带着 `A→B→C` 去 verify | 假状态长期停留 | 真实应是 `A→X`；从 `A→B→C` 无法只靠「回滚 2 步」得到 `A→X` |
| rollback 放在 verify **之后** | 验证期间所有读 grammar 的逻辑看到 `A→B→C` | GPU 验证耗时长 → **假状态暴露窗口被拉长** |
| **先 rollback 再 verify**（vLLM） | 假状态只存在于填 mask 的短同步段 | verify 后只需沿 `A→X` **正式 accept**，契约干净 |

**口径纠偏**：勿说「verify 后按接受长度从 k 回滚到 m」。接受长度 m 决定的是**正式 accept 几个 token**；试探态是在 verify 前就**整段撤掉**的。深文同步见 `03` §3.5.5。

这套「**试探推进 → 立刻 rollback → 真实 token 再正式推进**」正是 MindIE 缺的骨架。

### A.4 可行实现方案（落地导向、分阶段）

以下均为**设计建议**（非现状）。命名沿用现有类；xgrammar 原生能力以 vLLM 用法为参照，落地前**需对齐 MindIE 侧 xgrammar 版本**是否具备 `rollback` / `max_rollback_tokens`（**需进一步确认**）。

#### 1) 接口层：给 `XgrammarGrammar` / `GrammarBackend` 补齐能力

- `GrammarBackend.create_grammar`：构造 matcher 时传 `max_rollback_tokens = num_speculative_tokens`（k）——对应 xgrammar `GrammarMatcher(ctx, max_rollback_tokens=k)`，让底层 persistent execution stack 预留 k 步回滚窗口。
- `XgrammarGrammar` 新增：
  - `rollback(n)` → `matcher.rollback(n)` 并同步 `_num_processed_tokens -= n` / 复位 `_is_terminated`；
  - `validate_tokens(tokens)` → 逐个 `accept_token` 记录被接受前缀，末尾 `rollback(len(accepted))`，**只探测不推进**（草稿阶段可用）；
  - （可选）`accept_tokens` 支持"批量前瞻 + 返回接受数"，便于一次喂 k 个 draft。
- **注意**：现有 `num_tried_tokens`（replay 游标，含 rejected）与 rollback 的语义要对齐——rollback 只应回退**被 grammar 真正 accept** 的步数，不要动 replay 游标口径（现有 `sync_states_for_decode` 依赖 `num_tried_tokens` 做切片，别破坏）。

#### 2) bitmask 生成：单位置 → (1+k) 位置多行

- `_init_bitmask_buffer` 的 buffer 扩为 `[batch*(1+k), vocab//32]`（**量级**：batch=64、k=4、词表 12.8 万 → 64×5×15.6KB ≈ 5MB，仍 host 侧、μs~ms 级）。
- `grammar_bitmask` 对每个请求填 **1+k 行**，填充顺序（照抄 vLLM）：
  1. 草稿阶段：`for j in range(len(draft_tokens))`：先 `fill_bitmask(row_j)`，再 `accept_tokens([draft_j])` 前进一步，记 `state_advancements += 1`；
  2. 末尾 bonus 行：再 `fill_bitmask(row_{k})`（第 k+1 个真实/奖励位置）；
  3. **窗口算完整体 `rollback(state_advancements)`**，把 grammar 退回窗口前状态。
- 与前向 overlap 保留现有 host 侧并行填充思路（第 3 轮口径：host 填 mask 藏在 30ms 前向后）。

#### 3) 主循环整合：propose 与 verify 两阶段挂同一份多位置 mask

- **草稿阶段**：draft 模型每步用"当前位置那一行 mask"约束候选，避免草稿吐必被拒的非法 token（否则接受率崩）。
- **验证阶段**：主模型对 k+1 个位置的 logits，用**同一份** (1+k) 行 mask 逐位 apply（需把 `GuidedDecodingLogitsHandler` 的 apply 从 batch 行扩到 batch*(1+k) 行，并保证与 MTP `handle_sampling` 扩展后的 logits 行对齐——正好补上 A.2#3/#6）。
- **接受 m 个后推进**：验证得到接受数 m（≤k），grammar 从"窗口前状态"正式 `accept_tokens` 前 m 个真实 token，再 `accept` 第 m+1 个（bonus/真实采样）token。
- **正确性论证（同一份 mask ⇒ 无偏）**：非法 token 在草稿与验证两阶段都被同一 mask 置 -inf，目标模型不可能采样到它 → "grammar 判非法却被目标接受"从机制上不存在；mask 只清零非法 token、不改变**合法子集**上 q/p 的相对比例，在 mask 后归一化分布上做标准 rejection sampling 仍无偏（与第 5 轮结论一致）。**注意**：现有 MTP 是**贪婪 verify**，要享受这个无偏性需把 verify 升级为 mask 后的 rejection sampling（**范围扩大点，列为优化期**）。

#### 4) 回滚正确性：persistent stack + max_rollback_tokens + 边界 case

- 回滚窗口硬上界 = k，`max_rollback_tokens=k` 保证 `rollback(state_advancements)` 里 `state_advancements ≤ k` 永不越界。
- 边界 case：
  - **全部拒绝（m=0）**：草稿窗口 rollback 到窗口前，只 accept 主模型第 1 个真实 token，等价退化为单 token 约束解码；
  - **全部接受（m=k）**：正式 accept k 个 + bonus，grammar 前进 k+1；
  - **reasoning/think 跨投机窗口**：思考段 token 不喂 grammar，需类比 vLLM `trim_reasoning_for_advance` / `reasoning_end_token_index` 处理"窗口跨越 think→answer 边界"（**MindIE 侧当前无此逻辑，需新增；且 MindIE 是否有 reasoning parser 需进一步确认**）；
  - **terminated 跨窗口**：窗口内 FSM 提前到达接受态，其后位置 mask 应置全 1（现有 `fill_bitmask` 已有 `is_terminated → bitmask=-1` 分支，可复用）。

#### 5) 工程分期、验收标准与风险

| 期 | 目标 | 关键动作 | 验收标准 | 主要风险 |
|---|---|---|---|---|
| **MVP** | 串行正确性优先，不追性能 | 投机窗口内**逐 token 校验**：每接受 1 个 draft 就 grammar accept 1 个，遇非法即截断该请求本步接受数（等价"投机窗口内退化为单 token 约束"）；先不做多位置并行 mask | 开投机+约束，输出 100% 合法、JSON 可解析；关投机时与现状逐 token 对齐 | grammar 状态与 draft/target token 对齐；同步/异步推进时机统一（A.2#7）|
| **优化** | 拿回投机吞吐 | 多位置 (1+k) mask 并行填充、与前向 overlap；verify 升级为 mask 后 rejection sampling（保无偏）；apply 扩到 batch*(1+k) 行 | 接受率不因约束显著下降；TPOT 相对纯投机劣化在可接受量级（**需 benchmark**）| 多位置 mask 与 MTP logits 行对齐；异步调度叠加时序（第 4 轮 off-by-one 的放大版）|
| **兜底** | 保证不出错 | 检测到该请求带 grammar 且 rollback 能力不可用/未验证时，**自动对该请求关闭投机、退回单 token 约束解码**（"约束优先、投机降级"）| 任何情况下约束请求都合法产出，最坏只是该请求没享受到投机加速 | 降级判定要 per-request、不影响同 batch 其他纯投机请求 |

**总原则**：**约束优先，投机降级**——正确性（合法性）不可让步，性能可分期拿回。

#### 6) 时序框图：一次投机步内 grammar 的试探 → rollback → 正式 accept

```
                       ┌───────────────── 一个投机步 (k=4) ─────────────────┐
 grammar 真实态 S0      ① 试探填 mask（假设 draft 全接受；非正式提交）
   │                    propose 侧也可用同一套 mask 约束草稿候选
   │   fill row0 ─ accept d0 ─┐
   │   fill row1 ─ accept d1  │  state_advancements += 1 (每步)
   │   fill row2 ─ accept d2  │
   │   fill row3 ─ accept d3  │
   │   fill row4 (bonus 位)   │   ← 共填 (1+k)=5 行 mask
   ▼                          ▼
  S0 ──临时前进 k=4 步──────► S0+4 (假状态，仅用于算 mask)
                              │
                              │  ② rollback(state_advancements=4)  ← 必须在 verify 之前
                              ▼     max_rollback_tokens=k 保证不越界
  S0 ◄──────回到真实态──────── S0
   │
   │  ③ verify：主模型对 (1+k) 位置用【同一份多位置 mask】apply
   │     得到真实采纳序列（可能 ≠ draft 前缀截断，例如 draft ABC → 真实 AX）
   │
   ▼  正式推进：accept 真实采纳的 token（非「从 k 回滚到 m」）
  S0 ──前进 m+1 步──────────► S_{m+1}   ← 与真实输出严格对齐

  正确性锚点：propose 与 verify 用【同一份 mask】⇒ 非法 token 两阶段都 -inf
             ⇒ rejection sampling 在合法子集上仍无偏（约束不破坏投机分布）
  时序锚点：假状态暴露窗口 = 仅 ① 的短同步段；勿拖过 GPU 验证
```

### A.5 小结（做了什么调研、结论几期）

- **代码里确认的事实**：① 投机（MTP 插件）与结构化输出（xgrammar 约束）**插件层未打通**——MTP 的 propose/verify 全程不碰 grammar；② `XgrammarGrammar` **没有 rollback 接口**，`accept_tokens` 只前进；③ 建 matcher 时**未设 `max_rollback_tokens`**；④ bitmask **只按 batch×1 单位置生成**，FSM **每步只推进 1 个 token**；⑤ sampler 已能感知"bitmask 行数≠logits 行数"但只是**跳过**，没有多位置处理；⑥ **Serving 入口已硬互斥**（`ValidateMtpConstraints`）；Python 插件层仍无联动 raise——分层讲，勿说「完全无互斥」。
- **标注为"需进一步确认"**：绕过 Serving 直打引擎同开时的确切行为（静默 skip vs 异常）、MindIE 侧 xgrammar 版本是否具备 `rollback` / `max_rollback_tokens` 与其默认回滚窗口、MindIE 是否有 reasoning parser（think 标签跨窗口处理）。
- **对照蓝本**：vLLM 已用 `max_rollback_tokens=k` + `rollback()` / `validate_tokens()` + `(1+k)` 多位置 bitmask + **三段式**（试探填 mask → 立刻 rollback → verify 后正式 accept）实现该组合；rollback 在 verify 前是为缩短假状态暴露、并正确处理「draft 与真实路径分歧不只截断」的 case（`A,B,C` → `A,X`）。
- **方案分三期**：**MVP**（投机窗口内逐 token 串行校验，先保正确）→ **优化**（多位置并行 mask + rejection sampling 无偏 + 与前向 overlap，拿回吞吐）→ **兜底**（检测到约束请求即 per-request 关投机、退回单 token 约束，"约束优先、投机降级"）。当前产品选择是入口硬互斥，属兜底的契约化形态。
