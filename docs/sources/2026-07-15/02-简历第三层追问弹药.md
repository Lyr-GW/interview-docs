# 1. guidance「预留」边界

### 标准口述答案

> MindIE 结构化输出**唯一落地后端是 xgrammar**。枚举 `GuidedDecodingBackendType` 里目前只有 `XGRAMMAR = "xgrammar"`，没有 guidance / outlines 实现分支。  
> 简历写「接口层预留多后端（xgrammar / guidance）扩展抽象」，指的是：**配置与工厂入口按 backend 类型扩展的位置已经留好**（`StructuredOutputConfig.backend` + `GrammarBackend` 按类型创建），但 **guidance/llguidance 没有接线、没有编译路径、没有 matcher 适配**——属于扩展位，不是已交付能力。  
> 选型理由（工程向）：xgrammar C++ 内核、bitmask 与设备算子解耦，NPU 侧只接一条 `masked_fill_(-inf)` 就能落地；对「任意 JSON Schema」要 PDA/栈，Outlines 类 FSM 递归表达吃亏。  
> **对标 vLLM**：vLLM 是真多后端（auto 优先级链里 xgrammar / guidance / outlines 等可切换）；MindIE **不是**——我们是单后端交付 + 抽象预留。

### 证据边界

| 声明 | 证据 | 旗标 |
|------|------|------|
| 枚举仅 xgrammar | `MindIE-LLM/.../structured_output_manager.py` · `GuidedDecodingBackendType`（约 L88–91） | **实测/代码** |
| 默认 backend=XGRAMMAR、缓存 100 | 同文件 `_DEFAULT_GRAMMAR_CACHE_SIZE=100`、`StructuredOutputConfig` | **代码** |

# 2. TTFT−70% / E2E−50% 诚实卡

### 标准口述答案

> Motor KV 亲和在**高前缀重复**场景下，收益主因是少做 Prefill 重算，不是算子 magically 加速。  
> 简历「TTFT 降低 70%、E2E 降低 50%」我对外统一口径是：**Qwen3-32B、约 8K 输入、共享前缀约 6.5K 的代表性测算**，不是「某客户原始 A/B 日志」——除非当场能出示拓扑 + 请求构造 + 基线毫秒。  
> 机制可证：Coordinator 同源 tokenize → Conductor `/query` 最长前缀 → unified/load_gated 选实例 → 引擎侧 `cached_tokens` / Scheduler 权威日志。  
> 正确复测：两组都开 Prefix Cache，**只切** `load_balance` ↔ `kv_cache_affinity`，报 TTFT/E2E 的 P50/P95/P99、cached ratio、回退率、负载倾斜。

### 代表性测算公式与假设（可白板）

五段分解（见 `05` §6）：

```text
TTFT = T_tokenize + T_conductor + T_queue + T_prefill + T_delivery
```

代表性压缩公式（亲和收益打在 prefill 项上）[文档已有 · 专题 12 §3.1]：

```text
TTFT ≈ c₀ + T_prefill_full × (1 − h)

假设示例：
  c₀ = 80ms（调度 / Conductor / P→D / 首步开销量级）
  T_prefill_full ≈ 1230ms（单 P ≈6.5k tok/s、需算的 prefill 量级）
  Baseline h≈0.10 → TTFT ≈ 80 + 1230×0.90 = 1187ms
  Affinity  h≈0.78 → TTFT ≈ 80 + 1230×0.22 = 351ms
  降幅 (1187−351)/1187 ≈ 70.5%

E2E ≈ TTFT + (OSL−1)×TPOT
  例：+525ms decode（TPOT≈35ms×15）→ 1712ms → 876ms，约 −48.9%
```

| 假设 | 含义 | 脆度 |
|------|------|------|
| \(h\) 命中率抬升 | 亲和真正改的量 | **最脆**——流量重复率不够就不成立 |
| \(T_{prefill}\) / 吞吐 | 模型与并行配置 | 中 |
| \(c_0\) | 固定开销 | 相对稳；tokenize≈6ms/4K 已实测量级 |
| 两组都开 PC | A/B 公平性 | 方法论红线 |

边界：[经验量级] 重复率 80%+ 才可能逼近 −70%；20% 收益明显缩水；最坏 ≈ 纯 LB + 一次 tokenize，不灾难。

### 证据边界

| 声明 | 证据 | 旗标 |
|------|------|------|
| 1187→351ms、−70.5% | 专题 12 §3.1 测算表；wiki `kv-affinity.md` 方法 | **测算** |
| E2E −48.9% | 同上 + decode 假设 | **测算** |
| tokenize 4K≈6ms | TokenizerManager 墙钟 | **实测量级** |
| Conductor timeout 0.2s | `conductor_api_client.py` | **代码** |
| unified 公式 | `kv_cache_affinity.py` / `scheduler_server.py` | **代码** |
| 客户 4K「实测 −70%」原始日志 | 仓内无 A/B 报告、回放、拓扑快照 | **待补 / 勿宣称** |
| Conductor P50/P99 | 目标值有、报告无 | **待补** |

### 找不到客户日志怎么圆 + 红线话术

**怎么圆（推荐顺序）**

1. 先降调：「这是代表性测算，不是客户 raw log。」  
2. 立刻给公式假设与场景前提（模型 / ISL / 共享前缀 / \(h\)）。  
3. 转向机制证据：`cached_tokens`、Scheduler 重选日志、Conductor query。  
4. 补方法论：如何正确 A/B（两组 PC 都开、只切策略、分位数）。  
5. 若追「为什么简历写客户」：承认表述偏强，复盘后统一改口测算。

**红线（禁止）**

| 禁止 | 改为 |
|------|------|
| 「某客户实测日志 TTFT−70%」且拿不出文件 | 「代表性测算；机制可证」 |
| 用「关 cache vs 开 cache+亲和」刷百分比 | 两组都开 PC，只切调度 |
| 把 Prefill 节省说成「算子优化了 70%」 | 调度命中 → 少算 Prefill |
| 把 E2E−50% 说成任意负载都成立 | 高前缀重复 + decode 假设绑定 |

**极限诚实一句**

> 「仓里没有可复核的客户 A/B 原始包。我能辩护的是机制与测算假设；百分比必须以毫秒、分位数、样本量重新实测后才能写进对外材料。」

# 5. bitmask NPU 路径诚实卡

### 标准口述答案

> 采样前：xgrammar `fill_next_token_bitmask` 得到 **int32 压缩位图** `[batch, vocab//32]` → `GuidedDecodingLogitsHandler` → `apply_token_bitmask_inplace_npu`：  
> `repeat_interleave` 按 bit 展开 → `masked_fill_(-inf)` 屏蔽非法 logits。  
> 这是 **PyTorch / torch_npu 算子组合**，**不是**自研 Ascend C 自定义 kernel，也不是直接调 xgrammar 官方 CUDA kernel。  
> 工程取舍：element-wise、访存主导，框架算子已接近带宽墙；先正确性落地，性能不够再考虑融进采样 kernel 或专用 kernel。

### 证据边界

| 声明 | 路径 | 旗标 |
|------|------|------|
| NPU apply 实现 | `structured_output_bitmask.py` · `apply_token_bitmask_inplace_npu`（约 L46–63） | **代码** |
| Handler 挂载 | `pta_handlers.py` · `GuidedDecodingLogitsHandler` | **代码** |
| UT | `test_structured_output_bitmask.py` | **代码** |
| pin memory / ACL 双流逐行 | 对练口述设计取舍 | **部分待补**——勿装作背过每条 ACL API |
| apply 占 TPOT &lt;1%~3% | Profiling 手册估计 + 18 实录账 | **经验量级**；严格 A/B **待补** |

### 可优化点（主动加分）

1. **与采样融合**：masked_fill 与 argmax/softmax 访存重叠，独立 kernel 可能高估开销。  
2. **H2D**：压缩位图 + pin memory + 独立拷贝流，与前向 overlap。  
3. **多位置 bitmask**：若未来打通 MTP，buffer 从 `batch×1` 扩到 `batch×(1+k)`。  
4. **整图 / aclgraph**：mask+sampler 常因动态控制流在图外——讲清边界即可。

### 极限诚实话术

> 「别问我『自研了哪个 NPU bitmask kernel』——没有。就是 torch 侧展开位图再 `masked_fill_`。我贡献在约束链路正确性与异步时序，不在再造一个 element-wise kernel。」

# 6. async mask 错位：现象 → 修法 → 对 TPOT 的影响

> 交叉引用：[`docs/2026-07-10/04`](../2026-07-10/04-简历项目第三层追问弹药.md) §1/§5；口述细版见 [`docs/2026-07-06/03`](../2026-07-06/03-题库01参考回答.md) Q16；Profiling 补测见 [`05`](../2026-07-10/05-Profiling分层排查实战手册.md) §7。

### 标准口述答案（STAR 压缩）

**现象**

- 异步调度 + 结构化输出：非法 JSON、`GrammarMatcher` reject。  
- PD 场景典型症状：D 节点首 token **多一个 `{`**。

**根因（三个错位）**

1. **线程错位**：mask 曾在主线程 `preprocess` 生成；async 下 batch N 尚未 sample/accept，主线程已按过期 FSM 为 N+1 填 mask。  
2. **游标错位**：C++ `AddGeneratedToken` 无条件入 buffer；Python 若只用「接受数」切片，一次 reject 后永久 off-by-one → 用 `num_tried_tokens`（含 rejected）对齐。  
3. **顺序错位**：decode 若先 init grammar 再 sync，会跳过 replay；初始态 mask 只允 `{`。

**修法**

1. Async：bitmask 生成搬到 `forward_loop`（forward 前）；sample 后立即 accept；`generate_token_async` 清 `last_sampling_metadata`。  
2. 双游标：`num_tried_tokens` / `num_processed_tokens`。  
3. Decode：**先** `sync_states_for_decode` **再** init / 填 bitmask（manager 内注释固化）。

**对 TPOT 的影响（诚实分层）**

| 层面 | 说法 | 旗标 |
|------|------|------|
| 正确性修复本身 | 职责搬移到 forward 线程串行 mask→forward→sample→accept，消除竞态；**不是**为刷 TPOT | 机制 |
| mask 生成 | CPU 侧，理想可与 NPU 前向 overlap；藏不住时才进关键路径 | 机制 |
| apply | 访存主导，估计占单步很小比例 | 经验量级 |
| 交付期开关对比 | **无严格 A/B 报告**（平安二面 Q32 缺口） | **待补** |
| 补测预期 | 热缓存后 TPOT P50 增量 &lt;1%~3%；冷编译打在 TTFT +100–200ms | 估计；见 `05` §7 |

口述收口：

> 「这个 bug 的验收标准是正确性归零，不是 TPOT−x%。性能上我只能给机制账 + 补测方案；没有就把估计说成客户实测。」

### 证据边界

| 项 | 路径 / 文档 |
|----|-------------|
| sync 顺序注释 | `structured_output_manager.py` ~L649–658 |
| tried 游标 | `structured_output_grammar.py`；`sync_states_for_decode` |
| async 搬移 | `plugin_manager.py` forward_loop 分支 |
| 细口述 | `2026-07-06/03` Q16（已对照源码） |
| 母本 | `2026-07-10/04` §1、§5 |

### 极限诚实话术

> 「延迟推进/屏障会不会吐 overlap？会，正确性优先。我没有留下『修完前后 TPOT 对比』的正式报告；若考核性能，按 `05` §7 做 A/B，不拿故事代替直方图。」

# 7. MTP × 结构化硬互斥：产品理由 vs 工程理由

### 标准口述答案

> **Serving 入口硬互斥**：`InferParam::ValidateMtpConstraints`——mtp 开启且请求带 structured output（`response_format`）直接报错：  
> `"structured output (response_format) cannot be used with mtp"`（`infer_param.cpp`；UT 有断言）。  
> **产品理由**：对外契约清晰——同开要么静默错约束、要么投机收益与正确性打架；宁可 fail-fast，不让业务拿到「看起来在加速、JSON 已破防」的结果。  
> **工程理由**：插件层**未打通**——MTP propose/verify 不碰 grammar；matcher 创建未设 `max_rollback_tokens`；bitmask 仅 `batch×1` 单位置；无 rollback API。对照 vLLM：需要 `(1+k)` 多位置 mask + 三段式（试探填 mask → **立刻** rollback → verify 后正式 accept）。异步再叠一层时序，风险乘积过大，交付期选择互斥保正确性。

**分层别踩坑**（对齐专题 18 与代码）：

| 层 | 事实 |
|----|------|
| C++ InferParam | **有**硬互斥 |
| Python MTP 插件 | 与 grammar **零引用交集**；未联调 |
| 「完全无互斥」 | **错**——那是只 grep 了 Python；入口已拦 |

### 证据边界

| 声明 | 路径 | 旗标 |
|------|------|------|
| 硬互斥 | `infer_param.cpp` ~L216–224；`test_infer_param.cpp` ~L678 | **代码** |
| 插件未打通清单 | `interview-review/18` 附录 A | **文档+代码核对** |
| 未来打通方案 | 同 18：rollback、多行 bitmask、verify 挂载 | **设计建议，非现状** |

### 极限诚实话术

> 「互斥首先是工程做不到安全同开，产品上才包装成明确错误。不是『业务不需要 JSON+MTP』——是同开会错。要支持就按 vLLM 范式补 rollback 与多位置 mask，工作量可控但不是现在已交付。」

> **弱绑定**：简历边角/早期经历，不要抢结构化输出与 Motor 主战场时间。被问到 45–60 秒收束。

### STAR（可背短版）

| 段 | 口述 |
|----|------|
| **S** | 实习期推理/模型侧需要对比多配置、多指标，原始日志难读，评审靠表格和口头「感觉更快」。 |
| **T** | 把 TTFT / TPOT / 吞吐 /（可选）算子或阶段占比做成可对比可视化，支持同图多 run。 |

| **R** | 评审从「看日志」变成「看同一套轴」；更重要的是养成后来做 Motor/结构化时的 **profiling 分层习惯**（先 metrics 五段，再 nsys/msprof）。 |
