# 11 · KV cache utilization 与假命中（驱逐 / 事件空窗）

> 专题：亲和调度里两个常被「一阶补丁」对待、却决定高负载稳定性的因素。  
> 关联：[00 §5.1](00-概念与分层模型.md)（事件空窗）、[07](07-亲和与三级池化交互.md)、[09](09-ZMQ-KV-Events详解.md)  
> 工作区：`llm-d/`、`dynamo/`、`aibrix/`、`sglang/`、`router/`、`Mooncake/`、`MindIE-PyMotor/`

## 名词约定

| 说法 | 用法 |
|------|------|
| **KV cache utilization**（**KV 缓存利用率**） | 调度/文档正式名；llm-d 插件名含 `utilization` |
| **KV cache usage（percent）** | 监控/指标字段语感；如 vLLM `kv_cache_usage_perc`、端点属性 `KVCacheUsagePercent` |
| **GPU cache usage** | 强调 device 池占用（相对 CPU offload） |
| **block occupancy** / used-blocks ratio | 按块门控时（Dynamo `kv_used_blocks / num_gpu_blocks`） |
| 本文简称 | 首次后多用「**利用率**」；公式里写 $u_{\mathrm{kv}}$ 或 `kv_usage` |

**不是：** prefix hit rate / match rate（命中率）、waiting queue depth（排队深度）。  
利用率是 **memory pressure（缓存压力）** 信号，不等于「前缀还在」。

---

## 30 秒结论

1. **KV cache utilization** 度量的是显存/块池压力，不是「你的前缀还在」；完整用法应是 **接纳风险 + 驱逐先验**，多数框架只做到线性偏好低占用或硬踢超载机。
2. **假命中** 分两类：**驱逐滞后（假阳性）** 与 **事件空窗（多为假阴性；speculative 则是可控假阳性）**——对策不同，不可混为一谈。
3. 开源覆盖度：**llm-d / Dynamo** 最完整（utilization scorer 或 busy 门控 + Removed + speculative/predicted）；**AIBrix** 有 Event Sync 但 Cleared 空实现、默认路径不用利用率；**SGLang/vLLM Router** 近似树系统性不知驱逐；**Motor** 经 Conductor 可跟 Removed，但 **利用率与 speculative 均未进分**。

## 1. 问题定义

### 1.1 KV cache utilization

```text
kv_usage = u_kv ≈ used_kv_blocks / capacity
```

| 高利用率暗示 | 不直接等于 |
|--------------|------------|
| 难接纳大请求、易排队 | 某前缀一定还在 |
| 即将或正在 LRU 驱逐 | 命中在 L1 还是 L2/L3 |
| 亲和热机会自我拆台 | waiting 深度 |

亲和会把同前缀打到同一台 → 该机利用率升高 → 驱逐 → **高 $M$ 自我失效**。故利用率是亲和的 **稳定性约束**，不是普通负载装饰项。

### 1.2 两类假命中

```text
类型 A · 驱逐滞后（Eviction lag）——假阳性
  索引仍写「W 有前缀 S」，引擎已 BlockRemoved
  → 按高 M 打到 W → 实际全量/大部重算

类型 B · 事件空窗（Decision-before-metadata）
  路由已决策，BlockStored 尚未进索引
  → 多为假阴性（兄弟请求看不见新缓存，打散）
  → 若开 speculative：TTL 内可控假阳性（假定会写入）
```

| | 类型 A | 类型 B |
|--|--------|--------|
| 主因 | 未订 Removed / 丢事件 / Cleared 未实现 / approx 不知驱逐 | 决策领先元数据 |
| 高利用率时 | Removed 风暴，更严重 | 写入更密，空窗更密 |
| 解药 | Removed+Cleared+replay | speculative/predicted TTL |

二者与利用率 **正反馈耦合**：满 → 驱逐多 → 假命中多 → 更满。

## 3. 分框架实现细节

### 3.1 llm-d（EPP）

> 本地：`llm-d/docs/`、`llm-d/guides/`。插件实现在外仓 `llm-d-router` / `llm-d-kv-cache`。

#### KV cache utilization

| 项 | 内容 |
|----|------|
| 组件 | `kv-cache-utilization-scorer` |
| 数据 | `metrics-data-source` scrape → `core-metrics-extractor` → 端点属性 `KVCacheUsagePercent` |
| 进分 | Filter→Score→Pick；该 scorer 输出 $[0,1]$，**偏好更低利用率**（防碎片） |
| 典型权重 | **2.0**（与 queue 同级；prefix 常 3.0） |
| 配置 | `guides/*/router/*.values.yaml` 中 `pluginRef: kv-cache-utilization-scorer` |
| 文档 | `docs/architecture/core/router/epp/scheduling.md`、`datalayer.md` |

**局限：** 线性偏好低占用；**未**与「本请求未缓存 token 数」「前缀存活概率」耦合；与 prefix scorer 独立加权，高 $M$ + 高利用率的冲突靠权重硬调。

#### 驱逐 / 空窗

| 模式 | 行为 |
|------|------|
| **precise** | Indexer 订 `BlockStored` / `BlockRemoved` / `AllBlocksCleared`（`kv-indexer.md`） |
| **approx** | `approx-prefix-cache-producer` 路由后学习；**不订**引擎驱逐 |
| **speculativeIndexing** | `true`（precise 推荐）：路由后写入短 TTL 预测条目（默认 **~2s**），`BlockStored` 确认或过期删除。TTL = 预测条目最长存活时间，用来填「选机 → BlockStored 到达」空窗；详见 [09 §6](09-ZMQ-KV-Events详解.md)、[00 §5.1（3）](00-概念与分层模型.md) |

投递：Centralized（引擎 PUB→EPP SUB）或 Pod discovery（多 EPP 各订全量）。

**局限：** speculative 窗口内仍可能短暂假阳性；approx 路径假命中与引擎 LRU 无关，直到本地索引淘汰。

### 3.2 NVIDIA Dynamo（`dynamo/lib/kv-router/`）

#### KV cache utilization

| 项 | 内容 |
|----|------|
| 进 cost？ | **否**（cost 用 active prefill tokens / decode blocks / overlap） |
| 门控 | `active_decode_blocks_threshold` 等：worker 上报 `kv_used_blocks ≈ num_gpu_blocks × kv_cache_usage`（`components/.../vllm/publisher.py`）；超阈 **移出候选**（block occupancy 门控） |
| 过热抑 herding | `overlap_score_credit_decay`：相对最轻 worker 的 active prefill 超出量衰减 device overlap credit（`scheduling/selector.rs`）——驱动是 **token 负载**，不是裸利用率标量 |
| 配置 | `DYN_ACTIVE_DECODE_BLOCKS_THRESHOLD`、`prefill_busy_threshold*`；HTTP `/busy_threshold` |

**特点：** 把「满」做成 **硬过滤 + credit 衰减**，而不是 llm-d 式独立 utilization scorer。更接近「满了别去 / 满了别太信 overlap」。

#### 驱逐 / 空窗

| 机制 | 路径 / 行为 |
|------|-------------|
| `BlockRemoved` | `indexer/radix_tree.rs` → `apply_removed`；按 hash 删；mid-chain 有修复 |
| `AllBlocksCleared` | remove/clear worker 索引 |
| Precise + predicted | `router_predicted_ttl_secs` → `SideIndexer`（`llm/.../indexer/side.rs`）；`find_matches` 取 primary 与 side 的 **per-worker max** |
| 纯 approx | `--no-router-kv-events` + `router_ttl_secs`（默认 **120s**）本地预测；到期 `pruning.rs` |
| 互斥 | predicted TTL **不能**与纯 local approx 同时开（配置校验） |

**局限：** LMCache 集成文档称 Router **未完整消费全部 LMCache events**；predicted TTL 窗口内仍有假阳性。

### 3.3 AIBrix（`aibrix/pkg/`）

#### KV cache utilization

| 路径 | 是否用利用率 | 细节 |
|------|--------------|------|
| 默认 **`prefix-cache`** | **否** | 门控/并列用 `RealtimeNumRequestsRunning`（Gateway 在途）；`prefix_cache.go` |
| **`least-kv-cache`** | **是** | `gpuCache + cpuCache` 选最低（`least_kv_cache.go`） |
| PD **decode** scorer | **是** | `FreeGPUPercent = 100 - KVCacheUsagePerc×100`（`pd/decode_scorer.go`） |
| HPA | 是 | `gpu_kv_cache_utilization` 等 |
| 指标源 | scrape | `metrics.KVCacheUsagePerc` ← `kv_cache_usage_perc`（`pkg/metrics/metrics.go`） |

**结论：** 指标采了，**默认亲和主路径没用**；要利用率需换策略或走 PD decode 公式。

#### 驱逐 / 空窗

| 模式 | 行为 |
|------|------|
| 默认 approx | `PrefixHashTable` + 路由后 `AddPrefix`；仅 **时间 LRU**（`AIBRIX_PREFIX_CACHE_EVICTION_*`，常 20min）；**不知**引擎 Removed |
| **KV Event Sync** | `AIBRIX_PREFIX_CACHE_KV_EVENT_SYNC_ENABLED` + remote tokenizer；`pkg/kvevent/handler.go` → `SyncPrefixHashTable` |
| Removed | `ProcessBlockRemoved`（`sync_hash.go`）：删 engine↔aibrix 映射与 prefixMap |
| Cleared | `ProcessAllBlocksCleared`：**占位空实现 `return nil`**（L368–371）→ **全量 reset 后索引可能陈旧** |
| Speculative | **无**短 TTL；sync 模式下仍有 `AddPrefix` 推测写，但不以 TTL 管理空窗 |

**局限：** Cleared 缺口；跨 Gateway 时 approx 驱逐不同步 Redis（源码注释）；默认路径假命中风险高。

### 3.4 SGLang Model Gateway（`sglang/sgl-model-gateway/`）

#### KV cache utilization

- **不进** `cache_aware`。
- 负载信号：`Worker.load()` ← `load_counter`（`WorkerLoadGuard` 请求 ±1）。
- `power_of_two` 可拉 `/v1/loads` 的 **total_tokens**（仍非 utilization / waiting）。

#### 驱逐 / 空窗

文件头设计声明：*approximate radix tree … eliminating the need for direct cache state queries*（`policies/cache_aware.rs`）。

| 项 | 行为 |
|----|------|
| 匹配 | 字符级 `Tree`；`match_rate = matched_chars / input_chars` |
| 树维护 | 每次选中后 `tree.insert`；`evict_tenant_by_size(max_tree_size)` 裁 **路由树** |
| 引擎 Removed | **不处理** |
| Speculative | **无** |

**假命中：** 引擎已驱逐，路由器仍认为该 URL 有前缀，直到树 LRU 或负载失衡切走 → **系统性假阳性**；另有 **字符≠token/block** 误差。

PD 树键：`pool::model`（prefill/decode/regular 隔离），避免互踢 tenant。

## 4. 实现手法对照（怎么做的）

### 4.1 利用率的三种用法

```text
① 软加权（llm-d）
   S += λ · (1 - kv_usage)     # 与 prefix/queue 并列；kv_usage = utilization

② 硬门控（Dynamo busy）
   if used_blocks > threshold: 剔除候选   # block occupancy

③ 独立策略（AIBrix least-kv-cache）
   argmin(gpu_cache + cpu_cache)   # 不与 prefix 融合
```

| 用法 | 优点 | 缺口 |
|------|------|------|
| ① | 可与亲和共存 | 线性、无接纳尺寸、无存活先验 |
| ② | 防止打进爆机 | 阈值以下利用率不进分 |
| ③ | 实现简单 | 默认亲和路径用不上 |

**仍普遍缺失：** `free_blocks < uncached_tokens` 硬约束；$u_{\mathrm{kv}}\times M$ 的有效命中衰减；GPU 满但 L2/L3 有副本时的分层处理。

### 4.2 假命中三类缓解

```text
① 事件真值：Stored + Removed + Cleared + replay
② 空窗填补：speculative / predicted 短 TTL（路由后假定写入）
③ 近似认命：本地树/LRU，用时间换正确性（高 churn 下假阳系统性）
```

| 框架代表 | ① | ② | ③ |
|----------|---|---|---|
| llm-d precise | ✅ | ✅ ~2s | — |
| Dynamo | ✅ | ✅ predicted | 可选 120s approx |
| AIBrix sync | ✅ Removed；❌ Cleared | ❌ | 默认走 ③ |
| SGLang/vLLM | ❌ | ❌ | ✅ 主路径 |
| Motor | ✅ 经 Conductor | ❌ | — |

## 5. 耦合：为何要一起看

```text
亲和 → 热机利用率↑ → 驱逐↑ → Removed 增多 / 执行前等待变长
                         → 类型 A 假阳 + 类型 B 空窗同时恶化
                         → 若仍用裸 M 高权重 → 正反馈
```

完整形态（目标，非单一框架已实现）：

```text
M̂(w) = Index(w) · SurvivalPrior(u_kv, waiting, ISL−M) · SpeculativeState
score = f(M̂, load, waiting, …)
```

## 8. 面试口述（约 40 秒）

> KV cache utilization 和假命中是亲和在高负载下稳不稳的关键。开源里 llm-d 用 utilization scorer + precise 的 Removed/Cleared + 2 秒 speculative；Dynamo 用 busy 门控和 overlap 衰减，再用 predicted TTL 补空窗。AIBrix 默认 prefix-cache 只看 running，利用率在别的策略里；开了 Event Sync 能删 Removed，但 AllBlocksCleared 还是空实现。SGLang/vLLM 的 cache_aware 是近似字符树，根本不听引擎驱逐。我们 Motor 经 Conductor 能跟真值驱逐，但调度侧还没把 utilization 和 speculative 接进 unified——这是下一步该补的稳定性闭环。
