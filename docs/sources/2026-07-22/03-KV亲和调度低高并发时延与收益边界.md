# KV Cache 亲和调度：低并发与高并发时延、收益及边界

> 目的：解释 Coordinator 侧 tokenization、Conductor 查询、KV 前缀命中、Prefill 和排队在低/高并发下分别占多少，并给出可用于简历与面试的诚实口径。  
> 源码基线：`MindIE-PyMotor@e843fcfb80cc`。  
> 数字分级：**代码事实**、**已有实测/经验量级**、**代表性测算**、**目标/待测**。后两类不得表述为生产实测。

## 1. 结论先行

1. **4K tokens 约 6 ms 是 Coordinator 侧单请求 tokenization 墙钟，不是前缀命中耗时。**它发生在查询 Conductor 之前，是否命中尚未确定；不包含 Conductor HTTP、Scheduler 仲裁或引擎 Prefill。
2. **低并发时队列近似为空，收益主要由命中后少算多少 Prefill 决定。**以 8K 输入、6.5K 命中、完整 Prefill 约 1230 ms 的代表性假设计算，剩余 Prefill 约 254 ms；完全未命中仍约 1230 ms。
3. **高并发时不能只看命中长度，排队会成为主要变量。**现行 `unified` 把每个 endpoint 的 `prefill_cost` 交给中心 Scheduler，再用权威负载重算全局最小分数；热点积压超过缓存收益后，会主动选择无命中但更空闲的 endpoint。
4. **高并发没有一个脱离 QPS、并发、batch、模型和拓扑的固定毫秒数。**合理结论是给出成本公式、翻盘阈值和测试矩阵，而不是把低并发 Prefill 时间乘请求数当成线上 P99。
5. **冷启动仍然存在于缓存层，不存在于路由历史学习层。**新实例/新前缀无 KV 时命中为 0；Conductor 不可用或索引未就绪时回退 LoadBalance。方案不需要先积累路由历史，但不能消除 KV Cache 自身的预热。

## 2. 源码事实：一次请求实际经过什么

### 2.1 Tokenization 是命中判断前的固定成本

`KvCacheAffinityPolicy._ensure_token_ids()`：

- 在 Coordinator 侧执行 chat template/encode；
- 将结果缓存在 `req_info.token_ids`，单请求最多执行一次；
- 同一份 token IDs 用于 Conductor 查询、`isl` 计算和 Prefill 负载记账。

源码：

- `MindIE-PyMotor/motor/coordinator/scheduler/policy/kv_cache_affinity.py:178-212`
- `MindIE-PyMotor/motor/coordinator/domain/workload_calculator.py:79-92`

因此 6 ms 的准确含义是：

```text
T_tokenize(4K) ≈ 6 ms  [已有经验墙钟]
```

它与最终命中或未命中无关。若当前引擎仍按文本再次 tokenize，这也是 Coordinator 新增的路由成本，不能说成已经通过 token-in/token-out 消除了下游 tokenization。

### 2.2 Conductor 查询及失败回退

请求长度至少覆盖一个 KV block 时，Coordinator 调用 `/query`，请求体包含 `model`、`block_size` 和 `token_ids`；客户端超时为 0.2 s。短于一个完整 block 的 prompt 不可能在按整块哈希的索引中命中，代码直接跳过 HTTP 查询。

源码：

- 查询/短请求 fast path：`kv_cache_affinity.py:102-120`
- 0.2 s 超时：`conductor_api_client.py:161-192`
- 异常后的 KVA → LoadBalance → RoundRobin 回退：`scheduler_client.py:1217-1256`

需要区分：

| 数字 | 性质 | 正确解释 |
|---|---|---|
| Conductor timeout = 200 ms | 代码常量/预算 | 故障路径的上限，不是正常 P99 |
| P50 < 5 ms、P99 < 20 ms | 目标/待测 | 可用于压测验收，不能说成现有实测 |

### 2.3 命中长度与剩余 Prefill 成本

Conductor 返回每个 DP endpoint 的 `matched_tokens`。代码计算：

```text
prefill_cost = max(0, isl - overlap_credit × matched_tokens)
load_cost    = active_tokens + 0.3 × active_kv_cache  # Prefill role
score        = prefill_load_scale × prefill_cost + load_weight × load_cost
```

默认三个权重均为 1.0。相关源码：

- per-DP 命中和 `prefill_cost`：`kv_cache_affinity.py:231-265`
- `unified` 评分：`kv_cache_affinity.py:300-346`
- Prefill workload：`motor/common/resources/endpoint.py:37-58`
- 默认配置：`motor/config/coordinator.py:117-146`

这个分数是 **token 等价工作量**，不是毫秒预测器。当前实现没有显式加入不同 endpoint 的实际 Prefill 吞吐、网络拥塞或 KV 传输时间。

### 2.4 高并发下的权威仲裁

Worker 的负载视图可能滞后。现行 `unified` 将所有 endpoint 的 `prefill_cost` 发送给中心 Scheduler；Scheduler 用自己的新鲜负载重新计算：

```text
combined = prefill_load_scale × prefill_cost + load_weight × fresh_load
```

然后在全部 endpoint 上取最小值，平局优先 `prefill_cost` 更低者。源码：

- Worker 全量上报：`scheduler_client.py:727-797`
- Scheduler 全局重排：`scheduler_server.py:698-750`
- stale-view/burst 单测：`tests/coordinator/scheduler/test_scheduler_allocate_arbitration.py:269-450`

`load_gated` 不是软融合：它先保留 Top-N 最低负载 endpoint，再在集合内选最长前缀；源码位于 `kv_cache_affinity.py:349-395`。这使它更偏向保护负载长尾，但可能牺牲更多缓存命中。

## 3. 统一时延模型

对 endpoint `e`，可以把 TTFT 拆成：

```text
TTFT(e) = T_tokenize
        + T_conductor
        + T_schedule
        + T_queue(e)
        + T_prefill(isl, matched_e)
        + T_delivery
```

其中：

- 低并发：`T_queue ≈ 0`，主要比较剩余 Prefill；
- 高并发：`T_queue` 可能从毫秒上升至数百毫秒甚至更高，必须与缓存收益同时比较；
- 完全命中也不代表所有计算归零，框架仍有请求处理、调度和首 token 生成开销；
- 部分命中后只为 suffix 新建 KV，但 suffix 的 Attention 仍需读取已缓存前缀，因此真实耗时不一定与 suffix token 数严格线性。

下文用线性 token 等价近似做量级判断：

```text
T_prefill_hit ≈ T_prefill_full × (isl - matched) / isl
```

这是代表性估算，不是算子级精确模型。

## 4. 低并发状态

### 4.1 4K 输入示例

假设：

- `isl = 4096`；
- 完整 Prefill 按约 615 ms 估算（由 8K/1230 ms 代表性基线线性折算）；
- 命中 `3072` tokens，剩余 `1024`；
- tokenization 约 6 ms；
- 正常 Conductor 查询暂以 5～20 ms 目标区间展示，尚无实测分位；
- 不含排队、delivery 和引擎内部固定开销。

| 状态 | Tokenization | Conductor（目标区间） | Prefill（估算） | KVA 路径小计 |
|---|---:|---:|---:|---:|
| 命中 3K/4K | 6 ms | 5～20 ms | `615×1/4≈154 ms` | 约 165～180 ms |
| 完全未命中 | 6 ms | 5～20 ms | 约 615 ms | 约 626～641 ms |

在这组假设下，命中约 75% 前缀可少做约 461 ms 的 Prefill 工作。简历中应概括为“节省数百毫秒重复 Prefill”，不要把 165～180 ms 写成生产实测。

### 4.2 8K 输入示例

采用既有代表性测算参数：

- Qwen3-32B、TP4；
- `isl = 8192`；
- 完整 Prefill `T_full ≈ 1230 ms`；
- 命中 `6500` tokens，剩余 `1692`；
- 8K tokenization 未实测，按 4K/6 ms 线性外推约 12 ms；
- 其他固定服务开销 `c0 ≈ 80 ms`，属于代表性测算。

计算：

```text
T_prefill_hit ≈ 1230 × 1692 / 8192 ≈ 254 ms
```

| 状态 | Tokenization | Conductor（目标区间） | 固定开销 | Prefill | 预计 TTFT |
|---|---:|---:|---:|---:|---:|
| 命中约 6.5K | 约 12 ms（外推） | 5～20 ms | 约 80 ms | 约 254 ms | 约 351～366 ms |
| 完全未命中 | 约 12 ms（外推） | 5～20 ms | 约 80 ms | 约 1230 ms | 约 1327～1342 ms |

注意：既有“1187 → 351 ms”测算的 baseline 假设约 10% 命中，不是“完全未命中”；若明确讨论 0 命中，应使用约 1.33 s 的量级，而不是 1187 ms。

## 5. 高并发状态

### 5.1 高并发新增的三项成本

#### A. Coordinator CPU tokenization

如果单请求 4K tokenization 为 6 ms，则仅按 CPU 时间粗算：

```text
500 QPS × 6 ms ≈ 3 CPU-seconds / second
```

即理论上至少占用约 3 个核的纯 tokenization 算力；实际还要加 chat template、Python 调度、HTTP 和序列化开销。8K 若按 12 ms 外推，则 500 QPS 约消耗 6 个核。这里是容量换算，不代表实际 CPU 利用率或 P99。

#### B. Conductor 查询排队与超时

长请求通常每次都发起同步 `/query`。高 QPS 下需要观察 Conductor 的 QPS、P50/P95/P99、连接池等待和超时回退率。200 ms 只保护故障上界；若正常查询已经接近该上限，KVA 可能从优化变成额外长尾来源。

#### C. Endpoint 排队和 herding

同一热前缀会让多个 Worker 同时偏好相同 endpoint。若只依赖 Worker 本地快照，burst 可能全部灌向同一实例。中心 Scheduler 的全量重排会在每次权威记账后提高该 endpoint 的 `fresh_load`，使后续请求逐步分散。

### 5.2 一个可解释的翻盘阈值

设 A 命中 `m` tokens、B 完全未命中，忽略实例压力修正和 KV 占用项，默认权重均为 1：

```text
score_A = isl - m + load_A
score_B = isl     + load_B
```

B 获胜条件为：

```text
load_A - load_B > m
```

例：8K 请求在 A 命中 6500 tokens。A 的权威负载只要比 B 高出超过约 6500 token 等价工作量，`unified` 就会放弃这次命中，把请求发给更空闲的 B。

源码中 Prefill 请求有真实 token IDs 时，分配记账直接使用 `len(token_ids)`。因此一个新的 8K 请求分配给 A 后，A 的 `active_tokens` 可增加约 8192；在简化模型中，这已经超过 6500 的命中收益，下一请求可能转向冷但空闲的 endpoint。这不是算法失效，而是用一次完整 Prefill 换取更短排队和更稳尾延迟。

### 5.3 三种高并发缓存分布

| 缓存分布 | 现行调度倾向 | 命中率 | TTFT/P99 方向 |
|---|---|---:|---|
| 所有 endpoint 都已持有热前缀 | `prefill_cost` 接近，按权威负载自然摊开 | 高 | 最理想；命中与负载兼得 |
| 只有一个 endpoint 持有热前缀 | 低负载时优先热 endpoint；积压超过阈值后转向冷 endpoint | 中等 | 平均命中下降，但避免单点排队打爆 |
| 所有 endpoint 都是冷缓存 | 全部 `matched=0`，`unified` 退化为负载选择 | 低/零 | 首批请求承担完整 Prefill，后续随 KV Events 到达逐渐转热 |

因此高并发不能简单说“命中后约 350 ms”。如果热前缀只存在于一个 endpoint，部分请求会主动放弃命中并承担约 1.3 s 的完整 Prefill，以换取更低的排队尾延迟；如果缓存已在多 endpoint 复制，则高并发下仍可能同时获得约 350 ms 量级的低并发计算路径和更均匀负载。

### 5.4 `unified` 与 `load_gated` 的高并发差异

| 模式 | 规则 | 高并发特点 | 适合目标 |
|---|---|---|---|
| `unified` | 剩余 Prefill + 权威负载的软加权和 | 长命中可抵消一定排队；超过翻盘阈值后选择冷 endpoint | 平衡平均 TTFT、命中率与负载 |
| `load_gated` | 先取 Top-N 最闲，再比最长前缀 | 热 endpoint 不在 Top-N 时直接失去资格 | 严控热点与尾延迟，接受命中率下降 |

不能仅凭默认值判断哪种一定更快；需要按实际流量的公共前缀比例、endpoint 数、并发和 SLO 做 A/B。

## 6. 冷启动与异常状态

| 状态 | Conductor 视图 | 调度行为 | 影响 |
|---|---|---|---|
| 新实例/新前缀 | `matched=0` | 按负载选择并执行完整 Prefill | 自然缓存预热 |
| KV Events 已同步 | 返回真实 per-DP 命中 | 亲和 + 负载选择 | 正常收益阶段 |
| Conductor 重启/索引恢复 | 可能暂时不完整 | 假阴性或回退 LoadBalance | 性能下降，不影响输出正确性 |
| Conductor 查询超时/异常 | 返回空结果 | KVA → LoadBalance → RoundRobin | 可用性优先 |

准确口径是：“不依赖路由历史学习，但存在 KV Cache 和索引同步的预热窗口。”

## 7. 推荐压测方案：把估算变成实测

### 7.1 固定变量

- 模型、TP/DP、硬件和 block size 固定；
- 两组均开启引擎 Prefix Cache；
- 只切换 `load_balance` 与 `kv_cache_affinity`；
- 固定 ISL/OSL，至少覆盖 4K 和 8K；
- 前缀重复率覆盖 0%、20%、50%、80%；
- 分别测试缓存全冷、单 endpoint 热、所有 endpoint 热。

### 7.2 并发阶梯

建议按目标环境容量设置，而不是照抄固定 QPS：

```text
低并发：1 / 2 / 4 concurrent，确保 queue≈0
中并发：达到设备约 50%～70% 利用率
高并发：达到稳态容量上限及 1.2× 过载点
突发：一次性 burst 2×endpoint、5×endpoint、10×endpoint 个同前缀请求
```

### 7.3 必采指标

- `T_tokenize`、Conductor P50/P95/P99、超时与 fallback rate；
- `matched_tokens`、`cached_tokens`、真实 prefix hit ratio；
- Scheduler 最终 `(instance, endpoint)`、`repicked` 次数；
- 每 endpoint `active_tokens`、`active_kv_cache` 和请求数方差；
- TTFT P50/P95/P99、TPOT、E2E、Prefill time、queue time；
- Coordinator CPU、Conductor CPU/内存/QPS、引擎利用率。

### 7.4 验收判断

```text
净收益 = ΔPrefill + ΔQueue - T_tokenize - T_conductor - T_schedule
```

- 低并发重点验证 `ΔPrefill`；
- 高并发重点验证 `ΔQueue`、负载方差和 P99；
- 若平均 TTFT 降低但 P99 或负载倾斜恶化，应提高 `load_weight` 或评估 `load_gated`；
- 若 Conductor P99/超时率升高，应先扩容/缓存/异步化查询，而不是继续提高亲和权重。

## 8. 简历与面试推荐口径

### 简历

> 在 Coordinator 侧执行与引擎同源的 tokenization（4K tokens 单请求 CPU 墙钟实测约 6 ms），基于 Mooncake Conductor 全局索引获取 DP rank 粒度的最长前缀命中；设计 unified 与 load_gated 双模式，在低负载时优先复用缓存，高并发时结合 Scheduler 权威负载抑制热点和羊群效应，减少长前缀重复 Prefill。

### 面试补充

> 6 ms 是命中判断之前的 tokenization 固定税，不是命中耗时。低并发时，8K 输入命中约 6.5K 的代表性测算约为 350 ms TTFT，完全未命中约 1.33 s；高并发不能沿用这个固定值，因为 queue 会进入主导项。代码会在热点 endpoint 的负载差超过命中 token 收益时选择更空闲的 endpoint，并由中心 Scheduler 用权威账本全局重排。上述毫秒数是绑定模型和拓扑的测算，生产结论需要按并发阶梯做 A/B。
