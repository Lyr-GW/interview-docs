## 二、三分钟自我介绍

面试官您好，我叫林炜，硕士毕业于复旦大学，目前在华为昇腾做 MindIE LLM 推理服务开发。我的工作重点不是单一模型效果，而是推理系统里的正确性、延迟、吞吐和 KV Cache 利用率。我有三条比较完整的交付线。

第一条是结构化输出。我从零完成了从 JSON Schema 到采样硬约束的链路：xgrammar 将 Schema 编译为字节级 PDA，GrammarMatcher 在每个解码步维护可接受状态，再把允许 token 集合转换为 bitmask，最终在 NPU 侧屏蔽非法 logits。为了处理 Schema 编译的冷启动开销，我按 Schema 内容做 SHA-256，并设计默认 100 条、命中不调序的 FIFO 缓存。这里最有价值的不只是把功能跑通，而是我处理过异步调度叠加后的正确性问题：调度提前推进、mask 对应的 token 步和采样实际消费步不一致，会产生非法 JSON。我最终把请求状态、token 游标和 mask 生命周期重新对齐。当前 NPU bitmask 是 torch 算子组合，不是我自研 fused kernel；生产落地后端是 xgrammar，多后端只是扩展抽象。

第二条是 Motor KV Cache 亲和调度。多实例部署里，普通负载均衡不知道 KV 在哪里，字符级近似路由又可能因为 chat template、system prompt 或 tools 注入与引擎 token 序列不一致。我的方案是在 Coordinator 用同源 tokenizer 得到 token IDs，再通过 Mooncake Conductor 的全局索引查询最长可复用前缀。调度上提供 unified 和 load_gated：前者直接加权缓存收益与负载，后者先用负载门控 Top-N，再在候选中优先最长前缀，同时保留超时和异常时的 fail-closed 回退。4K tokens 前置 tokenize 约 6 ms；TTFT 约下降 70%、端到端约下降 50% 是高前缀重复代表性场景的测算，不是客户原始日志。我可以继续展开测算假设和收益边界。

第三条是 Agent 服务与 Server 工程。我做过 Qwen3、DeepSeek V3 / V3.1 的 Tool Call 和 Reasoning 流式解析，也参与 C++ 请求链路重构。这里我会明确区分：结构化输出是在生成期对 logits 做硬约束，Tool Call parser 是对模型输出做协议解析，两者可以协同，但保证层级不同。

为了从服务侧继续深入，我最近按源码补了 vLLM、SGLang 和 MindIE 的关键机制：PagedAttention 如何通过 block table 管理非连续 KV，Continuous Batching 和 Chunked Prefill 如何在 TTFT 与 TPOT 之间取舍；FlashAttention 如何用 SRAM tiling 和 online softmax 降低 HBM IO；投机推理如何用 Draft-Verify 减少 Target 串行轮次，以及拒绝采样怎样保证分布无损；并行方面能从 TP、DP、PP、EP、CP / SP 的切分对象、通信模式和显存代价做选择。我的优势是有从零交付大特性的工程经验，也愿意把这套能力继续下沉到调度、算子协同和系统性能分析。

## 三、核心项目追问

### 1. 为什么 xgrammar 不是一个普通 FSM？

**答：**JSON Schema 存在递归嵌套和配对结构，有限状态机不擅长表达任意深度的栈语义。更准确的口径是 xgrammar 使用字节级的下推自动机表示语法状态。每一步根据当前栈和已消费字节得到可接受终结符，再映射到 token 合法集合。

**追问：token 是子词，Grammar 是字节级，怎么对齐？**

tokenizer 词表中的每个 token 都能展开成字节串。Matcher 判断某个 token 的字节串能否从当前 PDA 状态合法转移；为了避免每步遍历全词表，会缓存与上下文无关的 token 判定，并对上下文相关部分增量处理，最后生成 bitmask。

**再追问：会不会影响输出质量？**

正确性上保证语法合法，但模型可能被迫选择低概率的合法 token，因此“格式正确”不等于“语义质量不下降”。性能上有冷编译、每步合法集合更新和 mask apply 三类开销；目前我的硬证据是缓存机制与实现链路，严格 TPOT A/B 数字需要用固定模型、Schema、并发和输出长度补测，我不会现场编百分比。

### 2. 编译缓存为什么是 FIFO，而不是 LRU？

**答：**当前代码事实是 SHA-256(schema) 为 key、默认 100 条、FIFO 淘汰，命中不调序。选择的核心目标是容量受控、实现简单并消除重复 Schema 编译。LRU 对热点适应更好，但会引入命中更新和并发维护成本；是否值得切换，需要看 Schema 复用分布与编译压力，不能把“理论更好”说成当前实现。

### 3. 异步调度下 mask 为什么会错位？

**答：**约束状态与“已提交 token”绑定，而异步调度可能让下一步调度、上一轮采样和输出回传重叠。如果 mask 按已调度步生成，但采样按已提交步消费，或请求复用后没有同步重置游标，就会把第 n+1 步的合法集合用到第 n 步。我的定位方法是逐请求记录 committed token、scheduled step、matcher cursor 和 mask owner，找出第一个不变量被破坏的时刻，再统一推进时机和生命周期。

**追问：怎么验证修复？**

覆盖同步 / 异步、单并发 / 高并发、流式 / 非流式、缓存冷 / 热、提前 EOS 和复杂嵌套 Schema；不仅验证最终 JSON 可解析，还要逐 token 检查 matcher 状态与采样位置一致，并用压测确认请求间没有状态串扰。

### 4. 为什么 token 级亲和优于字符级路由？

**答：**引擎复用的是 token 对应的 KV block，不是原始字符串。chat template、system prompt、tools 注入和 tokenizer 版本差异都会让“字符串看起来相同”和“引擎 token 前缀相同”不等价。用与后端同源的 tokenizer 得到 token IDs，再查引擎上报的 KV 索引，能更准确估算真实可复用前缀。

**追问：前置 tokenize 会不会得不偿失？**

收益条件是 `节省的 Prefill + 排队收益 > tokenize + Conductor 查询 + 调度开销`。已有 4K tokens tokenize 约 6 ms 的量级，而高重复长上下文可节省数百毫秒 Prefill；短 prompt、低重复或 Conductor 异常时收益变小，所以必须超时回退普通负载均衡。

### 5. 你如何证明 TTFT 下降 70%？

**答：**这不是客户原始日志，而是高重复代表性场景测算。分解为：

`TTFT = tokenize + conductor + queue + prefill + delivery`

亲和调度主要通过提高 prefix hit 减少 Prefill，也可能降低排队。测算场景绑定 Qwen3-32B、约 8K 输入、约 6.5K 共享前缀和高重复率，结果约从 1187 ms 降到 351 ms。它对命中率最敏感；重复率只有 20% 时不能套用 70%。生产验证应做 KVA ON / OFF 同流量 A/B，比较 TTFT P50 / P99、queue、prefill、hit rate、cached tokens 和回退率。

### 6. Conductor、Store、Transfer Engine 分别做什么？

**答：**Conductor 管全局元数据与调度视图，回答“KV 在哪里、最长可复用前缀在哪个实例”；Mooncake Store 提供分布式对象存储和副本 / 淘汰语义；Transfer Engine 负责跨 GPU、DRAM、SSD 的高性能传输。我的项目直接使用的是 Conductor 查询和 KV 事件这层，不会把 Store、RDMA 或 HCCL 底层实现说成我亲自开发。

## 四、拓展技术面追问

### 7. 投机推理为什么能够无损加速？

**答：**它用便宜的 proposer 一次生成 K 个草稿，再让 Target 在草稿前缀上一次并行计算 K 个位置的 logits，Verifier 从左到右接受，减少昂贵 Target 的串行执行轮数。Greedy 下逐 token 对比 Target argmax；采样场景若 proposer 分布为 q、Target 为 p，则以 `min(1, p(x)/q(x))` 接受，拒绝后从 `normalize(max(p-q, 0))` 恢复采样，从而保持 Target 分布。

**追问：什么时候负优化？**

高并发 / 大 batch 让 Verify 从 memory-bound 变为 compute-bound；高温或领域不匹配使接受率低；draft 模型过重；输出太短；额外权重和 KV 挤压可服务 batch；复杂 logits processor 需要逐草稿位置重放状态。决策不能只看接受率，还要看每轮期望提交 token 数与 draft、verify、调度开销的比值。

**再追问：MTP 和 EAGLE 有什么不同？**

MTP 通常是主模型训练时自带的多 token 预测头，与 Target 表征天然对齐；EAGLE 使用 Target 的内部特征训练轻量 draft 网络，并可构造树形草稿后用 tree attention 验证。工程上两者都要处理 lookahead slot、变长 batch、KV 提交 / 回收、EOS 和采样状态；我目前是源码和算法理解，没有把它写成已生产交付。

### 8. FlashAttention 和 PagedAttention 是一回事吗？

**答：**不是。FlashAttention 解决单次 Attention 计算的 IO 效率：Q/K/V 分块放入片上存储，用 online softmax 在块内完成计算，避免显式落地 O(N²) 的 score / probability 矩阵。PagedAttention 解决服务系统中 KV Cache 的内存管理：把逻辑连续 KV 映射到非连续物理 block，通过 block table 在 Attention kernel 中寻址，降低碎片并支持动态批处理。前者偏算子，后者偏 KV 虚拟内存与调度，两者可以组合。

**追问：Decode 阶段为什么 FlashAttention 收益不一定像 Prefill 那么大？**

Prefill 的 query 长度大，朴素 attention 会产生大量中间矩阵和 HBM IO；Decode 每步 query 很短，主要瓶颈常变成权重 GEMV、随上下文增长的 KV 读取和 kernel launch。Decode 优化通常还要结合 batch、量化、Graph、算子融合和投机推理。

### 9. Continuous Batching、PagedAttention 和 Chunked Prefill 如何配合？

**答：**PagedAttention 让 KV 按 block 动态分配，为请求随进随出提供内存基础；Continuous Batching 在每个调度 step 重新组合 running / waiting 请求，不必等整批结束；Chunked Prefill 把长 prompt 拆成受 token budget 控制的块，使 Prefill 能与 Decode 同批，避免长 Prefill 独占一个 step。代价是单请求 TTFT 可能上升，但 TPOT 尾延迟和集群吞吐通常更稳。

**追问：KV 不够时怎么办？**

可选 recompute、swap 或拒绝 / 延后新请求。vLLM V1 倾向释放块并把请求置为 preempted 后重算，逻辑简单但长 prompt 代价大；其他路径可能 swap，节省计算但付出传输开销。需要结合 watermark、max batched tokens、max seqs、chunked prefill 和 KV utilization 调参。

### 10. TP、DP、PP、EP 怎么选？

**答：**先看模型是否单卡放得下、节点互联、时延目标和流量规模。TP 切层内矩阵，能降单卡权重但每层有 AllReduce / AllGather，适合单机高速互联；PP 切层，通信频率较低但有 pipeline bubble，在线低 batch 时延不一定好；DP 复制模型、切请求，横向吞吐最好但每副本要完整权重；MoE 的 EP 按专家切分，代价是 token dispatch 的 AllToAll，可与 TP / DP 组合。长上下文还可能用 CP / SP 切 sequence 和 KV。

**追问：Attention 的 TP 通信在哪里？**

常见是 QKV 做列并行，各卡保留不同 head 并本地计算 Attention；O projection 做行并行，各卡只有部分贡献，最后 AllReduce。TP 增大能省显存，但 Decode 每层通信占比会上升，因此不是卡越多越快。

**再追问：调参先设 batch 还是并行度？**

先定 world size 和并行拓扑，再算单卡权重 / KV 预算与通信域，之后用剩余显存确定 KV 池，最后压测 max batch size、max batched tokens 和并发。反过来先拍 batch 容易 OOM，或者得到不可实现的吞吐目标。

### 11. 吞吐上不去，你怎么定位？

**答：**先分层而不是直接抓 profiler：

1. 服务层看 TTFT、TPOT、TPS、waiting / running；
2. 资源层看 KV usage、preemption、CPU / NPU 利用率；
3. 调度层看每 step token 组成、Prefill / Decode 比例、prefix hit、chunked 和 batch budget；
4. 最后用 msprof / nsys 看 Host 空隙、H2D、Attention / GEMM、HCCL / NCCL 和密集短 kernel。

GPU / NPU 空且 Host 有间隙，优先怀疑调度、Python / C++ 前端或 launch；设备忙但 TPS 低，再判断是 Prefill、KV 读取、量化反量化还是通信。每次只改一个变量，用固定流量做 A/B。

### 12. 你会怎么配置 vLLM 做第一轮优化？

**答：**先建立基线，再按瓶颈选旋钮：高重复 prompt 开 prefix caching；长 prompt 与在线 Decode 混合时开 chunked prefill；用 `max_num_batched_tokens` 和 `max_num_seqs` 做吞吐 / 延迟平衡；在安全范围内调高 GPU memory utilization 给 KV；单卡放不下再考虑 TP，而不是默认多卡；小 batch Decode 可评估 CUDA Graph；显存或带宽受限评估 FP8 / AWQ / GPTQ；低并发且输出较长时再评估 speculative config；PD 分离则同时核对 KV connector 与传输成本。配置是否有效必须回到 TTFT、TPOT、TPS、KV usage、preemption 和 profiler 验证。
