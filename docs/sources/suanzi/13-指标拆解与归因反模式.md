# 13 · 指标拆解、归因反模式与简历数据答辩

> 面试官最爱拿你简历上的 **-70% / -50%** 追问。本文把指标、归因、反模式写成可防守口径，并挂回算子层。

---

## 1. 指标定义（必须口径一致）

| 指标 | 定义 | 主要敏感段 | 你简历里的角色 |
|------|------|------------|----------------|
| **TTFT** | 发请求→首 token | 排队 + **Prefill 算子链** | 亲和主收益点 |
| **TPOT** | 后续每 token 间隔 | **Decode**（IFA+FFN+sample+mask）| 亲和间接；过载会恶化 |
| **E2E 时延** | 整段生成总时间 | Prefill + Σ Decode | 案例 -50% |
| **吞吐** | tok/s 或 req/s | 常 Decode-bound | 要看是否挤爆实例 |
| **前缀命中率** | 路由命中比例 | 决定少跑多少 Prefill | 实验必须汇报 |

**答辩句式**：  
> 案例是 4K 上下文、高前缀重复。TTFT -70% 来自少做计算密集 Prefill；E2E -50% 因仍含全部 Decode，降幅小于 TTFT 符合预期。

---

## 1.1 可背数字卡（Host vs Device 数量级）

| 量 | 量级（口述） | 用途 |
|----|--------------|------|
| 场景 | **4K** 上下文、**高前缀重复** | 一切百分比的前提 |
| tokenize 前置 | ≈ **6ms**（4K） | 证明前置划算 |
| 所省 Prefill | **数百 ms** 量级（勿编精确 ms） | 对比 6ms |
| TTFT | **−70%** | 主砍 Prefill |
| E2E | **−50%** | 仍含 Decode，降幅更小合理 |

**雷区一句**：不是 FA/PFA kernel 快了 70%。  
用词总表：[`24`](./24-答辩用词对照.md)；60 秒口述：[`21`](./21)。

## 2. 归因公式（口语版）

```
TTFT ≈ T_queue + T_tokenize + T_prefill(+T_transfer)

亲和命中时：
  T_prefill' << T_prefill
  T_tokenize 前置约 6ms（可忽略相对 Prefill）

故 ΔTTFT / TTFT 可达很高（案例 70%），前提是 Prefill 原本占 TTFT 大头。
```

算子展开：`T_prefill` ≈ Σ(Linear GEMM + PFA + Norm/RoPE/写 cache…)。

---

## 3. 归因反模式（面试雷区）

| 反模式 | 为何错 | 正确说法 |
|--------|--------|----------|
| 「算子优化了 70%」 | 你未改 PFA kernel | 「调度少调用了 Prefill」 |
| 「E2E 也应 -70%」 | Decode 仍在 | 解释 TTFT vs E2E 组成不同 |
| 「命中率 100% 则吞吐翻倍」 | 可能已 Decode-bound | 看阶段占比与 load_gated |
| 「bitmask 让生成变快」 | 通常略增开销换正确性 | 正确性优先；性能中性偏略慢 |
| 「字符级也行」 | 假命中风险 | token 级为 Attention/KV 正确性 |
| 「混部与分离没区别」 | OI 与争用不同 | 用 PFA/IFA 争用讲混部风险 |

## 4. 对照实验怎么答「你怎么测的」

最低可信清单：

1. 同模型、同硬件、同并发；开关亲和；  
2. 报告命中率、TTFT/E2E/吞吐、Prefill 时长分布；  
3. 场景：4K、高前缀（与简历一致）；  
4. 排除仅短 prompt（Prefill 本就不贵则 TTFT 降幅小）。

若被问「有没有算子级 profile」：  
> 当时验收以服务指标为主；机制上 Prefill 时长下降与少跑 PFA/Linear 一致。若深挖可用 msprof 看 Kernel 占比验证。

---

## 5. 结构化输出的「性能」怎么谈才不虚

| 可量化 | 难吹但真实 |
|--------|------------|
| 重复 Schema 编译 → 0：`SHA256(schema)` + **FIFO** 容量约 100、命中不调序（**勿说 LRU/128**） | 约束下可能拒采样/重试（质量） |
| 错位 bug 修复 → 正确性 | bitmask 本身很少是主瓶颈 |

与亲和对比：亲和动 **Device Prefill**；编译缓存动 **Host**；两者都写进简历但分层不同（见 `09`、`15`）。

## 6. 白板：时延构成

```
E2E = TTFT + TPOT × (N_out - 1) + 尾部

亲和主要减小 TTFT 中的 Prefill 块
load_gated 防止 TPOT 变差抵消收益
```

---

## 7. 与简历数字对齐的三句话（背诵）

1. **场景**：4K、高前缀重复、多 vLLM 实例。  
2. **机制**：token 级前缀路由 → 少跑 Prefill（计算密集）。  
3. **数字**：TTFT -70%、E2E -50%；E2E 降少是因为 Decode 仍在。
