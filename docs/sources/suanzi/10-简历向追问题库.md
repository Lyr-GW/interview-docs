## A. 结构化输出

### A1 ★ 用两分钟讲清你做的结构化输出
**意图**：独立交付含金量。  
**口述**：Schema → xgrammar 编译 → Matcher 维护合法 token 集 → 每步 bitmask 屏蔽 logits → 采样；对标 vLLM。缓存 **SHA256(key)+FIFO 容量约 100（命中不调序）**——勿说 LRU；预留多后端；修过异步下 mask 错位。  
**可转深**：bitmask=Vector element-wise；常在 Graph 外。  
**雷区**：说成「我写了 FA」。

### A2 ★ bitmask 会拖慢 Decode 吗？
**意图**：是否懂主瓶颈。  
**口述**：通常占比远小于 Linear/Attn；Decode 仍多是搬权重/KV。若 Host 造 mask 慢或同步点过多，才可能暴露。用 profiling 看。  
**可转深**：小算子税、与采样融合。

### A3 ★ 异步调度下 mask 错位的根因？
**意图**：连续批处理时序。  
**口述**：多请求共享执行引擎，每步 logits 与「该请求当前合法集」必须同一步；异步提前算 mask 或复用旧 mask 就会错。修的是调度/生命周期对齐。  
**可转深**：CB 下 slot、步进。

### A4 ☆ 编译缓存在优化分层里算哪一层？
**口述**：Host 侧避免重复编译，类似缓存/常量折叠；不降 Device FLOPs。Device 侧另谈 FA/融合/Graph。

### A5 ☆ 和约束解码论文/ xgrammar 的关系？
**口述**：算法与编译器用社区方案；工程贡献是 MindIE 全链路、缓存、后端抽象、并发正确性。

## B. KV 亲和调度

### B1 ★ 为什么 TTFT 能降 70%？算子角度怎么解释？
**意图**：因果是否正确。  
**口述**：高前缀重复 → 命中后少做/不做大段 Prefill。Prefill 是计算密集（PFA+多层 GEMM），OI 高、耗时长；所以省 Prefill 直接打 TTFT。不是 FA kernel 加速了 70%。  
**可转深**：`OI∝S_q`；tokenize 6ms vs Prefill 数百 ms。

### B2 ★ 为什么 tokenize 要前置到 Coordinator？
**口述**：token 级最长前缀匹配需要 token id；前置 4K≈6ms，远小于一次误判满量 Prefill。  
**可转深**：字符级在 tools/system 注入下易假命中 → 假前缀仍跑满 PFA。

### B3 ★ unified 和 load_gated 区别？算子层含义？
**口述**：unified 综合前缀收益与负载；load_gated 先滤过载再选最长前缀。避免所有人打到「前缀最甜但 Decode 已挤爆」的实例——那边 IFA/FFN 会因 batch/带宽恶化。  
**雷区**：只谈负载均衡不谈 Prefill 算子成本。

### B4 ★ 和 vLLM Router / SGLang 比你的差异？
**口述**：token 级 + 全局精确索引；不靠请求历史、无冷启动；防字符-token 不一致 miss。  
**可转深**：机内 Radix/Paged 与跨实例亲和正交（见 09§3.2）。

### B5 ★ PD 分离下亲和怎么做？对 PFA/IFA 意味着什么？
**口述**：原生支持分离与混部。分离时 P 池跑 Prefill 算子，D 池跑 Decode 算子；路由要匹配「前缀 KV 在哪一类节点/索引」。混部则同池要防 Prefill 大块挤 Decode。  
**可转深**：07 PD 节 + OI。

### B6 ☆ 亲和命中率很高但吞吐不涨，为什么？
**口述**：命中主要减 Prefill；若集群已 Decode-bound（权重/KV 带宽、采样、锁），TTFT 好但吞吐不动。看阶段占比。

### B7 ☆ Mooncake 你做了哪一层？
**口述**：用 Conductor 全局索引做匹配与调度；传输引擎/存力细节是基础设施协作，非我 AscendC 交付。可承认读过传输与池化文档。

## C. Tool Call / Server / 综合

### C1 Tool Call 和结构化输出一起问怎么答？
**口述**：Tool Call 偏解析与协议；结构化输出偏生成期约束。可组合：工具参数用 Schema 约束。都不改 FA 主路径，属输出侧。半页：[`23`](./23-ToolCall与结构化输出交界.md)。

### C2 ★ 你没写过算子，为什么能做推理优化？
**口述**：优化=调度+正确性+与硬件特性对齐。我交付框架特性，并用 Roofline/算子选型理解指导「减 Prefill、PD、Graph、融合该不该上」。手写 kernel 是协作边界。  
**展示**：主动答 PFA/IFA、online softmax 一句、GE vs aclgraph。

### C3 给一个「既有亲和又有结构化输出」的慢请求，怎么查？
**口述**：拆 TTFT vs TPOT。TTFT：是否 miss 前缀、Prefill 是否打满。TPOT：batch、KV 长、bitmask/采样是否异常同步、是否 Host-bound。分层 profiling（07）。

### C4 DeepSeek V3 客户既要 Tool Call 又要性能，你怎么谈？
**口述**：服务侧我做过解析；模型侧 Decode 走 MLA/MoE 算子链（prolog/GMM/MC2），调度上注意 EP/亲和与 PD。不把自己说成 MLA kernel 作者。

## D. 指标与答辩数据

### D1 ★ TTFT -70% 与 E2E -50% 为何不一致？
**口述**：TTFT 几乎只看 Prefill；E2E 含全部 Decode。前缀命中主要砍 Prefill，故 TTFT 降幅通常大于 E2E。  
**加分**：说明场景是 4K、高前缀重复率——前提成立才有这个数。

### D2 如何证明不是「测平了」？
**口述**：同硬件对比开/关亲和；看命中率、Prefill 时长分布、实例负载；排除仅压测短 prompt。

## E. 速背卡片（面试前 5 分钟）

1. Prefill 贵 → 亲和省 Prefill → TTFT↓  
2. bitmask = Vector；错位 = 异步时序  
3. 编译缓存 = Host（**FIFO≈100，勿说 LRU**）；Graph = Host 下发；FA = Device  
4. token 级防假命中 → 避免错复用 KV  
5. load_gated 防 Decode 挤爆  
6. 边界：框架主场；算子懂选型与归因

## F. 交界补强题（15–20 落地后）

### F1 ★ PD 分离后 Decode 节点没有前缀 KV，传还是重算？
**口述**：比 `T_xfer(S,L,D_kv,B_net)` 与 `T_prefill(S)`。长前缀 + 够用带宽 → 传；短前缀/网络差/布局不兼容 → 重算。亲和尽量走本地命中。详见 [`16`](./16-跨节点KV传输与重算账本.md)。

### F2 ★ bitmask 和 top-p 谁先谁后？会不会进 aclgraph？
**口述**：先 mask 再 temperature/top-p/采样。尾段常图外；中间层可 Capture。错位是步进契约。详见 [`15`](./15-Sampler-Logits-约束解码脚印.md)。

### F3 Vocab parallel 时合法集怎么作用？
**口述**：合法集下标必须对齐 logits 布局——先 gather 再 mask，或按分片切 mask。详见 [`18`](./18-LMHead与Vocab并行.md)。

### F4 有人说 MLA Decode 已经是计算密集，你怎么回应？
**口述**：短/中上下文仍常访存（W_absorb）；极长 + MTP 才更可能翻转。详见 [`19`](./19-MLA-Decode-Roofline可信摘要.md)。

### F5 口误自检（FIFO / Mooncake / Tool）
**口述**：缓存 FIFO 不是 LRU；Mooncake 是索引协作不是传输作者；Tool 是解析不是算子优化。总表 [`24`](./24-答辩用词对照.md)。
