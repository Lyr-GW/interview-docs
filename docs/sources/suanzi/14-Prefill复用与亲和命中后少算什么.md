# 14 · Prefill 复用形态：亲和命中后引擎实际少算什么

> 简历说「前缀命中降 TTFT」。面试官常追：命中之后**还跑不跑 Prefill？跑多少？KV 怎么接上？**  
> 本文给出框架开发者够用的算子级图像（不绑死某一版 MindIE 私有实现细节）。

---

## 1. 三种复用强度（从弱到强）

| 形态 | 含义 | 算子大致行为 | TTFT 收益 |
|------|------|--------------|-----------|
| **A. 仅路由亲和** | 请求打到可能有前缀的实例 | 仍可能做满量或较大 Prefill，靠机内 cache 再省 | 中等 |
| **B. 前缀命中 + 跳过已缓存 token 的计算** | 只对 suffix 做 Prefill | 少跑多层 Linear/PFA（已缓存段） | 高 |
| **C. 多实例 KV 迁移后再算** | 远端把 KV 搬来 | 传输 vs 重算权衡；算子侧仍可能短 Prefill | 取决于带宽 |

你的简历强调 **token 级匹配 + 持续路由到持缓存实例**，叙事上更接近 **让 B 更容易发生**（同实例命中机内前缀/Paged KV），而不是吹自己做了 C 的传输栈。

**诚实口径**：  
> 调度保证请求落到持有前缀 KV 的实例；实例内由引擎/Paged/prefix cache 决定跳过多少 Prefill 计算。我优化的是「命中概率与负载」，用客户指标验证。

---

## 2. 「只 Prefill suffix」时算子链怎么变

假设前缀长度 Lp，后缀 Ls，命中前缀：

```
满量：Prefill(Lp+Ls) 全部 token 进 PFA/Linear
命中：可能只对 Ls 做 Prefill，Attention 读已有 KV（paged）
```

| 模块 | 满量 | 前缀命中（理想） |
|------|------|------------------|
| Embedding/Linear×L | 对 Lp+Ls | 主要对 Ls |
| PFA | Q 长度≈Lp+Ls | Q 长度≈Ls，K/V 含历史 |
| scatter 写 KV | 写全部 | 主要追加 Ls |
| 后续 Decode | 同 | 同 |

**OI 直觉**：suffix Prefill 的 S_q 变小，单次 Prefill 更短 → TTFT 降。若 Ls 仍很大，收益取决于 Lp/(Lp+Ls)。高前缀重复率 = Lp 大、Ls 相对小 → 与简历场景一致。

---

## 3. 和 PagedAttention 的衔接

```
历史 KV 已在某实例的 block 池中（block_table 指向有效页）
新请求 suffix --> 引擎分配新 page --> scatter 追加
PFA/IFA 通过 block_table 读「旧页+新页」
```

亲和负责：**别把请求派到没有这些 page 的实例**（否则只能重算或走迁移）。

---

## 4. 和「字符级假命中」对比

| | token 级真命中 | 字符级假命中 |
|--|----------------|--------------|
| KV 内容 | 与新请求前缀一致 | 可能不一致 |
| 若强行复用 | 正确加速 | **正确性风险** / 必须重算 |
| TTFT | 降 | 可能不降甚至更差 |

这就是简历里坚持 token 级 + 全局索引的算子后果。

## 5. PD 分离下的额外问题（加分）

- Prefill 节点算出的 KV 要给 Decode 节点 → 涉及传输（Mooncake 等）；  
- 亲和可能作用在 P 侧（前缀复用）或联合索引；  
- 你「支持 PD 分离与混部」：面试承认分离时还有 **传输 vs 重算** 权衡，混部时是 **争用 vs 本地性** 权衡。

详传输见 [`16-跨节点KV传输与重算账本.md`](./16-跨节点KV传输与重算账本.md)；背景材料：`docs/kv knowledge/`（尤其 `06-vLLM-Mooncake-Motor`、`07-亲和与三级池化交互`、`10-昇腾HCCL与KV传输`、`11-KV缓存利用率与假命中`）。本文只要求能说到「调度与算子/存储的交界」。

---

## 6. 白板一句话

> 亲和提高「请求与已有 KV 页」的共址概率；TTFT 降是因为 Prefill 算子链少算了前缀段；Decode 算子链还在，所以要 load_gated。
