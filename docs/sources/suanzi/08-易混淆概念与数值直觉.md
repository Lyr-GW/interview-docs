## 1. 易混淆对照表（面试防踩坑）

| 容易混在一起 | 正确区分 |
|--------------|----------|
| **GEMM vs GEMV** | 都是矩阵乘；GEMV 是 M≈1 的窄 GEMM，Decode 单请求典型形态 |
| **计算密集 vs 跑得快** | 计算密集只说明瓶颈在算力；绝对 FLOPs 大反而可能更慢 |
| **OI 高 vs MFU 高** | OI 是算/搬比值；MFU 才是「硬件有没有在做有用功」 |
| **aclgraph vs GE** | 运行期省 Host vs 编译期省 Device；不是谁替代谁 |
| **aclgraph 融合？** | **不融合**；录的是原样 Kernel。融合靠手写大算子 / FX / GE |
| **FA vs PagedAttention** | FA=怎么算；Paged=KV 怎么存。常组合，不是二选一 |
| **PFA vs IFA** | Prefill 切 Q；Decode 切 KV。同一套 online softmax，tiling 相反 |
| **safe softmax vs online softmax** | 前者要看整行减 max；后者分块维护 running max/sum，数学等价 |
| **图融合 vs UB 融合 vs SuperKernel** | 改拓扑合并 / 中间留 UB / 多 kernel 一次启动——省的东西不同 |
| **大 batch 救 FFN vs 救 Attention** | FFN：拼 M 让单核变快；Attn：不能跨请求拼 KV，主要打满多核 |
| **MLA fractal 满 vs 整步计算密集** | M=128 只说明 Cube 不浪费；短上下文 W_absorb 仍可让整步访存密集（详见 [`19`](./19-MLA-Decode-Roofline可信摘要.md)，勿背「一律计算密集」） |
| **TP vs SP** | TP 切权重维；SP 切序列维（常配 LayerNorm）。TP **不切** M |
| **TP vs DP vs EP** | 共算一请求降延迟 / 多副本扩吞吐 / 专家分卡 |
| **W 量化 vs KV 量化** | 救权重搬运与显存 vs 救 KV 显存/长上下文读带宽 |
| **per-tensor vs per-token** | 一个 scale 快但粗 vs 每行 scale 精但贵 |
| **Host-bound vs memory-bound** | Host 下发跟不上 vs Device 在等 HBM。救法：Graph vs 量化/少搬 |
| **mla_preprocess vs mla_prolog** | Prefill 侧编码写 latent vs Decode 侧准备 Q'/K |
| **dispatch vs combine** | EP 把 token 发出去 vs 把专家输出收回来加权 |

## 2. M / N / K 三维（矩阵乘必懂）

任意 `C = A × B` 写成：

```
A[M, K] × B[K, N] → C[M, N]
     ↑        ↑         ↑
     行数      归约维     列数
```

| 符号 | FFN 里常指什么 | Attention BMM1（Q×Kᵀ）常指什么 |
|------|----------------|----------------------------------|
| **M** | token 数（batch×seq）| Q 的 token 数（Prefill 大，Decode≈1）|
| **K** | hidden / 中间维 | head_dim D |
| **N** | 输出维（如 intermediate）| KV 长度 S_kv |

**Cube 怕的是 M 太小**（fractal 要凑 16 行），不是 K、N 小。K/N 大只说明「有东西可切、可累加」，填不满 M 照样浪费。

---

## 3. 数值直觉①：大 batch 如何抬 FFN 的 OI

设 `C[M,N]=A[M,K]×B[K,N]`，FP16，只关心权重 B 的搬运（主导项）：

```
B 读取 ≈ K×N×2 字节（与 M 无关）
FLOPs  ≈ 2×M×K×N
OI_粗  ≈ FLOPs / (K×N×2) = M
```

| M | 粗算 OI（仅摊权重）| 相对单 token |
|---|-------------|--------------|
| 1 | ~1 | 1× |
| 32 | ~32 | 32× |
| 128 | ~128 | 128× |

> 真实 OI 还要计入 A/C 搬运，但数量级直觉成立：**M 拉大，权重被摊薄**。  
> 昇腾 910B 屋脊点约 209：M 到几十～上百，FFN 才有机会接近/超过屋脊点。

## 4. 数值直觉②：Online Softmax 手算（两块）

一行 score，切成两块：`S0=[2, 1]`，`S1=[3, 0]`（已含 scale，忽略 mask）。

**一次性 safe softmax（标准答案）**

```
整行 max = 3
exp = [e^{2-3}, e^{1-3}, e^{3-3}, e^{0-3}] = [e^{-1}, e^{-2}, 1, e^{-3}]
sum ≈ 0.368+0.135+1+0.050 = 1.553
P   ≈ [0.237, 0.087, 0.644, 0.032]
```

**Online 分块（应得到同一分布）**

块0：
```
m0 = max(2,1) = 2
l0 = e^{0} + e^{-1} ≈ 1 + 0.368 = 1.368
```

块1：
```
m1 = max(m0, max(3,0)) = max(2,3) = 3
scale = exp(m0-m1) = exp(-1) ≈ 0.368
l1 = scale·l0 + (e^{0}+e^{-3}) ≈ 0.368×1.368 + 1.050 ≈ 1.553  ✓
```

概率用最终 `m1,l1` 回推，与一次性结果一致。  
**要背的只有**：`scale=exp(m_old-m_new)` 用来缩旧的 `l` 和 `O`。

## 5. 数值直觉③：Decode Attention 为何「搬不动」

单 head、FP16、S_kv=4096、D=128：

```
读 K+V ≈ 2 × 4096 × 128 × 2B ≈ 2 MB（每 head）
若 32 head 且不共享：×32 → 约 64 MB / 步 / 层（量级示意）
计算：Q 只有 1 行，FLOPs 相对搬迁很小 → OI 低
```

GQA（8 KV head）或 MLA（~576 维）直接砍这笔搬运。这就是「架构级省带宽」。

## 6. 数值直觉④：Host-bound 何时出现

经验口径（口述用）：

- 单个 kernel Device 执行 **≪** Host 下发耗时 → 队列空泡 → Host-bound；
- Decode、小 batch、层数多、小算子碎 → 更容易；
- Prefill 大 kernel 往往 Device 侧已经很忙 → Graph 收益相对小。

**判断口令**：profiling 看「CPU 等待 / 下发间隙」大，而 NPU 利用率低 → 先 Graph；NPU 忙且 HBM 打满 → 先量化/融合/减 KV。

## 7. IFA 为什么要 Reduce（一句话说透）

Prefill（PFA）：每个核负责**不同的 Q 行块**，各自输出自己的 O 行，**不用合并 softmax 状态**。

Decode（IFA）：Q 只有 1 行，却把 **KV 切给多个核**。每个核只有局部 score → 局部 max/sum/O。  
必须把各核的 `(m,l,O)` 按 online softmax 规则再合并一次，才得到全局正确输出。  
**所以 IFA 多了跨核 reduce；PFA 通常没有。**

## 8. PagedAttention：写与读各谁干

| 动作 | 常见算子 | 说明 |
|------|----------|------|
| 本步新 K/V 写入非连续 block | `scatter_pa_kv_cache`（或融入 rope_cache）| 按 `block_table` 散列写 |
| Attention 计算 | PFA/IFA/FIA | **很多实现直接按 block_table 在 kernel 内 gather 读**，不一定先调用 `gather_pa_kv_cache` |
| 显式 gather | `gather_pa_kv_cache` | 需要连续 KV 的路径才用 |

不要说成「每次都先 gather 再 FA」——那只是一种实现。

## 9. TP Column / Row（配文字图）

```
Column Parallel（切输出）：
  X[M,H] × W[:, 0:N/2]     → 部分输出，常无通信
  X[M,H] × W[:, N/2:N]

Row Parallel（切输入后要加总）：
  X0 × W0 + X1 × W1  → AllReduce → 完整 [M,H]
```

- Attention：QKV 常 Column（按 head 切），O 常 Row（要 AllReduce）→ `matmul_all_reduce` 吃 Row。
- FFN：gate/up Column，down Row。

**M 始终完整**：TP 不把序列切短（切序列是 SP）。

## 10. W8A8 四个字展开

| 符号 | 含义 |
|------|------|
| W8 | 权重存 INT8（常静态标定）|
| A8 | 激活用 INT8（常 per-token 动态量化）|
| 计算 | Cube/TensorCore 走 INT8 吞吐更高，或至少搬运减半 |
| 配套 | 需要 scale；出结果常要 dequant 回 FP16/BF16 给后面层 |

KV INT8 是另一件事：缓存里的 K/V 低比特，Attention 读时 dequant 或用支持量化 KV 的 FA。

## 11. 与简历相关的易混（再加 6 组）

| 容易混 | 正确区分 |
|--------|----------|
| **亲和加速 vs 算子加速** | 亲和减少 Prefill **调用次数/长度**；算子加速是单次 PFA 更快 |
| **机内 prefix cache vs 跨实例亲和** | 同机复用 vs 路由到有 KV 的实例；正交 |
| **编译缓存 vs aclgraph** | 省 Schema 编译 CPU vs 省 kernel 下发 |
| **TTFT 优化 vs TPOT 优化** | Prefill/排队 vs Decode 带宽/batch/Graph |
| **bitmask 开销 vs Prefill 开销** | Vector 小头 vs 计算密集大头 |
| **假命中 vs miss** | 假命中可能错用 KV（正确性）；miss 只是慢 |
| **FIFO 编译缓存 vs LRU** | 实现是 **FIFO≈100、命中不调序**；勿说 LRU/128（[`24`](./24)） |
| **Tool Call 解析 vs 结构化约束** | 生成后解析 vs 生成期 bitmask（[`23`](./23)） |
