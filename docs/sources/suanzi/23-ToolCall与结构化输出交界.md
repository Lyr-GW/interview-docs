> 深挖解析：`docs/interview-review/14`、`17`；算子脚印：[`15`](./15)、[`11`](./11) §4。

---

## 两句话分清

| | **Tool Call / Reasoning 解析**（你的交付） | **结构化输出 / 约束解码**（你的交付） |
|--|---------------------------------------------|----------------------------------------|
| 何时 | 模型**已经生成**文本/标记之后 | **生成过程中**逐步约束合法 token |
| 做什么 | 按模型族模板抽出 tool 名、参数、reasoning | Schema→自动机→bitmask→采样 |
| 算子脚印 | **基本不进** Prefill/Decode 主算子链 | 落在 Decode **Step10**：logits 后（[`15`](./15)） |
| 主风险 | 模板漂移、多模型族兼容 | 异步 mask 错位、假合法集 |

**金句**：解析是**后处理**；结构化输出是**生成期约束**。可以叠用（工具参数也要求合法 JSON），但不是同一个模块。

---

## 叠用时怎么走

```
… → LM Head → (可选) Schema bitmask → sample → token
                                      ↓
                         拼出完整 tool 调用文本后
                                      ↓
                         Tool Call 解析器抽参 / 校验
```

- 工具参数若声明 JSON Schema → **生成期**可走你的结构化输出链路，减少「生成完再 parse 失败」。  
- 仅解析、不约束 → 仍可能出非法 JSON，靠重试或修复（质量/延迟账）。

## 面试收口

- DeepSeek V3 客户既要 Tool Call 又要性能：服务侧谈**解析**；模型侧性能转 MLA/MoE（`03`/`04`），**不说自己写过 mla_prolog**。  
- 与亲和正交：Tool 注入会改变 prompt → 更要 **token 级**匹配防假命中（`14`）。
