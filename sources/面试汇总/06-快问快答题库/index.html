    A. 结构化输出 / 编译缓存¶

    问 30 秒答点
    结构化输出解决什么？ 采样期硬屏蔽非法 token（logit→−inf），从「prompt 大概率合法」升级为「结构上必然合法」。
    链路一句话？ Schema→EBNF→字节级 PDA→mask cache；运行时 PDA→bitmask→apply→采样。
    为何说 PDA 不说 FSM？ JSON 无界嵌套要用栈；正则可退化成 FSM，但仓内笼统写「FSM」不严谨，正式说**字节级 PDA**。
    为何选 xgrammar？ PDA + ~99% 预计算；C++ 内核、bitmask 与设备解耦，NPU 只接 masked_fill_；Outlines 递归/嵌套弱。
    MindIE 有几个结构化后端？ 唯一落地 xgrammar；guidance 是扩展位/接口预留未接线；「真多后端」是 vLLM。
    编译缓存口径？ key=SHA-256；默认容量 100；FIFO（命中**不**调序）；禁说 LRU / 128。
    为何 FIFO 够用？ tool/Agent 的 schema 近乎全热，FIFO≈LRU；dict+next(iter) 实现最简。
    编译缓存 vs vLLM？ MindIE=按**条数** FIFO/100；vLLM 把上限下沉进 xgrammar 的**字节上限**（约 512MB 量级）。
    编译缓存 vs KV PrefixCache / Radix LRU？ 别串：FIFO/100 是「SO 编译产物」条数策略；PrefixCache/Radix LRU 是「KV 块」驱逐，层不同。
    开销分几段？ 冷编译打 TTFT：简单 +5–15ms、复杂 ~100–200ms；热路径 mask 常 <1%–3%（估计）可与 forward overlap。
    bitmask 在 NPU 怎么落？ int32 位图→repeat_interleave→masked_fill_(-inf)；torch 组合，非自研 AscendC；fused 是可优化点。
    bitmask/结构化在图里吗？ 常因动态控制流留**图外**；别假装整图包了 mask；中间稳定层可 Capture。
    MTP × 结构化？ InferParam 入口硬互斥；插件未打通（无 rollback、单位置 mask）；勿说已打通。
    async 调度 mask 为何错位？ 三类根因：线程时序 / 游标（num_tried_tokens）/ PD sync；修在 forward_loop 生成，验收标准是**正确性**不是 TPOT−x%。
    为何还要 parser？ 约束管「合法性」，parser 管「字段抽取 / 流式增量」，职责正交。
    四层代码是哪些？ manager / grammar / bitmask / GuidedDecodingLogitsHandler（pta_handlers）。

    练习题（盖答，抽背） - 默认容量多少？FIFO 还是 LRU？命中会不会调序？为什么不是「热 schema 更抗踢」？ - 为什么说是字节级 PDA 而不是随口 FSM？guidance 是真多后端还是接口预留？ - 画编译缓存图：100 + FIFO + 命中不调序，标出「新插入从哪进、踢谁、命中指针动不动」。

    B. Function Call¶

    问 30 秒答点
    全链路一句话？ Encode(tools→template)→Generate→Decode(Reasoning + ToolCallsProcessor，finish_reason=tool_calls)。
    为何每模型族一个适配器？ 无统一协议：Qwen3 XML、DS V3 token+json、V3.2 DSML <invoke>，格式各异。
    流式为何用 token 计数？ 半标签 / 半多字节截断会让正则误判；按 token ID 计数 O(1) 且与生成粒度对齐。
    4-Case 状态机？ 用 start/end special token 的**出现次数**驱动阶段，而非对全文跑 regex。
    JSON Completor？ 自研递归下降 Full/BraceOnly；对残缺 arguments 做增量补全；零三方依赖。
    Hard Cut-off？ DSML：end token 之后永久返回空 delta，挡住块外幻觉。
    解析失败兜底？ 五层软降级，绝不 500；根治靠约束生成（SO）。
    vs vLLM 最大差别？ MindIE=token 计数；vLLM Hermes=每步全文 regex（通用但 O(n)）。
    与结构化如何集成？ vLLM：tool_choice→guided + structural tag；MindIE 两条路未打通、无 structural tag。
    FC vs SO 一句区分？ FC=协议解析的软保证；SO=生成期硬约束；可组合：用 SO 硬约束 arguments 的 Schema。
    Schema-aware coercion？ DSML 依据 tools schema，把 XML 里的字符串转成数值/布尔。
    残缺 JSON 谁更自研？ MindIE 自研 Completor；vLLM 用 partial_json_parser（Hermes 实为字符串 diff）。

    练习题（盖答） - Tool Call 链一句 + 约束 ⊥ parser + 流式 4-Case 各说清。 - 半标签截断为什么用 token 计数而不是 regex？举一个会误判的例子。

    C. KV 亲和（Motor）+ Mooncake¶

    问 30 秒答点
    Motor 一句话？ tokenize 前置→Conductor 做**精确最长前缀**→unified/load_gated 打分→Scheduler 权威仲裁。
    为何做到 token 级？ 字符级易被 tools/template 改写骗出**假命中**；token 级与引擎哈希同构。
    为何要中心索引？ 本地树无驱逐感知→假阳性；Conductor + kv-events 才是真索引。
    tokenizer 从哪加载？ Coordinator 本地 model_path 同源；禁说「运行时从引擎拉」。双 tokenize 只影响命中，不改输出。
    tokenize≈6ms 值不值？ 4K≈6ms 墙钟 ≪ 一次误判满量 Prefill；是经验墙钟不是 P99。
    TTFT−70% 怎么说？ 代表性测算：Qwen3-32B、8K、共享≈6.5K，h 0.10→0.78，≈1187→351ms；主因 Δprefill；非客户 raw log。
    TTFT 五段分解？ tokenize + Conductor + queue + prefill + delivery；KV 亲和主砍的是 prefill。
    Conductor 超时？ 客户端预算约 0.2s 硬上界→回退 LoadBalance；常态 ms 级；P50<5ms/P99<20ms 是**目标非实测**。LB 不是纯轮询。
    unified vs load_gated？ unified：亲和 + 负载统一打分；load_gated：先负载门控（Top-N）再最长前缀——防热点抵消 Prefill 收益。
    五参数记什么级？ mode(默认 unified) / load_weight / overlap_credit / prefill_load_scale / load_gate_topn（字面 0→语义 2）；三权重默认 1.0，topk=3；只背角色量级，勿背假生产最优。
    防 herding 三版？ overlay 打散 → top-k 随机/选优 → unified 全局重排；top-k 只在候选集内重排。
    D 实例注册 Conductor？ 不注册；索引只挂 P/U；Decode 消费已绑定 KV，再抢亲和会打乱负载。
    什么叫「假命中」？ 路由以为有前缀 KV，落地无块/已驱逐→仍付 Prefill 或失败；比假阴更伤 TTFT。
    假阳 vs 假阴偏哪边？ Motor 偏**安全假阴**（查真索引+Removed 事件）；近似树易假阳。
    KV Events 是什么？ 索引与引擎块生命周期同步的**通知面**（非传 KV 数据面）：Stored=可复用上报、Removed=驱逐→删索引防假命中。
    KV Events 传输形态？ ZMQ PUB + 必要时 replay；per-DP-rank 发布对齐并行副本；丢包走 replay/超时回退，不硬猜仍命中。
    引擎 PrefixCache vs Motor 亲和？ PrefixCache=引擎内/进程内**按哈希链复用块（APC、ref_cnt 防用中驱逐）；Motor=**跨实例查 Conductor；两层正交，开 prefix caching ≠ Motor。
    Mooncake 三件套？ Conductor=前缀索引控制面 / Store=KV 池 / TE（Transfer Engine）=传输；Motor 只做「查 Conductor 决策」这一步。
    Connector 是官方三层之一吗？ 否；Connector 是引擎侧适配器，≠ Conductor/Store/TE。
    Store 记账/驱逐？ Master 记账、Client 搬数；near-LRU + soft/hard pin/busy 优先级；「Conductor 删冷 KV」错——删 KV 是 Store 的事。
    传 vs 重算怎么判？ 比 T_xfer≈|KV|/BW 与 T_prefill：长前缀+够带宽→传；短/差网→重算；重算∝p²、传∝p，长前缀更倾向传。
    为何不用 LMCache？ 栈已选 Mooncake+Conductor+Ascend Connector；LMCache 是平级竞品，非缺省。
    Conductor 重启窗口安全吗？ 残缺索引 → 假阴性安全退化 LB；不像本地树假阳性无界。

    练习题（盖答） - 近似树「字符前缀很长」为何仍可能灌到冷副本？Stored 与 Removed 各解决什么？ - 何时选混部 U、何时 P+D？handoff 与 concurrent/layerwise 的 TTFT 直觉差在哪？D 为何不注册？ - 默写 unified 打分式 + load_gated 两步；旁注五参数默认值。 - 画超时回退链：触发条件 → 亲和 None → LoadBalance →（再失败）round_robin；标 200ms=配置。

    D. 投机解码¶

    问 30 秒答点
    为何能加速？ decode 是 memory-bound：便宜 draft 猜 k 个 → target 一次 verify；用闲算力换步数。
    「无损」指什么？ 拒绝采样后分布 ≡ 纯 target；不是「永远加速」。
    拒绝采样四步？ draft 猜→target 并行验→接受概率 α=min(1,p/q)→拒则从 norm(max(0,p−q)) 重采样（可加 bonus）。α 那步保证同分布。
    单 token 延迟？ (T_draft+T_verify)/τ，要 < T_target 才划算。
    大 batch 为何失效？ 变 compute-bound，verify 抢算力→吞吐反降；接受率也易掉。
    演进线？ EAGLE（特征自回归/树草稿）→ MTP（训练期多 token 头）→ DSpark（半自回归 Markov + 按负载裁 k）。
    MindIE verify 方式？ 贪心逐位比对；≠ vLLM 默认的概率拒绝采样。
    与结构化？ 入口硬互斥；插件未打通；vLLM 有多位置 mask + rollback。
    高温对接受率 τ？ 熵↑→min(1,p/q)↓→τ↓，可能负加速。
    能报加速比吗？ 无自测曲线**不报 %**；只讲 draft-verify 机制 + 接受率失效场景。
    DSpark 怎么说才不踩线？ 两词拆开讲机制，卖点相对 MTP 一句；加速比只引**公开出处**，不说「我线上测过」。
    SGLang Spec V2 / overlap？ SGLANG_ENABLE_SPEC_V2、eagle-topk=1、draft/verify 走 overlap schedule；未生产压测，只讲开关不报自制加速比。
    树命中 = 投机接受？ ≠：Radix 树砍重复 prefill；Spec 砍 decode 步延迟，别混。

    练习题（盖答） - 默写拒绝采样四步，标出哪一步保证与原模型同分布。 - 大 batch 接受率掉了，怎么一句话解释而不是报假加速比？ - MindIE 三插件能背出哪三个名字？相对 MTP/DFlash 各差一句。

    E. vLLM 配置与 Scheduler¶

    问 30 秒答点
    十类配置口诀？ 缓存2（prefix/亲和）+ 批3（chunked/max_num_seqs/max_num_batched_tokens）+ 算3（量化/CUDA Graph/并行）+ 猜1 + 拆1(PD)。
    降 TTFT 先开啥？ prefix caching、chunked prefill、PD 分离、KV 亲和、量化减 prefill。
    提吞吐主旋钮？ 先扫 max_num_seqs / max_num_batched_tokens 两个。
    CUDA Graph？ V1 默认 piecewise，decode 消 launch 开销；只有调试才 enforce-eager。
    chunked prefill 解决什么？ 把长 prefill 切块与 decode 混批，压掉 HOL、稳住 TBT。
    chunked vs PD？ 同题不同解：chunked 低成本混部；PD 隔离干扰+独立扩缩，代价是传输。
    vs SGLang 调度？ vLLM continuous+chunked；SGLang overlap schedule 更早；前缀：block 哈希 vs Radix 树。
    waiting_by_reason？ capacity=KV/并发不足；deferred=LoRA/KV transfer 等待。
    编译未完怎么调度？ vLLM 用 Future 非阻塞，未编完的请求留 WAITING、不进 batch。
    Scheduler 与 bitmask？ mask 常在 scheduler/CPU 侧生成，worker 只 apply。
    enforce-eager 何时用？ 只在调试 / 动态图问题时；生产别默认关 CUDA Graph。
    PD 配置入口？ kv-transfer-config 等；D 侧可等远端 KV 就绪。
    怎么开配置最稳？ 按目标选（TTFT→prefix/chunked；吞吐→CG/预算；显存→量化/KV 量化），忌背清单。

    练习题（盖答） - 按目标（降 TTFT/提吞吐/省显存）各举两类旋钮，各用半句讲背后原理。 - chunked prefill 要解决的 HOL 是什么？和抢占/recompute 怎么分工？

    F. 量化 / PD / Profiling¶

    问 30 秒答点
    量化三角？ 显存/带宽/算力权衡：小 batch 延迟→W4A16；大吞吐→FP8/W8A8。
    W8A8 vs W4A16？ W8A8 算力+带宽双收；W4A16 强压权、decode 带宽友好但有反量化税。
    GPTQ vs AWQ？ GPTQ 最小化重构误差；AWQ 护激活大通道，LLM 上常更稳。
    FP8 四要素？ scale、粒度、KV dtype、attention 用 FP32 累加。
    KV 量化收益？ 长上下文 slot↑、decode 带宽↓。
    量化验收三层？ PPL → 标准 bench → 业务 A/B；归因二分→分项→逐层。
    PD 主收益？ 干扰消除 + 独立扩缩，不是「传一定比算快」。
    传 vs 重算临界？ T_tx≈|KV|/BW vs T_recompute；短 L / 无 RDMA 倾向混部；带宽阈值 ~6/19GB/s 是**论文量级**非自测。
    Prefix × PD？ 只传 delta suffix 的 KV。
    Profiling 四层？ metrics → 排队/KV → step 形态 → nsys/msprof。
    吞吐上不去第一步？ 看 waiting vs running、kv_usage、preempt；一半问题是配置。
    nsys 见 GPU 空白？ CPU/launch-bound → 先上 Graph，先别换 FA。
    −70% 归因？ 主因 Δprefill（亲和命中），不是算子 magically −70%。

    练习题（盖答） - FP8 四要素默写；反问：哪个场景该用 W4A16 而不是 FP8？ - 吞吐上不去，Profiling 从哪层入手？GPU 空 vs GPU 满怎么分叉？

    G. 算子（Prefill/Decode · FA · Graph · MoE）¶

    问 30 秒答点
    Prefill vs Decode 算子？ Prefill 计算密→PFA（切 Q）；Decode 访存密→IFA（切 KV + reduce）；多核策略相反。
    decode 为何 Cube 利用率低？ M=1，fractal 仅 1 行有效（≈1/16），且要等 KV 搬运。
    FA / online softmax？ tiling + running max/sum，数学 ≡ safe softmax，才能分块。
    FA 反向为何 recompute？ 前向不存 S/P 省显存，反向用 m/l 重算——算力换显存。
    Decode 为何开 Graph？ 小 batch 多层易 Host-bound，Capture&Replay 省 dispatch，不改 kernel 算力。
    Graph 与 PagedAttention 矛盾？ Graph 要固定 shape/地址，block_table/seq 每步变；解：FULL+metadata、PIECEWISE、padding 到 capture sizes。
    aclgraph vs CUDA Graph vs GE？ aclgraph≈CUDA Graph 思想（省 Host）；GE=编译期融合（降 Device）；可叠加；不吹手写 AscendC。
    Norm/RoPE 为何融合？ Vector 小算子启动 + UB↔GM 往返大，融合留片上——也是 GE 收益源。
    EP / All2All / MC2 一句？ EP 切专家；All2All 搬 token；MC2 通算融合藏通信。
    128 选 6、8K 每专家？ 8K×6/128 = 384 token（均值），是白板量级非自测。
    为何偏 EP+完整专家+GMM？ 小专家再大 TP → GEMM 更碎、Cube 饿；EP+all2all/MC2 更贴。
    MoE 门控链路口令？ 门控 → permute/重排 → GMM → finalize/还原。
    −70% 的算子解释？ 省的是整条 Prefill 链（PFA+大 Linear），不是 PFA kernel 本身加速 70%。
    没写 AscendC 怎么谈？ 用 Roofline + 选型协作谈；L4/L5 调度与特性动 TTFT/正确性；不谎称生产 kernel。

    练习题（盖答） - 为什么 decode 想开 Graph？与变长 block_table 的矛盾怎么解？ - EP/All2All/MC2 各一句；128 选 6、8K 每专家均值怎么算？能报自家 MoE 加速比吗（不能）？

    H. Dynamo / 竞品一句对标¶

    问 30 秒答点
    Dynamo 一句话？ Frontend + KV Router + KVBM + NIXL；--router-mode kv 用 cost 选 worker。
    cost 公式核心？ adjusted_prefill = raw − 分层 overlap credit；cost = scale×adjusted + decode_blocks。
    默认权重直觉？ Device 1.0 / Host 0.75 / Disk 0.25（介质延迟比）；shared 默认 0。
    KVBM G1–G4？ Device→Host(D2H)→Disk(NIXL)→Remote(NIXL)；亲和权重进同一 cost 公式。
    PD 时 overlap？ Prefill 完整 overlap；Decode overlap=0。
    approximate 模式？ 无 kv-events 时用预测 + TTL（默认 120s），有假命中风险。
    Dynamo vs Motor？ Dynamo=栈内 cost+tier 权重（平台绑定）；Motor=tokenize+Conductor precise lookup（旁挂精确查）。
    Dynamo vs llm-d？ Dynamo 统一 KVBM+cost；llm-d EPP 插件化 + K8s，PD 常用 NixlConnector。
    NIXL vs Mooncake？ NIXL=传/存抽象（G3/G4）；Conductor=前缀索引控制面——别混层。
    vs vLLM/SGLang Router？ 常见字符级近似树=猜；Motor=token 级真索引+驱逐事件=查。
    vs AIBrix？ 自研分层/打分多，Mooncake 非其中心路径；仍用「猜 vs 查」框架对标。
    语义路由 vs 实例级亲和？ 语义路由是**模型级**选强弱/预算（δ=maxsim_hard−maxsim_easy 切档）；Motor 是**实例级**选副本；两层正交可叠。
    没部署过的竞品名？ 不编 API；迁移到「猜缓存 vs 查索引」框架；反问是要精确命中还是省中心。

    练习题（盖答） - cost 公式白板有哪几项直觉？KVBM G1–G4 怎么和 Motor「查索引再打分」一句对标？ - prefixaware 算不算语义路由？（不算，实例级近似前缀，别答串）

    I. 简历数字诚实 / 高压应急¶

    问 30 秒答点
    TTFT−70% 是客户实测吗？ 否，代表性测算（公式可白板：TTFT≈c_0+T_prefill(1−h)）；无 raw log，不说客户实测。
    E2E−50% 任意负载成立吗？ 不成立：短 decode 才接近；长输出稀释；同场景测算约 −48.9%。
    5000+ / 「约一万行」怎么拆？ SO 累计 0→1（多 PR+测），现状 wc≠累计；Server 讲 Handler/Interface 路径 + UT/IT；cloc / before-after 待补，禁把 2320/7239 说成删行数。
    编译缓存多少？LRU？ 默认 100、FIFO、命中不调序；禁 LRU/128。
    tokenize 6ms 与 200ms 超时？ 6ms 是经验墙钟；Conductor 0.2s 是**客户端预算 ≠ 已测 P99**。
    门控默认背哪些？ mode=unified；三权重默认 1.0；topn 字面 0→语义 2；topk=3；勿背假生产最优。
    「70% 是客户日志？」 代表性测算，可白板假设，无 raw log，要数走复测。
    「缓存不是 LRU128？」 现行 FIFO/100，旧口径过时，命中不调序。
    「这个你没测过？」 承认无 A/B，讲测法，标**待补**，不报假 P99/加速比。
    「一年就跳是不是不行？」 拉力 > 推力：方向收窄换赛道；先认跟踪缺口再切 SO+亲和；禁涨薪/嫌加班。
    被问穿不会怎么办？ 边界诚实 → 迁到已知（SO/亲和）→ 反问场景；禁编。
    失分 15s 救场句？ 「这块跟踪不够系统。工程交付有把握——SO 或亲和，您先听哪个？」

    练习题（盖答） - 「你简历数字是不是包装的？」给 2–3 句应急模板（测算 / 待补 / 未压测），不道歉式崩溃。 - 「你的缺点是什么？」选一个有补强路径的弱项，60s 讲「暴露 → 怎么补 → 本夜卡片」。 - 「和跨团队意见冲突怎么处理？」给一个接口边界/口径/oncall 的 60s STAR，结尾落到可验证证据。

    J. 跨板块串线（易翻车）¶

    问 30 秒答点
    Agent 循环里 SO+FC+KVA 怎么串？ FC 定协议；SO 硬约束 arguments；KVA 复用多轮 tools 前缀——三条正交。
    为何 PD 的 D 不注册 Conductor？ Decode 消费已绑定 KV，再抢亲和会打乱负载；索引只挂 P/U。
    投机+约束：产品互斥还是工程不能？ 两边都有：入口 fail-fast（产品）；根因是 rollback/多位置 mask 未做（工程）。
    −70% 别归因算子？ 测算砍的是 prefill 工作量（命中率）；PFA kernel 本身没 magically ×0.3。
    NIXL 有了还要 Mooncake？ NIXL=传/存抽象；Conductor=前缀索引控制面——层不同。
    为何不 Dynamo 全栈？ 同一问题不同形态：我们交付链是 Motor precise + Mooncake，不是缺 Dynamo。
    bitmask 要不要进 aclgraph？ 动态控制流常留图外；图内收益在稳定中间层；先正确再谈 fused 采样。
    热路径 TPOT 与冷编译关系？ 热 <1%–3%（估计）；冷 100–200ms 打 TTFT；缓存 FIFO/100 保热。
    假命中和 −70% 的关系？ 假阳吃掉 Prefill 节省 → 测算收益缩水；Motor 偏安全假阴 + Removed 事件。
    假命中 ← 靠什么感知？ ZMQ Removed 事件；Motor 无本地近似树；假阳 ≠ round-robin。
    「删了一万行」和「5000+」怎么不混？ 一万行=Server 重构（Handler/Interface，LOC 待补）；5000+=结构化交付；两者别相加。

    练习题（盖答） - 树命中、投机接受、prefix caching 命中三者有何本质区别？ - HCCL 能传 KV 吗？（HCCL=集合通信；点对点 KV 走 TE/Connector）
