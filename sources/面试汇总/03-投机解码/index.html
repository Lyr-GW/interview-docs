    投机解码（Speculative Decoding）面试汇总¶

    本篇把「拒绝采样白板 + 演进线 + MTP/MindIE 三插件 + DSpark + 结构化互斥」整合成一篇内容型正文，按主题组织，可背可默写。 口径约定：加速比数字一律标来源（论文/公开），凡涉及 DSpark 等公开数字均标「论文跟读、非本仓生产实测」；机制类结论标 [机制]，文档已核实结论标 [文档]，代码核实结论标 [代码]。无自家压测曲线，绝不把公开百分比说成「我线上测过」。

    一、draft-verify 与拒绝采样（白板核心）¶

    1.1 动机：decode 是 memory-bound¶

    自回归解码一次前向只出 1 个 token。decode 阶段（小 batch）是 memory-bound：每步都要把全部权重从 HBM 搬进计算单元，算力大量闲置。投机解码就是拿这份**闲置算力**做交易——用便宜的 draft 猜 k 个 token，再让 target 一次前向并行验这 k 个，把「k 次 target 前向」压缩成「1 次 target 前向 + k 次廉价 draft」。[机制]

    1.2 白板四步（可默写，约 90–120 秒）¶

    从左到右四格，边写边说：

    步① Draft（草稿）
    上下文 c 已知
    draft 模型（小模型 / 草稿头 / MTP 层…）自回归或并行块
    产出 x1, x2, …, xk，并记下各步草稿分布 q_i（或等价 logit）
    口述：draft 便宜、可错；目标是抬高「一次 verify 覆盖的期望长度」

步② Target Verify（验证）
    target 对「c + x1..xk」做一次前向
    并行拿到各位置真实分布 p1, p2, …, pk（及可选 bonus 位）
    口述：算力账本发生在「verify 一次换 k 步候选」

步③ Accept / Reject（逐位接受）
    从 i=1 起：
      α_i = min(1, p_i(x_i) / q_i(x_i))     # 以 α_i 概率接受 x_i
      若拒绝：
        · 丢弃 x_i 及其后全部草稿
        · 从修正分布 r = norm(max(0, p_i − q_i)) 采样 1 个 token
        · 本轮 verify 结束（后面草稿作废）
    金句：拒绝不是「随便换个词」，而是按 standard 拒绝采样修正，保证分布对

步④ Bonus（可选）
    若 1..k 全接受，还可从 target 在下一位置的分布再采 1 个 token
    bonus 从 target 采，不破无损

    性质：数学无损。 拒绝采样保证最终输出序列的分布与 target 单独解码完全一致（lossless）。这是投机解码区别于「用小模型近似」的根本点。[机制]

    1.3 无损定义（必背一句 + 四条边界）¶

    项 口径
    是什么 最终 token 序列的分布 ≡ 只用 target 自回归 的分布（lossless）。[机制]
    靠什么 standard 拒绝采样：min(1, p/q) 接受 + 拒后 norm(max(0, p−q)) 重采样。[机制]
    不是什么 不是「一定更快 / 一定更高吞吐」；慢了也仍叫无损。[机制]
    易混一 Medusa 等 typical acceptance 路径常**非严格无损**——开口先问验证器是不是拒绝采样。[文档]
    易混二 MindIE MTP 常见**贪心逐位比对**；≠ vLLM 默认概率拒绝采样，别混称同一套。[代码]

    1.4 期望加速比直觉（一页公式）¶

    衡量「划不划算」永远只看这一个比值：

    τ  = 本轮期望接受长度（含 bonus 按实现计）
T  = T_draft + T_verify        # 一轮 draft→verify 墙钟

单 token 延迟 ≈ T / τ
不开投机的退化情形：τ=1、无 draft 开销 → 单 token 延迟 = T_target
要对齐/超过自回归：需  T/τ  <  T_target
加速比直觉 ≈  T_target · τ / (T_draft + T_verify)  > 1
    • 分子（T_draft + T_verify）：一轮 draft→verify 总耗时。T_draft 串行自回归类随 k 线性涨（DFlash/DSpark 并行主干近似与 k 无关）；T_verify ≈ target 正常跑 1 步（只要 batch 未到 compute-bound）。[机制]
    • 分母 τ：每轮实际接受 token 数，由逐位接受率 α（min(1,p/q) 的期望）决定；草稿越准、接受率越高，τ 越大。[机制]

    失效三叉（每条都翻译成「分子变大」或「分母变小」）：

    τ↓        猜不准 / 分布不匹配 / 高温采样 → 草稿大量被拒，τ→1
T_verify↑ 大 batch / GPU 已饱和 → compute-bound，verify 不再免费
T_draft↑  串行 draft 单步延迟 × k 逼近 T_target → 收益被吃光

    1.5 大 batch 失效（一句钉死）¶

    小 batch 时 verify 吃的是 memory-bound 的闲置算力（白捡）；大 batch / GPU 已饱和后 decode 变 compute-bound，并行验 k 个草稿是从别的请求嘴里抢算力 → 单请求或略快，系统总吞吐常下降。[文档]

    补刀：生产系统常在高负载时自动缩短甚至关闭投机长度（DSpark 的置信度调度即为此设计）。无压测曲线只谈定性，不报假百分点。[文档]

    1.6 其余失效场景（补全）¶

    除接受率低、大 batch 两条外，还有：draft 本身太贵（串行 draft 单步 ×k 逼近 target 一步）；输出短（prefill 占主导，decode 收益摊不够固定成本）；显存税（draft 权重 + 草稿 KV + 验证临时张量挤压 KV 池 → 最大 batch 变小 → 间接掉吞吐）。[文档]

    一句话总结：「接受率不够高、或 GPU 本来就不闲，投机解码就会失效；前者浪费了草稿，后者抢了别人的算力。」[文档]

    二、方法演进线：每代一句「解决了什么」¶

    总纲金句（开口 10s）：演进是两条腿——把 draft 更准（多头 → 特征自回归 → 动态树 → 联合训练），把 draft/verify 更省（树并行验 → 一次出整块 → 按负载裁 k）。每一步都专修上一代最痛的那个瓶颈，同时留下新短板给下一代。[文档]

    2.1 倒背表：核心改进 + 遗留短板¶

    代 解决了上一代的什么（一句） 遗留短板（半句）
    Vanilla SD（2023，Leviathan/Chen） 首提 draft-verify + 拒绝采样框架，证明「小模型猜、大模型验」可**数学无损** 独立小模型与 target 分布对不齐，接受率天花板低；还要单独部署维护
    Medusa（2024） 去掉独立 draft，target 顶层加多头，零额外模型部署 各头独立、无块内序列依赖；typical acceptance 常**非严格无损**
    EAGLE-1（2024） 把自回归从 token 层挪到更平滑的**特征层** + 真实 token 反馈，复用 target 的 emb/LM head draft 仍逐 token 串行，T_draft ∝ k
    EAGLE-2（EMNLP 2024） 发现置信度≈接受率 → 动态草稿树，免重训抬 τ 依赖置信度校准；本质仍串行
    EAGLE-3（2025） 去掉 l_fea 特征预测枷锁 + 直接预测 token + 多层特征融合 + 训练时模拟多步自回归，打破 scaling 训练侧复杂度/成本上升；draft 本质仍严格自回归，单步延迟没降
    MTP（DeepSeek-V3，2024/2025） **预训练阶段**就联合训练草稿头，推理白捡、对齐天然好、零额外蒸馏 依赖模型自带 MTP 权重，第三方无权重则套不上；层间仍串行
    DFlash（2026-02，UCSD/Z Lab，ICML 2026） 用 block diffusion 一次并行去噪出整块，T_draft 与块长近乎解耦 纯并行牺牲块内依赖 → suffix decay（块越长后缀越烂）；实现复杂
    DSpark（2026-06-27，DeepSeek） 半自回归（并行主干 + 轻量 Markov 头补连贯）+ 置信度调度（按 GPU 负载裁 verify 长度），同补「块内不连贯」「高并发验证浪费算力」 工程复杂度全场最高；目前主要在 DeepSeek-V4 自研体系验证，第三方通用性待检验

    2.2 箭头串读（30s，可直接背）¶

    Vanilla 证明可行 → Medusa 去独立模型 → EAGLE-1 挪到特征层 → EAGLE-2 加动态树 → EAGLE-3 破 scaling 瓶颈 → MTP 联合训练零对齐成本 → DFlash 并行去噪解耦延迟与块长 → DSpark 半自回归 + 按负载调度验证长度。每一步都是「上一步优势场景不变，劣势场景被下一步专门修补」——这正是「为什么还要发明新方法」的标准回答框架。[文档]

    2.3 EAGLE 三代细讲（Q15 常考「EAGLE 为什么好」）¶

    • EAGLE-1 特征层洞察：① feature 序列比 token 序列更平滑、规律性强，轻量单层网络就能外推；② 把**上一步实际采样出的 token**（经 target embedding）与 feature 拼接输入草稿网络，消除随机采样带来的歧义。草稿头仅约一层 transformer decoder，复用 target 的 embedding/LM head——这就是「拿大模型的层来用」的准确形态。同时用**树形草稿 + tree attention** 一次并行验整棵树。[文档]
    • EAGLE-2 动态树：静态树上下文无关，而「好猜」程度强依赖上下文（10+2= 后基本确定，开放问题则发散）。发现草稿置信度与真实接受率高度相关（well-calibrated），用置信度动态决定树往哪长——难位置少扩、易位置多扩，免重训抬接受长度。[文档]
    • EAGLE-3 破 scaling：特征预测损失 l_fea 成了 scaling 枷锁（被迫拟合 top-layer feature，加数据很快饱和）。改为直接预测 token（解放表达力，吃下更大数据）+ 多层特征融合 + training-time test（特殊 attention mask 模拟推理时多步自回归，消除训练-推理不一致）。效果：最高 6.5× 加速、比 EAGLE-2 约 +1.4×，SGLang batch 64 下吞吐 +38%。[文档·公开论文][文档]

    2.4 DFlash 一句（Q14）¶

    DFlash 把「草稿怎么来」从串行自回归换成 block diffusion 并行去噪，一次前向出一整块草稿；靠把 target 的多层特征注入 draft 的每层 KV 保住接受率。 verify 侧不变，仍是标准拒绝采样，无损。公开：>6× 无损加速、比 EAGLE-3 最多快 2.5×（[文档·公开论文/LMSYS 博客]）。纯并行代价是 suffix decay。[文档]

    2.5 提高接受率的五个手段（Q37）¶

    1. 让 draft 看到 target 内部表征：EAGLE 复用 emb/LM head 并以 top-layer feature（EAGLE-3 多层融合；DFlash 注入每层 KV）为输入；
    2. 数据对齐：用 target 生成/蒸馏数据训练 draft，对齐条件分布；
    3. 联合训练：MTP 把草稿头放进预训练，一致性天然最好；
    4. 结构性手段：树形草稿 + 动态扩展（EAGLE-2），同算力下提高「至少一条路径被接受」的概率；
    5. 推理期自适应：按置信度提前止损、按负载调整草稿/验证长度（DSpark）。[文档]

    三、草稿模型的权重从哪里来¶

    3.1 先纠正一个常见误区¶

    “草稿模型”是 proposer 的统称，并不一定对应一份完整的小语言模型权重。当前常见来源可归为五类：

    1. 现成小模型 checkpoint：直接加载同家族的小参数模型；
    2. 目标模型蒸馏出的专用 drafter：EAGLE、ReDrafter、PARD、MLP speculator、DFlash、DSpark 等，通常是一份与 target 一一绑定的 sidecar checkpoint；
    3. 附加预测头：Medusa、Hydra 等，在 target 顶部新挂并训练若干 head；
    4. 主模型预训练时内置的权重：MTP/NextN，与 target 一起训练并随 checkpoint 发布；
    5. 无学习权重的 proposer：n-gram、prompt lookup、Lookahead、Memory/Suffix decoding 等，从已有 token 或运行时状态直接找候选。

    因此，上线前不能只问“框架支不支持 speculative decoding”，还要问：这个 target 是否有匹配的 draft 权重、权重是否包含在原 checkpoint 中、tokenizer/vocab 是否兼容、draft 是否针对当前 target 版本训练。[机制/文档]

    3.2 各类方法的权重来源总表¶

    方法 草稿权重从哪里来 target 是否改动 部署时额外加载什么 可移植性
    Vanilla SD：独立小模型 直接使用已预训练/指令微调的小模型 checkpoint，常选 target 同家族的小尺寸版本；也可用 target logits/生成数据再蒸馏 否 一份完整 draft 模型权重及其 KV Cache 中等；通常要求词表/tokenizer 可对齐，且越接近 target 分布接受率越高
    Medusa-1 冻结 target，在其顶部新增并训练多个 Medusa heads；训练数据可来自原微调数据，也可由 target self-distillation 生成 主干冻结 仅额外 heads/adapter 权重；target 主干仍单独加载 低到中；heads 与具体 target hidden size、词表和版本绑定
    Medusa-2 Medusa heads 与 target 主干联合微调 是 联合训练后的整套 target + heads checkpoint 低；必须使用配套联合 checkpoint，不能把 heads 随意挂到原 target
    EAGLE-1/2/3 冻结 target，用 target 的 hidden states、token/概率分布监督一个轻量、target-specific draft 网络；EAGLE-3 使用多层 target 特征和 TTT 训练 否 target + 一份 EAGLE sidecar；部分 embedding/LM head 可从 target 初始化或共享 低；draft 与 target 架构、层选择、hidden size、词表及具体版本绑定
    Hydra / Hydra++ 冻结 target，在其 hidden state 上训练一组“前一草稿 token 会条件化后一 head”的顺序依赖 draft heads；新增 heads 由目标数据/蒸馏信号学得 否 target + Hydra heads checkpoint 低；heads 与 target 的 hidden size、词表和具体版本绑定
    ReDrafter 训练一个以 target hidden state 为条件的轻量 RNN drafter，并通过 target 知识蒸馏学习多步候选；不是从 target 直接裁层 否 target + target-specific RNN drafter checkpoint 低；条件特征接口、词表及训练 target revision 必须匹配
    PARD（Parallel Draft Model） 另行训练/发布可并行预测多个草稿位置的小型 draft checkpoint；当前 vLLM 示例为 target Qwen3 搭配 amd/PARD-Qwen3-0.6B 否 一份 PARD draft 权重；vLLM 以 draft_model + parallel_drafting 加载 低到中；必须核对官方声明的 target 配对、词表和并行草稿配置
    MLP speculator 为指定 target 训练轻量 MLP accelerator，使草稿预测同时条件化 target context vectors 与已采样 token；通常以 *-accelerator sidecar 发布 否 target + MLP speculator checkpoint 低；accelerator 与模型家族、hidden size、词表及 target 版本绑定
    原生 MTP / NextN 在 target 预训练阶段，将 MTP 层与主干联合训练；权重通常直接存入 target checkpoint 训练时一体化；推理时主干不变 若官方 checkpoint 完整，通常无需第二个完整模型；框架读取其中的 mtp.*/NextN 权重 很低；必须是自带 MTP 的模型或另行训练 MTP 层
    后训练 MTP speculator 从原生 MTP 层开始，只微调 MTP 层；embedding 和 LM head 通常冻结并与 verifier 共享，之后可 stitch 回 checkpoint 或作为 sidecar 主干冻结 微调后的 MTP 权重 低；仍绑定 target，但比从零训练 drafter 便宜
    DFlash 针对每个 target 另训轻量 block-diffusion drafter；训练时以 target hidden features 为条件，并用真实/采样续写 token 做块级交叉熵监督 否，target 通常冻结 target + target-specific DFlash checkpoint 低；不能把某个 target 的 DFlash 权重无条件换到另一模型/版本
    DSpark 另训 parallel backbone、Markov/RNN sequential head 与 confidence head；训练需要 target 输出分布监督，confidence head 与 drafter 端到端训练后再校准 否 target + DSpark sidecar checkpoint 低；结构、特征接口、词表和校准曲线均与 target/服务系统相关
    Self-speculative / LayerSkip 直接复用 target 自身部分层：跳层、提前退出或使用浅层子网络；纯版本没有新模型权重 否；若训练 early-exit head 则例外 通常无第二份模型，最多额外 exit head 较高，但效果依赖 target 是否适合跳层以及框架支持
    n-gram / Prompt Lookup / Suffix 没有神经网络权重；从当前 prompt、已生成文本或历史语料索引中匹配后续 token 否 哈希表、trie、suffix array/cache 等运行时索引 高；重复模式强时有效，开放式生成接受率有限
    Lookahead/Jacobi 没有专用草稿权重；反复调用 target 或复用 target logits，通过 Jacobi 迭代与 guess set 构造候选 否 解码状态与候选池 高；有额外 target/迭代计算，收益依实现和场景而定
    Memory Decoding 没有神经网络草稿权重；候选来自历史 token 序列/trie 记忆 否 内存索引及缓存状态 高；更适合代码、模板或重复前缀明显的流量

    Medusa 的两种训练方式由原论文明确区分：Medusa-1 冻结 backbone 只训 heads，Medusa-2 联合训练；无训练语料时还可使用 target self-distillation。Medusa 论文[文档]

    截至 2026-07，vLLM 官方方法页已列出 EAGLE、MTP、独立 Draft Model、PARD、MLP、n-gram 和 suffix 等路径；这是一份**框架版本相关的实现清单**，不等于投机解码研究算法的全集。Hydra/ReDrafter 等即使论文和权重存在，也仍要检查目标推理框架是否有对应 loader、kernel 和 verifier 集成。参考 vLLM Speculative Decoding、PARD 部署文档、MLP Draft Models 文档、Hydra 论文 与 ReDrafter 论文。

    3.3 独立小模型：权重不是从大模型“裁下来”的¶

    Vanilla speculative decoding 最直观的做法是选择同系列的小模型，例如“大模型 target + 同 tokenizer 的 1B/3B draft”。这份 draft 权重通常来自它自身的预训练和指令微调，不是运行时从 target 截取前几层自动得到。两者只是模型家族、训练数据和 tokenizer 相近，因此分布更容易对齐。[机制]

    若现成小模型接受率不够，可做 target-specific distillation：

    业务/通用文本
   → target 产生 token、logits 或生成轨迹
   → 用 CE / KL 等目标训练小 draft 拟合 target 分布
   → 得到单独的 draft checkpoint

    这里要区分“权重来源”和“监督来源”：**draft 的初始权重**可以来自现成小模型，**后续监督信号**来自 target；蒸馏后它仍是一份独立完整模型，要额外占权重显存和 draft KV Cache。

    标准实现还要求 draft 与 target 在 token 空间中可比较。最稳妥的是共用 tokenizer/vocab；若词表不同，需要显式 vocab mapping 或专门的 heterogeneous speculative decoding 支持，不能直接拿 q(x) 与 p(x) 做逐 token 验证。

    3.4 Medusa：只训练新增 heads，或连主干一起训练¶

    Medusa 在 target 最后一层 hidden state 上挂多个预测头，每个 head 负责预测更远的未来位置。heads 的权重是新增参数，不是简单复制 target 的 LM head：

    • Medusa-1：冻结 target，只优化新增 heads。可以使用原始 SFT 数据的真实 token 作为标签，也可以让 target 对无标签 prompt 生成数据做 self-distillation。产物通常是体积较小的 Medusa adapter/head checkpoint。
    • Medusa-2：同时更新 target 主干与 Medusa heads，以提高候选准确率。产物是联合微调后的完整模型，heads 不能脱离这份主干单独迁移。

    面试金句：Medusa-1 的权重是“冻结 target 后学出来的外挂多头”；Medusa-2 则是“target 和多头共同变成一份新 checkpoint”。[文档]

    3.5 EAGLE：target-specific 的知识蒸馏 sidecar¶

    EAGLE 运行时通常同时指定 base_model_path 和 ea_model_path，这说明 EAGLE 权重并不天然包含在 target 中，而是一份单独训练的轻量 drafter。EAGLE 官方仓库[文档]

    典型训练链路为：

    1. 冻结 target，将训练文本送入 target；
    2. 提取 target 的 hidden states；EAGLE-3 通常取浅层、中层和末层特征并融合；
    3. 用真实 next token、target logits/概率及多步 rollout 训练轻量 draft decoder；
    4. target 的 embedding 可用于初始化 draft embedding，LM head/embedding 也可按实现共享或映射；真正新增并保存的是 draft decoder、特征融合投影等参数；
    5. 最终生成与特定 target 配套的 EAGLE-*.safetensors sidecar。

    在线训练时 target 常驻并实时产生 hidden states；离线训练则先把 hidden states/target probabilities 落盘，再只加载 drafter 训练。两条路径改变的是训练系统成本，不改变“EAGLE 权重由冻结 target 蒸馏、与具体 target 绑定”的本质。Speculators EAGLE-3 训练文档[文档]

    3.6 MTP：权重随主模型预训练产生¶

    原生 MTP 与 EAGLE 最大的权重来源差异是：**MTP 不是模型发布后再给 target 蒸馏一个外挂，而是在 target 预训练时就加入额外未来-token loss，MTP 层和主干一起更新。**因此 DeepSeek-V3/V4、Qwen3-Next、Qwen3.5/Qwen3.6 等模型若官方 checkpoint 包含 MTP 参数，推理框架可直接把这些层当 drafter。[文档]

    需要注意三个工程细节：

    • 一些量化、裁剪或格式转换产物可能漏掉 mtp.* 权重；“模型架构支持 MTP”不等于手里的 checkpoint 一定包含 MTP tensors。
    • 若权重缺失，应从**同一版本、同一精度来源的官方完整 checkpoint**提取/转换，或另行训练匹配的 MTP speculator；不能从普通主干层凭空推导。
    • 当前也可在原生 MTP 权重上做领域后训练：冻结 target 的 embedding/LM head 和主干，仅微调 MTP 层，再 stitch 回模型或单独部署。vLLM MTP 文档[文档]

    3.7 DFlash 与 DSpark：都需要另训 target-specific drafter¶

    DFlash 的权重来自单独训练的 block-diffusion draft model。训练时 target 冻结并提供 hidden features，drafter 根据 mask block、上下文及这些内部特征并行恢复真实/采样续写 token，使用偏重块前部位置的交叉熵训练；target feature 被注入 drafter 的 KV/层中是运行时条件，不代表 DFlash 直接复制 target 权重。发布和部署时通常是一组“target 名称 + 对应 DFlash checkpoint”。DFlash 论文[文档]

    DSpark 也不是直接复用 V4 的 MTP 权重。它有自己的 parallel backbone、轻量 Markov/RNN head 和 confidence head；训练需要 target 的输出分布作为监督，confidence head 与 drafter 端到端训练，之后再做置信度校准。target 主干保持不变，最终把 DSpark sidecar 与 V4 target 协同部署。DSpark 论文[文档]

    面试区分句：MTP 权重随 target 预训练而来；EAGLE/DFlash/DSpark 是 target 发布后仍可单独训练的专用 sidecar；n-gram/Lookahead/Memory 则根本没有神经网络草稿权重。

    3.8 选型时检查哪些文件与契约¶

    拿到一个投机解码方案时，按以下顺序检查：

    1. 配对关系：draft model card 是否明确写了对应 target 的完整 repo、版本和 revision；
    2. 词表契约：tokenizer、vocab size、special tokens、chat template 是否一致；不同词表是否提供 t2d/d2t 映射；
    3. 结构契约：hidden size、目标特征层、MTP depth、draft layer 数与推理框架配置是否匹配；
    4. 权重完整性：checkpoint 是否实际包含 MTP/Medusa/EAGLE/DFlash 所需 tensors，量化转换是否保留这些权重；
    5. 训练口径：draft 用什么数据、哪个 target revision 的 logits/hidden states 监督，是否覆盖当前业务域；
    6. 运行成本：额外权重、draft KV、临时 verify tensor 会占多少显存，是否挤压 target 的 KV Cache 和最大 batch；
    7. 验收指标：不能只测接受率，还要测平均接受长度、draft/verify latency、TPOT、总吞吐、P99 与高并发退化。

    四、MTP 核心直觉 + MindIE 三插件对照¶

    4.1 MTP 核心直觉（30s 可扩）¶

    DeepSeek 在**预训练**里就挂若干顺序 MTP 模块（各带一层 transformer），第 i 个模块基于主干 hidden state 与前序 token 预测第 i+1 个未来 token。本意是加密训练信号、提升数据效率；推理时这些模块直接当 draft 头做投机——与主干联合训练、共享表征，所以分布对齐天然好，不必再做一轮独立蒸馏。DeepSeek-V3/R1 线上即用 MTP 自加速。[文档]

    问 答要点
    训练时干什么？ 顺序 MTP 模块预测未来 token；加密训练信号、提数据效率。[文档]
    推理时干什么？ 模块当 draft 头做投机；vLLM 侧对应 method: "mtp"。[文档]
    为何对齐好？ 与主干**联合训练、共享表征**，非事后另训小 draft。[文档]
    谁能用？ 需模型预训练预留 MTP 权重（如 DeepSeek-V3/R1）；第三方无权重则套不上、须另训或换 method。[文档]
    相对 EAGLE？ EAGLE 多是推理侧另训/挂草稿；MTP 训练期一体化，对齐成本最低，但可移植性被权重绑定。[文档]

    代价半句：依赖自带 MTP 权重；层间仍串行，T_draft 随层/块长走。[文档]

    4.2 MindIE 三插件对照表¶

    MindIE-LLM 把投机类技术分两条产品线，不跟论文一一抄名，而是三个共享同一 Plugin 基类的插件。骨架同构——候选来源不同，verify 都走贪心逐位比对。[文档]

    插件 候选从哪来 对照一句 互斥
    MTP 模型自带 MTP 层真实小前向 ≈ vLLM MTPSpeculator；DeepSeek 线上主路径之一 ↔ LA / Memory
    Lookahead Jacobi 迭代并行猜 + guess set（N/W/G 参数） 并行解码族，无额外训练权重 ↔ MTP；↔ Memory
    Memory Decoding 历史/trie 前缀树记忆检索候选 适合代码补全等重复模式，可动态长度 ↔ MTP；↔ LA
    • 共享钩子（记名即可）：model_inputs_update / sample_preprocess / plugin_verify / plugin_cache_update。[文档]
    • 对照金句：vLLM = drafter 多态 + verifier 统一（拒绝采样）；MindIE = Plugin 多态 + 贪心 verify 统一。[文档]
    • 无损边界一句：MindIE 贪心下 ≡ 自回归（min(1,p/q) 退化为「target 是否也选中同一 token」）；采样场景仅支持有限后处理（如重复惩罚）；**不要**把「MindIE 无损」说成「已实现 vLLM 式概率拒绝采样」。[代码]

    4.3 vLLM vs MindIE 架构小结¶

    维度 vLLM（GPU Speculator 体系） MindIE-LLM（Plugin 体系）
    抽象 BaseSpeculator/DraftModelSpeculator 类层级 + Triton kernel 复用 Plugin 统一接口，各插件独立实现 DecodingPolicy
    验证 Rejection Sampler：standard（概率无损）/synthetic/block 三种 逐位贪心比对（verify_greedy_one_batch），贪心下等价自回归
    图 CUDA Graph（DSpark 单图覆盖「并行主干 + 序列化 Markov 采样」） ATB 图模式（C++ 侧组图），Python 侧负责输入构造与校验
    支持 EAGLE/EAGLE3、MTP、DFlash、DSpark、ngram、medusa、suffix MTP、Lookahead、Memory Decoding
    硬件 NVIDIA/AMD GPU 昇腾 Atlas 800I A2/A3、300I Duo

    五、DSpark（相对 MTP 的卖点）¶

    5.1 一句定调 + 两词拆开¶

    DSpark（2026-06-27，DeepSeek；随 DeepSpec 开源，挂在 V4 上的推理优化模块，不是新模型）= Confidence-Scheduled（按 GPU 实时负载裁送验草稿长度 k）+ Semi-Autoregressive（并行主干一次出块 + 轻量低秩 Markov 头从左到右偏置校正，补块内连贯）。[文档·公开论文]

    词 一句
    Semi-Autoregressive 主干并行（快）+ Markov 薄层串行校对（连贯）；串行的是**偏置**，不是再跑一层 transformer。
    Confidence-Scheduled confidence 头估每个草稿 token 的「存活概率」→ 负载低多验（抬 τ），负载高少验（防 T_verify 抢算力）。

    5.2 相对 MTP / DFlash 的卖点（同时优化那个延迟公式）¶

    • 相对 MTP 卖点一句：MTP 草稿头仍串行小前向（T_draft 随层走）；DSpark 用**并行主干压 T_draft、用 **Markov 抬 τ、用**负载调度兜住高并发下 T_verify 暗涨**——专修「串行 draft 贵 + 纯并行 suffix decay + 大 batch 验证不免费」三个短板。[文档]
    • 相对 DFlash：DFlash 纯并行易 suffix decay；DSpark = 类并行主干 + Markov 补连贯 + 按负载裁 k，修 suffix decay 与高并发验证浪费。[文档]

    三管齐下对应同一个 (T_draft+T_verify)/τ 公式：Markov 头**推高分母 τ**；并行主干 + 几乎免费校对**压低分子 T_draft**；置信度调度**防分子 T_verify 在高并发下暗涨**。[机制]

    5.3 Markov 头为什么几乎不涨 T_draft（第三层常问）¶

    一阶马尔可夫假设 P(下一词|全部历史) ≈ P(下一词|上一词)，完整 V×V 转移矩阵约 10¹⁰ 参数存不起。DSpark 用**低秩分解**：markov_w1（V×r embedding，把上一词映射到 r 维隐向量）+ markov_w2（r×V 投影，展开成对当前词的偏置），V×V ≈ (V×r)·(r×V)，参数量压到 2×V×r。主干一次并行出 base_logits；Markov 头每步只做「embedding 查表 + 一次低秩小 GEMM」，不重新跑 transformer 层，从左到右滚动采样并可进同一 CUDA Graph。是**校对而非重新创作**。[机制/文档]

    边界：rank 以模型配置为准（通常几十到上百），未查权重勿报死数。

    六、与结构化输出硬互斥（一句 + 分层）¶

    一句：MindIE 上 MTP 与并行解码插件互斥；MTP × 结构化输出（response_format）在 Serving InferParam 入口硬互斥（ValidateMtpConstraints 报错 structured output (response_format) cannot be used with mtp），产品 fail-fast。勿说「可以一起开」。[代码]

    分层讲（防踩坑）：C++ InferParam 入口**有硬拦**；Python 插件层仍**未联调**（无 rollback、单位置 bitmask、propose/verify 不碰 grammar）。只 grep mtp_plugin.py 会误判成「完全无互斥」；只看入口又会以为「工程已打通」——两层都要讲。vLLM 有多位置 mask + rollback 路径，MindIE 尚未对齐到同能力，故入口 fail-fast。[代码]

    七、快问快答（合并去重）¶

    7.1 原理与账本¶

    # 问 答要点
    1 四步顺序？ draft 出 x,q → target 一次前向得 p → α 接受 / 拒后修正采样 → 可选 bonus。[机制]
    2 α 公式？ min(1, p(x)/q(x))。[机制]
    3 拒绝后采样哪个分布？ norm(max(0, p−q))，并丢弃该位之后草稿。[机制]
    4 无损指什么？ 输出分布 ≡ 纯 target 自回归；**非**吞吐/永远加速保证。[机制]
    5 单 token 延迟 / 加速比？ 延迟 (T_draft+T_verify)/τ；加速 ≈ T_target·τ/(T_draft+T_verify)>1。[机制]
    6 τ 是什么？ 每轮期望接受长度，接受率越高通常越大。[机制]
    7 为何 memory-bound 才划算？ 闲置算力「白捡」做并行 verify；GPU 饱和后不再免费。[机制]
    8 大 batch 为何失效？ compute-bound，verify 抢共享算力，系统吞吐常↓。[文档]
    9 高温采样对 τ 的影响？ p/q 更发散，min(1,p/q) 期望↓ → τ↓ 甚至负加速；生产常降 k 或关投机。[机制]
    10 除接受率/大 batch 外还有哪些失效？ draft 太贵、输出短、显存税（挤 KV 池间接掉吞吐）。[文档]

    7.2 演进线¶

    # 问 答要点
    1 Medusa 解决啥？ 去独立 draft；多头零额外部署；但头独立、typical 常非严格无损。[文档]
    2 EAGLE 相对 Medusa？ 特征层自回归 + 真实 token 反馈，序列依赖更强。[文档]
    3 EAGLE-2 一手？ 置信度动态长树，免重训抬 τ。[文档]
    4 EAGLE-3 破什么？ 去 l_fea、直接预测 token + 多层融合，破 scaling。[文档]
    5 MTP 为何对齐好？ 与主干联合训练、共享表征。[文档]
    6 DFlash 解决啥？ 并行去噪一次出块，解耦 T_draft 与 k；易 suffix decay。[文档]
    7 DFlash vs DSpark？ DFlash 纯并行；DSpark = 并行主干 + Markov 补连贯 + 按负载裁 verify。[文档]
    8 DSpark 相对 MTP 卖点？ 并行压 draft 延迟 + 负载裁 verify；MTP 仍串行头。[文档]
    9 DSpark 加速多少？ 公开 vs MTP-1：Flash +60–85% / Pro +57–78%；非我实测。[公开]
    10 你们线上开了 DSpark 吗？ 跟读论文与演进；未做本仓生产压测，不报自家 %。[诚实]

    7.3 MTP / MindIE / 结构化¶

    # 问 答要点
    1 MTP 训练本意？ 多 token 预测作预训练目标；推理白捡 draft。[文档]
    2 第三方无 MTP 权重？ 套不上；须另训或换 method。[文档]
    3 MindIE 三插件名？ MTP / Lookahead / Memory Decoding。[文档]
    4 候选源各一句？ 自带头小前向 / Jacobi 迭代 / trie 记忆检索。[文档]
    5 三插件 verify？ 统一贪心逐位比对 ≠ vLLM 默认概率拒绝采样。[代码]
    6 插件间互斥？ MTP ↔ 并行解码；LA ↔ Memory。[文档]
    7 MindIE 无损 = vLLM 无损？ 贪心下 ≡ 自回归；≠ 默认 min(1,p/q)，别混成一句。[代码]
    8 MTP × 结构化能同开？ 入口硬互斥（InferParam）；插件层未打通。禁「可同开」。[代码]
    9 Vanilla draft 权重哪来？ 通常是同家族现成小模型 checkpoint，也可再用 target logits/轨迹蒸馏；不是运行时从大模型裁层。[机制]
    10 EAGLE / DFlash / DSpark 权重哪来？ 冻结 target，以其 hidden states/输出分布监督另训 target-specific sidecar；三者结构不同，权重不能混用。[文档]
    11 Medusa 权重哪来？ Medusa-1 冻结 target 只训新增 heads；Medusa-2 连 target 主干一起联合微调。[文档]
    12 哪些方法没有草稿权重？ n-gram/prompt lookup/suffix、Lookahead/Jacobi、Memory decoding；候选来自文本匹配、迭代状态或历史索引。[机制/文档]

    八、第三层追问（深挖弹药）¶

    ① 高温采样（temperature 高）对 τ 的具体影响？ → p/q 更发散，min(1,p/q) 期望接受率↓ → τ↓，甚至负加速；生产常降 k 或在高负载关投机。[机制] → 边界：给不出具体 α(T) 曲线就说「定性：熵↑接受率↓」，不编百分点。

    ② DSpark Markov 头为何几乎不涨 T_draft？ → 主干一次并行出 base_logits；头是 V×r/r×V 低秩偏置，每步 ≈ embedding + 小 GEMM，不是再跑一层 transformer；可进同一 CUDA Graph。[文档/机制] → 边界：rank 以模型配置为准，未查权重勿报死数。

    ③ 为何还要发明 DSpark（面试官「上周新论文」）？ → DFlash 修了串行 draft 延迟，但留下块内不连贯（suffix decay）+ 高并发下 verify 暗涨；DSpark 用半自回归修 τ、用负载调度兜 T_verify——专修上一代劣势场景。[文档]

    ④ 「57–85% 是你测的吗？」 → 否。复述自专题已收录的 DeepSeek-V4 **公开线上**相对 MTP-1 口径；本仓无同配置复现曲线。要谈落地，先讲 (T_draft+T_verify)/τ 与接受率失效，再谈是否值得接 DSpark 类 method。[公开 + 诚实]

    ⑤ 「Lookahead 不用训练权重，为啥不默认全开？」 → 与 MTP 文档互斥；且 Jacobi/guess 接受率、延迟随场景差，不是免费午餐。Memory 更偏重复模式。选型看候选源是否匹配业务，而非「无权重=必开」。[文档]

    ⑥ 为何说「与结构化互斥」要分层讲？ → Serving 入口已硬拦（InferParam）；插件层仍未联调（无 rollback / 多位置 mask）——产品 fail-fast 盖住工程缺口。只 grep Python 会误判「无互斥」；打通路径 = 补齐 vLLM 范式，属设计非现状，勿说「已打通」。[代码]

    九、一页抄写版（闭卷默写对齐）¶

    账本:   延迟=(Td+Tv)/τ ；加速≈Tt·τ/(Td+Tv)>1
四步:   Draft(q,x1..xk) → Verify(target一次前向得p) →
        Accept α=min(1,p/q)；拒→norm(max(0,p−q))+截断后缀 → Bonus(可选)
无损:   分布≡target自回归；≠永远加速
大batch: compute-bound → Tv 抢算力 → 系统吞吐↓

演进(准): Medusa多头 → EAGLE特征AR → E2动态树 → E3去l_fea直接预测token → MTP联合训
演进(省): 树并行验 → DFlash一次出块(suffix decay) → DSpark半AR+负载裁k

权重来源:
        Vanilla=现成同族小模型/再蒸馏；Medusa=新增heads（M2联合改主干）
        Hydra=顺序依赖heads；ReDrafter/PARD/MLP=另训的target-specific轻量drafter
        EAGLE/DFlash/DSpark=冻结target监督另训的配套sidecar
        MTP=随target预训练写进checkpoint；ngram/LA/Memory=无神经网络草稿权重

MTP:    预训练顺序头 → 推理白捡draft；对齐好；要自带权重；层间仍串行
MindIE: MTP | Lookahead(Jacobi) | Memory(trie)；互斥；贪心verify ≠ vLLM拒绝采样
DSpark: Confidence-Scheduled + Semi-Autoregressive
        并行主干 + 低秩Markov校对 + 按GPU负载裁verify长度k
        公开(非我测) vs MTP-1: Flash +60–85% / Pro +57–78%
        τ: vs EAGLE-3 +26–31% / vs DFlash +16–18%
MTP×结构化: InferParam 入口硬拦；插件未联调 → 分层讲，勿说可同开
红线:   MindIE贪心≠vLLM min(1,p/q)；typical≠严格无损；公开数≠自家实测
