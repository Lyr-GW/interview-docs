    简历口径与口述话术（内容型汇总）¶

    本篇把「口径红线 / 自我介绍 / 三大项目 STAR / 简历数字诚实表 / 第三层追问弹药 / Server 重构边界 / 易混概念 / 反例踩坑 / 高压应急话术」整合为一篇可背正文。原则贯穿全篇：代码真相优先于简历圆数；缺证据只讲路径与机制，标「待补」，绝不现场编数（假 API、假 LOC、假加速比、「应该差不多」一律禁止）。数字冲突时，一律以第一章「口径红线」为准。

    一、口径红线速查（8 条 + 30 秒自检）¶

    任何与下表冲突的旧口述 / 旧简历表述，以本表为准。

    # 红线 正确口径 错误口径（禁止） 代码/文档锚点
    1 编译缓存 默认 100 条，FIFO（命中不调序） LRU / 128 structured_output_manager.py L95、L1054–1057
    2 tokenizer Coordinator 本地 model_path 同源加载 运行时从引擎拉取 Coordinator / 亲和链路
    3 bitmask apply torch NPU 算子组合 masked_fill_ 自研 fused kernel structured_output_bitmask.py
    4 xgrammar 字节级 PDA（可递归嵌套） 仓内写「FSM」不严谨 结构化专题
    5 MTP × 结构化 入口硬互斥 「可以一起开」 infer_param.cpp ValidateMtpConstraints
    6 KVA 注册 D 实例 不注册 Conductor；仅 P/U 上报 D 也上报索引 _KVA_ROLES={P,U}
    7 TTFT−70% 代表性测算（可讲公式与假设） 「客户原始日志实测 −70%」（除非拿出日志） TTFT 测算卡；诚实卡
    8 guidance MindIE 唯一落地后端 xgrammar；多后端是扩展抽象/预留 「我们线上有 guidance」 GuidedDecodingBackendType 仅 XGRAMMAR

    30 秒自检口令：

    1. 缓存？→ 100 / FIFO，命中不调序
    2. 70%？→ 测算，不是甩客户日志
    3. bitmask？→ torch 组合，可优化点才是 fused
    4. MTP + 结构化？→ 互斥
    5. D 实例？→ 不注册 Conductor
    6. xgrammar？→ 字节级 PDA，非随口 FSM
    7. tokenizer？→ **本地同源**加载
    8. guidance？→ 唯一 xgrammar，多后端是预留

    口袋口令：100 FIFO · 测算非日志 · 6ms tokenize · 200ms 超时预算 · 5000+ 待 cloc · 1万行待 diff · 1394≠累计。

    二、三分钟自我介绍（90s / 180s 两版 + 禁踩雷）¶

    用法：面试官说「先介绍一下」→ 默认 90s；「多讲讲 / 三分钟」→ 180s。

    0 · 开口前 10 秒过脑（红线钉子）¶

    钉子 正确嵌入 禁止出口
    编译缓存 「FIFO、默认 100」一带而过 LRU / 128
    TTFT−70% 「代表性测算」三字必出 客户 raw log / 实测 −70%
    bitmask / 算子 「torch 组合；主场在服务侧」 自研 fused / 生产写 kernel
    身份 「推理服务交付」 「我最擅长通信/HCCL」

    口诀：100 FIFO → 测算非日志 → 主场 SO+亲和 → 不吹 kernel。

    1 · 90 秒版（默认开场 · 可直接背）¶

    我目前在华为昇腾做 MindIE LLM 推理服务，主线是把特性从 0 推到可维护交付。三条线： 一、结构化输出：Schema → xgrammar 字节级 PDA → bitmask → 采样硬约束；编译缓存是 FIFO、默认 100。异步调度下 mask 错位我修过，验收钉在正确性。bitmask 落地是 torch NPU 算子组合，不是自研 fused kernel。 二、Motor KV 亲和：Coordinator 同源 tokenize → Conductor 精确最长前缀 → unified/load_gated；相对字符级近似树，我们是 查真索引 + 驱逐事件。主收益是少 Prefill；简历 TTFT−70% 是 代表性测算，不是甩客户 raw log。 三、Tool Call：协议适配 + 流式解析（Encode/Generate/Decode）；和结构化硬约束是不同保证层级，集成边界我讲得清。旁路有 Server Handler/Interface 重构——路径可指，LOC 待补不编。 今天希望深聊推理加速与调度；你可以从简历任意一条往下挖。

    时长自检：约 85–95s；三条各约 25s；结尾交棒一句。

    2 · 180 秒版（「多讲讲 / 三分钟」· 可直接背）¶

    我目前在华为昇腾做 MindIE LLM **推理服务**开发，入职约一年，主线是服务侧系统问题：延迟、吞吐、缓存命中、正确性。三条可核交付：

    结构化输出 0→1：从 Schema 编译到采样硬约束整条链路我扛过——xgrammar 做字节级 PDA，再落到 bitmask 约束 logits。编译缓存按内容哈希，策略是 FIFO、默认 100，命中不调序。异步调度场景里出现过 mask 与 token 游标错位导致非法 JSON，我按线程/游标/顺序三因修过，验收是正确性，不拿未测 TPOT 百分比圆场。bitmask 在 NPU 上是 torch 组合 masked_fill_；fused 是可优化点，我不上场吹自研 kernel。线上落地后端是 xgrammar；guidance 多后端是预留抽象，不是「线上有 guidance」。MTP 与结构化在入口 硬互斥，不能一起开。个人累计交付量级简历写 5000+，钉在这条线，不和 Tool/Server 行数混算。

    Motor KV 亲和：问题是多实例下「猜缓存 vs 查缓存」。我们做 token 级精确亲和——先同源 tokenize（含 tools 透传、fail-closed），再查 Mooncake Conductor 全局索引，打分用 unified/load_gated，并演进过防 herding。相对 SGLang/vLLM Router 本地字符树，我们错向偏向安全假阴。PD 下 D 实例不注册 Conductor，只 P/U 上报。TTFT−70% 是按命中少 Prefill 的 代表性测算，公式与假设可讲；不是客户原始日志实测。Mooncake 我用边界讲：Conductor 查、Store 放、TE 传；通信/HCCL 细节 非主战场，分层正交即可。

    Tool Call + Server：Tool Call 侧重协议适配与流式状态机解析，和结构化「生成期硬约束」互补；MindIE 上 structural tag 一类深度集成边界如实讲。Server 侧把 Handler（引擎/PD）与 Interface（协议/流式）拆开，合并的是复制粘贴链路；「约 1 万行」以 PR diff 为准，今晚不编 before/after。

    相对最熟是 KV 亲和 + 结构化 0→1；算子/投机我用同套归因语言协作，系统跟踪还在补。希望今天往调度、约束解码或 PD 任一方向深挖。

    时长自检：约 2:40–3:00；SO 约 70s / 亲和约 55s / Tool + 收尾约 35s。

    3 · 结构骨架（忘词时按格填）¶

    身份：昇腾 · MindIE · 推理服务 · 0→1 交付
  ├─ SO：PDA→bitmask→采样 · FIFO/100 · async 错位 · torch 非 fused
  ├─ 亲和：tokenize→Conductor→门控 · 查≠猜 · 测算70% · D不注册
  └─ Tool：协议+流式解析 · （可选）Server 边界 · LOC待补
交棒：可从简历任一条挖

    90s 砍法：每条只留「做了什么 + 一个红线钉子」；砍掉 MTP/guidance/herding/Server 展开。 180s 加法：每条补「失败模式或边界一句」+ 明确「最熟两块」。

    三、三大项目 STAR（结构化 / KV 亲和 / ToolCall）¶

    对方点名一条 → 只背对应段，勿三段连背（那是自我介绍的活）。Result 数字只走第四章诚实表。忘词骨架：S 场景痛点 → T 交付目标 → A 关键动作 3 点 → R 诚实结果 + 交棒。

    1 · 结构化输出 STAR（60–75s）¶

    段 可背句
    S MindIE 要交付「JSON Schema → 采样硬约束」整条链路；异步调度下曾出现 mask 与 token 游标错位，非法 JSON 直出。
    T 0→1 落地：Schema 编译、字节级 PDA、bitmask 约束 logits；正确性验收优先，不拿未测 TPOT% 圆场。
    A ① xgrammar 字节级 PDA → bitmask → 采样；② 编译缓存 FIFO、默认 100，命中不调序；③ async 错位按线程/游标/顺序三因修；NPU bitmask=torch 组合，非自研 fused。
    R 个人累计交付量级简历 5000+ 行（钉 SO 线，待补 author cloc，≠ 现状 wc≈1394）；线上后端仅 xgrammar（guidance=预留）；MTP×SO 入口硬互斥。热路径 bitmask 占 TPOT <1%–3% 仅经验量级，A/B 待补。

    口述粘贴版（约 70s）：

    S：要在 MindIE 上把 Schema 约束做到采样硬保证；异步路径还出过 mask 错位。 T：我扛 0→1：编译 → PDA → bitmask → 采样，验收钉正确性。 A：落地 xgrammar；缓存 FIFO/100；修三因错位；bitmask 用 torch masked_fill_，不上场吹 fused kernel。 R：这条线累计 5000+（待 cloc）；不是**客户测出 TPOT−x%。guidance 没接；MTP 和结构化**不能一起开。

    禁出口：LRU/128；「线上有 guidance」；「我写了 Ascend fused」；把 Tool/Server 行数混进 5000+。

    2 · Motor KV 亲和 STAR（70–90s）¶

    段 可背句
    S 多实例下「猜缓存 vs 查缓存」：字符级近似树易假阳；高前缀重复流量 Prefill 重算贵。
    T Coordinator 同源 tokenize → Conductor 精确最长前缀 → unified/load_gated 选实例；假命中可感知驱逐。
    A ① tokenize 前置（tools 透传、fail-closed）；② 查 Mooncake Conductor 真索引 + ZMQ Removed；③ 门控防 herding；PD 下 D 不注册。
    R 简历 TTFT −70% / E2E −50% = 代表性测算（非客户 raw log）：例 1187→351 ms（\(h\) 0.10→0.78）；tokenize 4K≈6 ms；Conductor timeout 0.2 s 是**配置预算**，P50/P99 待补。重复率不够则收益缩水——最脆是 \(h\)。

    口述粘贴版（约 80s）：

    S：多副本时本地字符树是猜；我们要在高重复场景少做 Prefill。 T：做成 token 级精确亲和：查索引、可回退、可防扎堆。 A：同源 tokenize → Conductor 最长前缀 → unified/load_gated；驱逐看 Removed；D 不注册。 R：−70%/−50% 是 Qwen3-32B、~8K、共享前缀~6.5K 的测算（1187→351 类），不是**甩客户日志。超时只报 **200ms 预算，不编 P99。

    禁出口：「客户实测 −70%」；把 −70% 说成 FA/算子魔法；编 Conductor P99；说 D 也注册。

    3 · Tool Call STAR（+ Server 半句旁路 · 55–70s）¶

    段 可背句
    S Agent 场景要多模型族协议适配 + 流式解析；和 SO「生成期硬约束」不是同一保证层级。
    T 交付 Encode/Generate/Decode 状态机与协议适配；边界如实：structural tag 深度集成不夸满。
    A ① 协议适配器按模型族分支；② 流式 4-Case / Hard Cut-off；③ 与 xgrammar 硬约束正交——parser 是事后软保证。旁路：Server Handler/Interface 拆分，合并复制粘贴链路。
    R Tool/Reasoning 不混进 SO 的 5000+；Server「约 1 万 行」= 经验量级/待补 before-after，**禁止**用现状 Handler≈2320 / Interface≈7239 冒充「删了这么多」。

    口述粘贴版（约 60s）：

    S：Tool Call 要在流式里稳解析，且别和结构化硬约束混谈。 T：协议 + 状态机解析可交付；集成边界讲清楚。 A：Encode/Generate/Decode；约束⊥parser；Server 侧 Handler/Interface 分层只一句。 R：能力是 Agent 可接；行数 不吹精确 LOC——1 万行待 diff，5000+ 钉在 SO。

    禁出口：把 Tool 说成「也是 5000+」；「删了正好 1 万行」；夸成已接齐所有 structural tag。

    5.2 TTFT−70% / E2E−50% 诚实卡¶

    标准口述：

    Motor KV 亲和在**高前缀重复**场景下，收益主因是少做 Prefill 重算，不是算子 magically 加速。简历「TTFT 降 70%、E2E 降 50%」对外统一口径是：Qwen3-32B、约 8K 输入、共享前缀约 6.5K 的代表性测算，不是「某客户原始 A/B 日志」——除非当场能出示拓扑 + 请求构造 + 基线毫秒。机制可证：Coordinator 同源 tokenize → Conductor /query 最长前缀 → unified/load_gated 选实例 → 引擎侧 cached_tokens / Scheduler 权威日志。正确复测：两组都开 Prefix Cache，只切 load_balance ↔ kv_cache_affinity，报 P50/P95/P99、cached ratio、回退率、负载倾斜。

    代表性测算公式（可白板）：

    TTFT = T_tokenize + T_conductor + T_queue + T_prefill + T_delivery
TTFT ≈ c₀ + T_prefill_full × (1 − h)

假设：c₀=80ms；T_prefill_full≈1230ms（单 P≈6.5k tok/s）
  Baseline h≈0.10 → 80 + 1230×0.90 = 1187ms
  Affinity  h≈0.78 → 80 + 1230×0.22 = 351ms
  降幅 (1187−351)/1187 ≈ 70.5%
E2E ≈ TTFT + (OSL−1)×TPOT
  +525ms decode（TPOT≈35ms×15）→ 1712ms → 876ms，约 −48.9%

    脆度：\(h\) 命中率抬升**最脆**（流量重复率不够就不成立）；\(T_{prefill}\)/吞吐中脆；\(c_0\) 相对稳；两组都开 PC 是方法论红线。重复率 80%+ 才可能逼近 −70%；20% 收益明显缩水；最坏 ≈ 纯 LB + 一次 tokenize，不灾难。

    找不到客户日志怎么圆（推荐顺序）：① 先降调「代表性测算，不是 raw log」；② 立刻给公式假设（模型/ISL/共享前缀/\(h\)）；③ 转机制证据（cached_tokens、Scheduler 重选日志、Conductor query）；④ 补方法论 A/B；⑤ 若追「为什么简历写客户」，承认表述偏强，复盘后统一改口测算。

    极限诚实话术：

    「仓里没有可复核的客户 A/B 原始包。我能辩护的是机制与测算假设；百分比必须以毫秒、分位数、样本量重新实测后才能写进对外材料。」

    5.4 bitmask NPU 路径诚实卡¶

    标准口述：

    采样前 xgrammar fill_next_token_bitmask 得到 int32 压缩位图 [batch, vocab//32] → GuidedDecodingLogitsHandler → apply_token_bitmask_inplace_npu：repeat_interleave 按 bit 展开 → masked_fill_(-inf) 屏蔽非法 logits。这是 PyTorch / torch_npu 算子组合，**不是**自研 Ascend C 自定义 kernel，也不是直接调 xgrammar 官方 CUDA kernel。工程取舍：element-wise、访存主导，框架算子已接近带宽墙；先正确性落地，性能不够再考虑融进采样 kernel。

    证据边界：

    声明 路径 旗标
    NPU apply 实现 structured_output_bitmask.py · apply_token_bitmask_inplace_npu（约 L46–63） 代码
    Handler 挂载 pta_handlers.py · GuidedDecodingLogitsHandler 代码
    apply 占 TPOT <1%~3% Profiling 手册估计 经验量级；严格 A/B 待补

    可优化点（主动加分）：① 与采样融合（masked_fill 与 argmax/softmax 访存重叠）；② H2D 压缩位图 + pin memory + 独立拷贝流与前向 overlap；③ 多位置 bitmask（打通 MTP 后 buffer 从 batch×1 扩到 batch×(1+k)）；④ 整图/aclgraph 边界讲清。

    极限诚实话术：

    「别问我『自研了哪个 NPU bitmask kernel』——没有。就是 torch 侧展开位图再 masked_fill_。我贡献在约束链路正确性与异步时序，不在再造一个 element-wise kernel。」

    5.5 async mask 错位：现象 → 修法 → 对 TPOT 的影响¶

    现象：异步调度 + 结构化输出出现非法 JSON、GrammarMatcher reject；PD 场景 D 节点首 token 多一个 {。

    根因（三个错位）：

    1. 线程错位：mask 曾在主线程 preprocess 生成；async 下 batch N 尚未 sample/accept，主线程已按过期 FSM 为 N+1 填 mask。
    2. 游标错位：C++ AddGeneratedToken 无条件入 buffer；Python 若只用「接受数」切片，一次 reject 后永久 off-by-one → 用 num_tried_tokens（含 rejected）对齐。
    3. 顺序错位：decode 若先 init grammar 再 sync，会跳过 replay；初始态 mask 只允 {。

    修法：① bitmask 生成搬到 forward_loop（forward 前），sample 后立即 accept，generate_token_async 清 last_sampling_metadata；② 双游标 num_tried_tokens/num_processed_tokens；③ decode 先 sync_states_for_decode 再 init/填 bitmask。

    对 TPOT 的影响（诚实分层）：正确性修复本身是把职责搬到 forward 线程串行 mask→forward→sample→accept 消除竞态，不是**为刷 TPOT；mask 生成 CPU 侧理想可与 NPU 前向 overlap；apply 访存主导占单步很小比例；**无严格 A/B 报告（待补）；补测预期热缓存后 TPOT P50 增量 <1%~3%，冷编译打在 TTFT +100–200ms。

    极限诚实话术：

    「这个 bug 的验收标准是正确性归零，不是 TPOT−x%。性能上我只能给机制账 + 补测方案；没有就把估计说成客户实测。」

    5.6 MTP × 结构化硬互斥¶

    标准口述：

    Serving 入口硬互斥：InferParam::ValidateMtpConstraints——mtp 开启且请求带 structured output（response_format）直接报错 "structured output (response_format) cannot be used with mtp"（infer_param.cpp；UT 有断言）。产品理由：对外契约清晰，宁可 fail-fast，不让业务拿到「看起来在加速、JSON 已破防」的结果。工程理由：插件层未打通——MTP propose/verify 不碰 grammar；matcher 创建未设 max_rollback_tokens；bitmask 仅 batch×1 单位置；无 rollback API。对照 vLLM 需要 (1+k) 多位置 mask + 三段式（试探→立刻 rollback→verify 后正式 accept；rollback 必须在验证前，因真实路径可能 ABC→AX 不只截断，且假状态不能拖过 GPU 验证）。异步再叠一层时序，风险乘积过大，交付期选择互斥保正确性。

    分层别踩坑：C++ InferParam 有**硬互斥；Python MTP 插件与 grammar **零引用交集（未联调）；「完全无互斥」是**错**（那是只 grep 了 Python，入口已拦）。

    极限诚实话术：

    「互斥首先是工程做不到安全同开，产品上才包装成明确错误。不是『业务不需要 JSON+MTP』——是同开会错。要支持就按 vLLM 范式补 rollback 与多位置 mask，不是现在已交付。」

    六、Server / HTTP 重构边界¶

    6.1 60 秒电梯稿¶

    Server 把一次推理请求拆成两层：Handler（SingleLLMReqHandlerBase → Prefill / Decode / PnD）对接引擎、PD 通信、metrics；Interface（SingleReqInferInterfaceBase → OpenAI / TGI / Triton / vLLM / …）对接协议、detokenize、流式缓存。Prefill HTTP 与 Decode gRPC 语义差大，所以是**共享基类 + 角色特化**，不是硬揉成单实现。请求用 make_shared + shared_from_this 绑异步回调，析构清 map 防泄漏。简历「削减冗余约 1 万行」是多协议×角色**重复链路合并**的量级目标；分层路径我指得到，before/after LOC 与 PR 待补——上场不编数。

    6.3 虚函数 vs 热路径（必背）¶

    虚调用落在**请求级 / 协议层**（每请求几次到每 token 回调级），纳秒级相对毫秒级推理步长不可见。真正热路径是采样、Attention、NPU forward——不在这层 HTTP/Handler 抽象里。若极端担心 per-token 虚派发，可用 CRTP 静态多态；我们优先可读性与删重复，未把 Server 层当吞吐瓶颈。原则：热路径下沉算子/引擎；非热路径用清晰抽象换维护成本。

    红线：不要说「虚函数优化了吞吐百分之几」——无 profiling 附件则只谈定性数量级对比。

    5 precise approximate Approximate=路由器影子树/历史猜谁有；Precise=引擎事件养真值索引再 /query——猜可假阳灌热，查错向偏安全假阴 「和 SGLang cache_aware 一样吗？」不拆档 → 假阳故事讲不出
    6 跨实例亲和 APC APC=同实例内满块 hash 链自动跳过 prefill；跨实例亲和=路由送到「存货最长」的机——两层正交，落点后仍靠 APC 真省算 「开了 prefix cache 还要 Motor 干嘛？」答「一样」→ 多 replica 打散讲不清
    7 Conductor Store/TE Conductor 查谁有前缀+选路（不搬字节）；Store 管放哪/淘汰；TE 管怎么搬（RDMA/TCP…）。Motor 只对接 Conductor 决策 「Mooncake 底层怎么实现？」揉成「一个存 KV 的东西」或把 Connector 说成官方组件 → 穿
    8 约束解码 tool parser 约束限制采样「下一 token 合法」；parser 在 Decode 抽 name/arguments、管流式 delta——正交、都要跑；开 xgrammar ≠ 可省 ToolCallsProcessor 「约束了还要 parser？」答「不用」→ 穿

    一页速记：1 嵌套要栈→PDA · 2 命中不调序 · 3 tokenize 前置 · 4 串行就绪 vs 并行叠传 · 5 查索引 vs 猜影子树 · 6 选机 vs 本机跳算 · 7 查/放/搬 · 8 采样合法 ⊥ 字段抽取。

    上场开口句（可选）：

    「我先拆易混词：PDA≠FSM、FIFO≠LRU、token 查≠字符猜、APC≠跨机亲和、约束≠parser——再按你们追的那一层下钻。」

    八、反例与踩坑（错答 → 为什么砸 → 正答）¶

    嘴里若冒出左列同构句，立刻停，改念正答。

    ① 跳槽动机：涨薪 + 嫌加班¶

    • 错答：「行情好想涨点薪；华为加班、周末加班比较多。」
    • 为什么砸：拉力=0、推力拉满；后面会被钉死在钱/苦，难翻技术主线。
    • 正答：拉力>推力（推理栈深度 / 端到端特性交付空间）；华为成长一句肯定。

    ② Mooncake：「只会用、底层不懂」¶

    • 错答：「底层我可能不是特别了解。」或揉成「就是个存 KV 状态的东西」。
    • 为什么砸：简历写了 Conductor/亲和，底层零分层 → 「配置工」嫌疑。
    • 正答：三层——Conductor 查+选 / Store 放 / TE 搬；Motor 只决策。

    ③ vLLM 配置：「一下子想不到」¶

    • 错答：「一下子想不到。」（零输出）
    • 为什么砸：日常搭 Motor+vLLM 却说不出加速旋钮 → 判定「不会迁移到配置层」。
    • 正答：按目标选——prefix / chunked / CG·aclgraph / 投机 / KV dtype / 预算。

    ④ 投机：「EAGLE 忘了 / DSpark 没看」¶

    • 错答：「EAGLE 有点忘记了。」「新论文还没了解原理。」
    • 为什么砸：自称看过投机却断在演进线与拒绝采样白板 → 最大知识失分区。
    • 正答：讲演进线 + 拒绝采样四步。

    ⑤ 编译缓存：LRU / 128¶

    • 错答：「编译缓存是 LRU，大概 128。」/「命中会挪到最近。」
    • 为什么砸：与源码 FIFO/100、命中不调序冲突；最高危口径，翻仓即穿。
    • 正答：FIFO / 默认 100 / 命中不调序。

    ⑥ TTFT−70%：「客户日志实测」¶

    • 错答：「客户场景实测 TTFT 降了 70%。」+ 拒给日志仍死撑实测。
    • 为什么砸：要 raw log / 假设清单时无法出示 → 数字信用崩盘。
    • 正答：代表性测算；五段 + 假设表；1187→351 是假设代入。

    ⑦ 约束自动机：随口 FSM¶

    • 错答：「就是个 FSM 约束 next token。」
    • 为什么砸：Schema 嵌套要栈；FSM 会被追「正则够不够」。
    • 正答：字节级 PDA（xgrammar）。

    ⑧ D 实例也注册 Conductor¶

    • 错答：「P/D 都上报索引，D 也能被亲和选中。」
    • 为什么砸：与 _KVA_ROLES={P,U} 不符；污染「谁写了可复用前缀」语义。
    • 正答：D 不注册；亲和只选 P/U。

    ⑨「开了约束就不用 tool parser」¶

    • 错答：「上了 xgrammar / structural tag，Decode 直接合法，parser 可省。」
    • 为什么砸：约束≠字段组装与流式 4-Case；Hard Cut-off、finish_reason 仍在 Decode。
    • 正答：约束 ⊥ parser，都要跑。

    ⑩「APC / prefix cache = 跨实例亲和」¶

    • 错答：「开了 automatic prefix cache 就等于多机 KV 亲和。」
    • 为什么砸：APC 只救同实例；多 replica 乱打仍稀释；ZMQ 是通知不是搬块。
    • 正答：两层正交——APC 本机跳算 + Motor/Conductor 选机。

    自检：嘴里若出现「涨薪」「底层不太懂」「一下子想不到」「LRU」「客户实测七十」「就是个 FSM」「D 也上报」「约束了不用解析」「开了 prefix 就行」——整句作废，改念正答。

    九、高压追问应急话术（错例 + 正确模板）¶

    总框架（10s 过脑）：① 诚实边界 5–10s → ② 同构迁移 30–45s → ③ 反问目标 10s。三个安全词：测算 · 待补 · 未压测。禁止：假 API、假 LOC、假加速比、「应该差不多」。

    开口锚：

    「这个具体点我没在生产落地 / 没留证据，我不瞎编。同构问题上我能讲：问题形态、取舍轴、失败模式——然后按你们更关心延迟、吞吐还是正确性对齐一版。」

    场景库（12）¶

    1 ·「这个你不会？」 - ❌ 沉默/尴尬笑，或硬猜半句假细节。 - ✅ 「这个具体实现我没跟到代码级，我不编 API。同构上我熟悉的是【SO 约束链路 / Motor 精确亲和】——问题都是『正确性 vs 延迟 vs 可运维』。你们这题更偏哪一轴？我按那轴拆权衡。」

    2 ·「这个你没测过？」 - ❌ 「应该差不多能到 x%。」/ 随口报 P99。 - ✅ 「对，这项我**没有可复核的 A/B 包**，所以不上场报百分点。我能讲测法：只切一个变量、报分位。要数标 待补。」

    3 ·「简历数字对不上 / 和文档不一致」 - ❌ 「那份旧的作废。」/ 死撑某一版数字。 - ✅ 「以今天统一口径为准：FIFO/100；TTFT−70%=代表性测算。若简历更满，是表述过冲，不是另有隐藏实测。数字冲突选**可辩护的弱表述**。」

    4 ·「和代码不符被抓包」 - ❌ 「可能是旧分支。」含糊到底 / 否认看得见的符号。 - ✅ 「你指的这段我认——以仓库为准。口述偏了的是【X】；正确是【Y，路径一句】。短接：LRU→FIFO/100；guidance→线上仅 xgrammar。」

    5 ·「竞品 / 某论文你没写过」 - ❌ 「他们应该也是 radix + 打分吧。」编组件名。 - ✅ 「这个我**没部署、没复现**，不装读过仓库。同构问题我讲：猜缓存 vs 查索引 / draft-verify 接受率。Motor=precise lookup+驱逐；不编组件名。」

    6 ·「你简历写了但讲不清细节」 - ❌ 堆术语遮羞 / 「太底层了记不清」。 - ✅ 「我负责【调度/协议/入口互斥】边界，不是 kernel/transport 全文。按链路：输入→决策→失败模式→观测；没写过的层直接标边界。」

    7 ·「连续追问把你问穿了」 - ❌ 「那我也不知道还能说什么……」/ 抱怨题偏。 - ✅ 「这块跟踪不够系统，暴露了，我记下来了。工程交付有把握——结构化 0→1 或 Motor token 级亲和。您想先听哪块？」

    8 ·「客户 / 业务方数字呢？日志拿来」 - ❌ 「客户那边敏感不能给，但确实是 −70%。」 - ✅ 「没有可出示的客户 raw log，不把 −70% 说成客户实测。代表性测算：五段 + \(c_0+T(1-h)\)，1187→351 是假设代入。机制用 cached_tokens/选点日志。」

    9 ·「LOC / 删了多少行？精确数」 - ❌ 「大概删了一万行。」（无 cloc） - ✅ 「精确 before/after 待 cloc，不上场编。能指 Handler/Interface/协议边界与 UT/IT；5000+ 钉结构化线，不和 Tool/Server 混算。」

    10 ·「加速比多少？能到多少倍？」 - ❌ 「一般 2–3 倍吧。」（无曲线） - ✅ 「无压测曲线**不报加速比**。投机讲 draft-verify/接受率/无损定义；亲和讲少 Prefill 测算场景。要数：固定负载、只切一旋钮、报分位。」

    11 ·「你是不是背的稿？换个问法」 - ❌ 重复同一段电梯稿，不接新约束。 - ✅ 「卡片是为了口径不翻车，不是只会一段。你改约束我改权衡——P99 还是吞吐、混部还是 PD。按新目标重排：瓶颈假设 → 改哪层 → 怎么验收（低重复率就不硬撑 −70%）。」

    12 ·「和上一段矛盾了 / 你刚才不是这么说的」 - ❌ 「我没说过。」/ 两套说法硬拧。 - ✅ 「以**更弱、可核**的那句为准。收回过满表述，改口测算/待补/未压测/代码路径。机制真冲突就画边界（通知面 vs 数据面 / MTP 互斥 / D 不注册），对齐仓库。」
