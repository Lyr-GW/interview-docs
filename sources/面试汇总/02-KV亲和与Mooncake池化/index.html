    KV 亲和与 Mooncake 池化 · 内容型汇总¶

    本篇把 Motor KV 亲和调度、Mooncake 三层池化、假命中与驱逐感知、Herding 与负载门控、PD 分离/混部选型、竞品对标、语义路由与 HCCL 边界整合成一篇内容型知识汇总，按主题组织正文，供系统复习与面试展开。

    贯穿全篇的口径红线（先记住，答题时自然带出，不要报未核实数）： - Motor 是**决策层**：tokenize 前置 → 查 Mooncake Conductor /query → unified/load_gated 打分选实例；真正的搬/存由引擎侧 Connector 调 Store/TE。 - D 实例不注册 Conductor，只有 P 与混部 U 注册；D 只消费 KV、不写可复用前缀。 - TTFT −70% / E2E −50% 是 32B、8K 长输入、高复用下的代表性测算，不是客户 raw A/B 日志。 - block_size 128：Motor/Conductor 侧常配 128，索引与引擎必须一致，否则精确亲和静默全 miss 退化 LB。 - Conductor 查询超时 200ms（0.2s）是配置目标，不是实测 P99；P50<5ms / P99<20ms 是目标非实测。 - Connector 不是 Mooncake 官方组件，是 vLLM/SGLang 各自写的引擎适配层。

    一、Motor KV 亲和总览¶

    1.1 问题：多实例下前缀缓存碎片化¶

    单实例 vLLM 有 automatic prefix caching（APC）：KV 按 block 组织，每块链式哈希 block_hash = H(parent_hash, tokens_in_block, extra_keys)，新请求按 block hash 找最长命中前缀、跳过已算部分的 prefill。但 APC 只在**实例内**生效。N 个实例做 round-robin 时，同前缀请求落到"上次算过它的那台"的概率只有 1/N——集群前缀命中率随实例数线性稀释；而 prefill 恰是 TTFT 主项，长前缀反复重算直接把 TTFT 打爆。

    典型客户流量是多轮对话、RAG、Agent/Function Call：每个请求携带很长公共前缀（system prompt、tools schema、历史轮次），前缀重复率极高。KV 亲和调度（prefix-cache-aware routing）就是要在请求进来前判断"哪个实例已缓存了它的最长前缀"，在缓存收益与负载均衡间做权衡。

    1.2 Motor 三步：tokenize 前置 → 查 Conductor → 打分选机¶

    Motor（MindIE-PyMotor）是多实例调度层：拉起并管理底下多个 vLLM 引擎实例，在其上做负载均衡、KV 亲和、PD 分离路由与高可用。KV 亲和核心三步：

    1. Tokenize 前置：Coordinator 层用 HuggingFace AutoTokenizer + apply_chat_template(conversation, tools, add_generation_prompt=True)，加载与下层引擎**同一份模型权重目录**的 tokenizer，把请求转成与引擎**逐字节一致**的 token ids。tokenize 失败则 fail-closed 返回 []、回退 LoadBalance。
    2. 查询最长前缀：拿 token ids 通过 conductor_api_client 调 Mooncake Conductor 的 /query，按 KV block（token 级整块哈希）索引返回各实例各 DP rank 的 longest_matched。Motor 本地**没有** radix tree，前缀索引整体托管给 Conductor；prompt 短于一个 block 走 fast path 跳过查询。
    3. 亲和与负载权衡：unified（软融合加权和）或 load_gated（负载硬门控）两种模式，避免"所有同前缀请求灌进一个实例"造成热点。

    一次 tokenize 三处复用：① Conductor 前缀查询；② isl（输入长度）参与亲和打分；③ 用真实 token 数记 prefill 负载账（比字节长度启发式更准）。副产品还解锁入口长度预校验（超 max_model_len 直接拒绝）。

    1.3 token 级 vs 字符级：精度换开销的反向取舍¶

    业界"猜缓存"派（SGLang router / vLLM Router 的 cache_aware）用**本地近似 radix tree，节点 key 是 char（字符级）**，刻意用精度换 tokenize 开销、零同步、无引擎驱逐反馈。字符级的两个硬问题：

    1. 字符前缀相同 ≠ token 前缀相同——tokenizer 合并规则、chat template、system prompt、tools 注入都会让 token 序列分叉，估出的"命中长度"与引擎按 token block 哈希的真实命中对不上；
    2. 引擎 prefix cache 以 block（引擎侧常 16 token）为粒度，字符数无法对齐 block 边界，无法精确估算可复用 block 数。

    Motor 走 token 级：精确前缀亲和的硬条件是**调度层看到的 token ids ≡ 引擎 prefix cache/kv-events 里哈希的 token ids**，任一不一致 → block hash 全 miss → 精确亲和静默退化成负载均衡。分叉来源：chat template、tools 注入、reasoning/thinking 模板、Harmony/Mistral tool 序列化、多模态占位符展开。所以精确亲和路线都收敛到同一哲学：不在路由层"自己猜"怎么 tokenize，要用引擎同源的 render 逻辑。 代价是调度层毫秒级 CPU + tokenizer 一致性维护（部署约定：同 model_path、同默认 template）。

    tools 必须透传：函数调用请求的 tools 会被 chat template 注入 prompt，漏传会导致 token 序列分叉、命中长度虚高——这是修过的真实 bug。

    1.4 双模式：unified 与 load_gated¶

    • unified（默认）——单一融合分数，全局最小者胜：亲和收益与实时负载加权融合，命中够大可翻盘负载；空闲无命中的 endpoint 也可能赢，从而避免同前缀请求全灌一台热机。
    • load_gated——两阶段硬门控：先按负载筛出 Top-N 最闲 endpoint，再在其中比最长前缀命中。给出**硬负载上界**：亲和永远出不了"最闲集合"，适合负载敏感、宁可牺牲部分命中率也不出长尾的场景。

    粒度精确到 DP rank：vLLM DP 部署下每个 DP rank 有独立 KV 池，注册与打分都落到 (instance, endpoint=DP rank)，不是实例级——否则收益打到 1/DP。

    1.5 与 vLLM Render 的关系（同哲学、异实现）¶

    vLLM 把 chat template + tokenize + 校验产品化成 /v1/*/render（可 vllm launch render 无 GPU 独立部署），llm-d Precise 用它拿精确 token 再查 ZMQ 索引。Motor 在 Coordinator 用 TokenizerManager 同源 apply_chat_template(+tools) 做等价的事：哲学相同（调度必须看引擎真实 token），实现上 Motor 是进程内同源加载、他们是 HTTP 调官方 OnlineRenderer。Render 解决"token 从哪来要一致"；亲和系统还要额外解决"查谁、怎么和负载叠加、多进程如何不 herding"——这些是 Render 本身不提供、调度系统要补的差异化。

    演进叙事：字符级本地树（便宜不准）→ 调度层自建 tokenize（Motor TokenizerManager）→ 引擎同源 Render API（vLLM /v1/*/render）→ 精确亲和 = Render tokens + KV-Events 真实索引（Motor Conductor / llm-d Precise）。

    1.6 完整架构：控制面、通知面与 KV 数据面¶

    Motor KV 亲和由三条彼此正交的链路组成。把它们分开理解，才能避免把 Conductor、KV Events 与 Mooncake Store 混为同一个组件：

    1.6.1 启动与注册：先让索引知道“可选的缓存持有者”¶

    1. Motor 拉起或发现 P（Prefill）与 U（混部）实例；每个 DP rank 都是独立 endpoint，拥有独立 KV 池。
    2. Coordinator 对每个 (instance, DP rank) 调 Conductor /register，携带 instance_id、dp_rank、block_size、该 rank 的 KV-event endpoint 与 replay endpoint。D（Decode-only）不注册：它不产生可复用前缀，不能成为 KV 亲和的命中目标。
    3. 引擎开始运行后，满 KV block 写入即发 BlockStored，LRU/释放即发 BlockRemoved；Conductor 按 hash 链更新 PrefixCacheTable。订阅端若发现 seq 缺口，通过 replay endpoint 增量补齐；Conductor 重启则由 Motor 对账并重注册。

    这里的边界很重要：Conductor 只维护“逻辑前缀 → endpoint”的索引，不保存 KV tensor；Store/TE 才是实际 KV 的存储与传输层。

    1.6.2 一次请求的完整决策与执行路径¶

    1. 入口渲染：Worker 用与引擎同源的 tokenizer 和 chat template（含 tools）生成 token_ids，并做输入长度校验。结果缓存在请求对象中，供后续查询、打分和真实 token 负载记账复用。
    2. 索引查询：若 prompt 至少覆盖一个 KV block，Worker 调 Conductor /query；Conductor 返回每个 (instance, DP rank) 的 matched_tokens。短 prompt 没有整块可命中，直接跳过查询。
    3. 本地策略计算：Worker 从 SHM 读取负载快照。unified 计算每个 endpoint 的 prefill_cost 与本地分数；load_gated 先选最闲 Top-N，再在其中按命中长度比较。
    4. 全局仲裁：Worker 将候选送入 Scheduler。对 unified，发送全量 endpoint 的时不变 prefill_cost，Scheduler 代入权威新鲜 load 重算 scale × prefill_cost + weight × load，取全局最小；对 load_gated，仅在 worker 提出的固定 top-k 内仲裁，以保留硬门控语义。
    5. 请求落点与记账：最终 (instance, endpoint) 被返回 Worker，Worker 将请求转发至对应 vLLM endpoint；负载账本以真实 len(token_ids) 记录 Prefill demand。请求进入/完成时，生命周期事件更新 active token 与 KV 占用，供下一次决策使用。
    6. 缓存闭环：引擎命中的 block 直接复用，仅计算尾部 Prefill；新生成的满 block 再通过 KV events 进入 Conductor 索引。若运行在 PD 分离拓扑，P 端完成 Prefill 后的 KV 交接由 Connector/TE 完成，D 的选择属于后续 Decode 路由，不把 D 误当成前缀缓存命中目标。

    1.6.3 异常路径：亲和失效只能降级，不能阻塞或误导¶

    tokenize / tools 渲染失败       → 返回空 token_ids / 不做亲和 → LoadBalance
Conductor 超时、无响应、无索引  → 放弃亲和                 → LoadBalance
prompt 小于一个 KV block        → 没有可命中整块           → LoadBalance
Scheduler 候选不可用            → 选择最小工作负载 endpoint → 引擎继续处理
KV event 延迟或索引缺口         → replay / 重注册；短期少命中 → 多算 Prefill，而非错误复用

    这体现了设计原则：KV 亲和是性能优化而不是可用性依赖。失败时允许损失命中收益，但不能让错误 token、过期索引或慢 Conductor 把请求错误地固定到热点或阻塞主路径。

    二、五参数与打分公式（可默写）¶

    2.1 五旋钮速记¶

    参数（配置键） 物理含义 调大 调小 默认
    kv_affinity_mode 选 unified（软加权和）/ load_gated（字典序硬门控） — — "unified"
    kv_affinity_load_weight load 项权重；1.0 = 1 token 排队 ≡ 1 token 待算 prefill 更怕热点，空闲机更易赢 更追命中；0 = 纯亲和 1.0
    kv_affinity_overlap_credit 命中 1 token 折抵多少 prefill 工作量 命中更值钱、更易翻盘 命中折扣缩水，更像无缓存 1.0（命中即全额省）
    kv_affinity_prefill_load_scale prefill_cost（待算量）整项缩放 放大"还要算多少"权重 缩小待算项，负载项相对更响 1.0
    kv_affinity_load_gate_topn 仅 load_gated：最闲集合宽度 门控变宽、亲和可选面更大 门控变窄；1 ≈ 纯 LB 字面 0 → 运行时语义 2

    旁路常量（常一起默）：_AFFINITY_CANDIDATE_TOPK = 3（V2 候选宽）；负载分 = active_tokens + 0.3 × active_kv_cache（0.3 是启发式常数，非第一性）；Conductor 查询超时 0.2s → 回退 LB。

    2.2 unified 公式（默写）¶

    prefill_cost = max(0, isl − overlap_credit × matched)
score        = prefill_load_scale × prefill_cost + load_weight × load
选 endpoint:  argmin(score)   // 平局偏低 prefill_cost（更好亲和）
    • 量纲全是 token：第一项是"扣掉命中后还要真算的 prefill 工作量"，第二项是该 endpoint 实时负载分。
    • 翻盘：能——命中够大 → prefill_cost 小 → 可压过高 load。
    • 退化：load_weight=0 → 纯最长前缀。
    • 防 herding：unified 升到 V3 全量全局重排。

    2.3 load_gated 公式（默写）¶

    ① candidates = nsmallest(N = load_gate_topn, endpoints by load)
② 在 candidates 内: matched 降序；平局取更低 load
禁止: 亲和把请求拉出"最闲 Top-N"
    • 亲和 vs 负载：亲和**永远不能**压过门控。
    • 防 herding：钉 V2 固定 top-k，故意不带 prefill_cost 软重排。
    • 退化：topn=1 → 近似纯 LB。
    • 为何必须单开模式：字典序硬约束 ≠ 任何有限权重的加权和——想"绝对优先负载"需 load_weight→∞，那会压死亲和变纯 LB，仍不是"先负载再亲和"的字典序。所以不能靠调 load_weight 冒充 load_gated。

    2.4 翻盘临界（测算例，可算账）¶

    锚：isl=8192、matched=6000、三系数全 1.0（对练设定，非压测曲线）。

    epA（高命中已排队）: score = (8192 − 6000) + load_A = 2192 + load_A
epB（零命中空闲）:   score = 8192 + load_B
epB 胜 ⟺ load_A − load_B > 6000   (= overlap_credit × matched)

    调参直觉：↑credit → 空闲更难翻盘；↑load_weight → 空闲更容易赢；↑scale → 命中优势更响。一个在飞 8K prefill 加负载约 ×1.3（≈10.6K 负载）常已超过 6000 门槛。均为机制账。打分再精，假阳仍靠 ZMQ Removed 纠偏；utilization/speculative 未进分。

    三、Herding 三版演进与负载门控¶

    3.1 问题：多 worker 滞后视图下的羊群效应¶

    多个 Coordinator worker 进程并发调度时，每个 worker 本地负载视图有滞后。一波同前缀 burst 进来，所有 worker 都算出同一个"最优" endpoint → 全部灌过去 → 打爆（herding）。

    3.2 三版演进¶

    版 一句话 命运
    V1 overlay worker 本地虚加 in-flight 负载，只对本进程可见；跨 worker 仍齐刷同一 endpoint，TTL 两难、与真实账本双写 整体移除（不是调 TTL）
    V2 top-k worker 报亲和 best-first top-k=3 候选；Scheduler 慢路径用权威新鲜账本在候选内重选最低负载 落地；load_gated 钉在此版
    V3 全局重排 unified：每 endpoint 上报时不变 prefill_cost + 两个权重标量；Scheduler 用新鲜 load 全局重算 combined = scale×prefill_cost + weight×load 取全局 min（平局偏低 prefill_cost） unified 最终形态；load_gated 刻意不升级

    三版为何这样走： - V1 为何删：作用域错了——overlay 只对本 worker 可见，8 个 worker 各自"最优"仍是同一 endpoint，跨 worker herding 一点没解；TTL 太快等于没有、太慢会长期高估该机负载让命中好的实例接不到请求；请求真正落账后 overlay 未清会双算。局部视图打补丁修不好全局一致性，方向本身错。 - V2 为何不够：k=3 是拍的。若 top-3 恰好都被这波 burst 打热，而第 4 名（命中略差但完全空闲）才是全局最优——它不在候选集里，Scheduler 无从选起。k 调大又增加消息体积与仲裁成本。 - V3 分工精髓：利用 unified 分数可分解——prefill_cost（亲和折扣后待算量）不随时间变，load 才是易腐数据。worker 把每个 endpoint 的 prefill_cost 全量上报，Scheduler 用自己账本的新鲜 load 重算全部 endpoint、取全局 min。Scheduler 不需要 prompt、不需要 tokenizer、不需要再查 Conductor，一次 O(endpoints) 线性扫描即可，没有截断，"第 4 名才是最优"的场景自然覆盖。

    3.3 为何 load_gated 不升 V3¶

    load_gated 语义是"硬负载上界"——选择永远不出最闲 Top-N 集合。若让 Scheduler 拿软分数全局重排，等于把硬约束松绑成软权衡，模式的存在意义就没了。所以它 prefill_cost=None、钉 V2 固定 top-k。

    3.4 worker 提案 + Scheduler 权威仲裁¶

    分工：Conductor 回答"缓存在哪"，Worker 回答"综合打分谁最优"（亲和 + 本地负载视图），Scheduler 回答"最终去哪"（权威新鲜负载，防并发 herding）。负载分发走共享内存（SHM）+ ZMQ，worker 每次调度前从 SHM 补新鲜负载，心跳过期或实例版本变化才拉全量。LoadBalance 候选也走同一套"worker 提案 + scheduler 权威重选"仲裁基础设施。

    四、假命中与驱逐感知¶

    4.1 两类假命中¶

    • 类型 A 驱逐滞后（假阳性）：索引还说"W 有前缀 S"，引擎已 BlockRemoved。主因是未订 Removed / 丢事件 / 近似树。解药：Removed + Cleared + replay。
    • 类型 B 事件空窗：路由已决策、BlockStored 未进索引，多为假阴性；speculative TTL 是可控假阳。解药：speculative/predicted 短 TTL（Motor 目前**无** speculative）。

    4.2 假阳性为何比 round-robin 更糟¶

    本地近似树无驱逐感知时，会**确定性**把同前缀 burst 全灌到"它以为有、实已 LRU 掉"的实例——既满量重算，又制造热点排队；RR 至少按 1/N 摊开，还有偶然命中。

    场景锚（对练设定，非实测）：Qwen3-32B、ISL=8192、共享前缀≈6.5K、单 P ~6.5k tok/s、N=4。

    项 RR（N=4） 假阳打到已驱逐机
    有效前缀命中 期望 ~1/N 0（整段重算）
    Prefill 量级 期望重算约 6567 tok ≈ 1.01s 满量 8192 ≈ 1.26s（多 ~250ms）
    热点排队（burst B=8） 摊到 4 台，均摊排队 ~0.5s 全落 A，均摊 ~(B−1)/2×1.26 ≈ ~4.4s（上界）
    主伤害 稀释 热点排队 ≫ 少命中那 250ms

    一句话：假阳性不是"少赚一点命中"，是"确定性灌爆一台空壳机"——比没有亲和的随机摊开更差。 4.4s 是"全 miss + 全落一台"的上界，非期望值。

    4.3 盈亏阈值：为何"精度比省一跳值钱"¶

    中心 /query 常态目标 P50<5ms、P99<20ms（0.2s 是超时上界非常态）；本地树一次级联 miss 可到 ~1s。令中心多付 5ms > p×1000ms → 仅当误判率 p<0.5% 时"省一跳"才划算；而 tools/chat-template 字符错位在生产是常态，做不到 <0.5%，故**精度比省一跳值钱**。本质判断：错一个 block = 成百上千 token 重复 prefill。

    4.4 Motor vs 本地近似树¶

    维度 Motor + Conductor SGLang / vLLM Router cache_aware
    索引 中心真值（token/block 级） 调度侧本地近似字符 radix
    驱逐感知 ✅ 订 BlockRemoved（+ replay） ❌ 系统性不知引擎 LRU
    匹配误差 tokenize 一致性风险（部署约定可控） 字符≠token，tools 易级联 miss
    假命中主风险 事件延迟/Conductor 残缺 → 假阴为主 假阳系统性 + 字符误差
    失败错向 残缺 → 少认命中 → 退化 LB（安全） 假阳 → 灌空壳机（危险）
    运维税 Conductor + ZMQ + replay 无中心组件，但正确性靠时间淘汰

    一句选型：猜缓存（近似树）省一跳、假阳无界；查缓存（Conductor）付毫秒级查询，错向偏向安全假阴。

    4.5 有了 Removed 是否零假命中？¶

    否。仍有：丢包无 replay、AllBlocksCleared 未实现（如 AIBrix 空实现）、决策–Stored 空窗、Scheduler 用陈旧 prefill_cost 的短窗高估。但 Motor 路由错**只伤性能**（引擎 miss 重算），不伤正确性；要稳还需 utilization 门控 + 可选 speculative TTL。利用率耦合一句：亲和灌热 → 利用率↑ → 驱逐风暴 → 假阳更猛，故 utilization 是稳定性约束不是装饰项。

    五、ZMQ KV-Events（通知面，不搬 KV）¶

    5.1 定位¶

    ZMQ KV Events 是引擎把"哪个 block 写入/踢出"实时广播出去的**通知流**——不是 KV 数据面。Mooncake Conductor（或 Dynamo Indexer 等）SUB 这条流，养出"谁手里有什么前缀"的真值索引；Motor 只 HTTP /query 读结果打分。搬 KV 是 TE/Connector 的事。

    5.2 三类事件¶

    事件 索引动作 亲和含义 关键字段
    BlockStored 按 hash 链插入/延长前缀 M(w) 可上升 block_hashes、parent_block_hash、token_ids、block_size、lora_name、medium
    BlockRemoved 删对应 hash M(w) 立刻下降 → 砍假阳 block_hashes、medium
    AllBlocksCleared 整实例/池清空 该 worker 前缀视图清零 —

    medium 标 GPU/CPU/DISK，供分层命中权重——池化与亲和的接头字段。事件按调度步聚合成 KVEventBatch（带时间戳 + 单调递增 seq）批量发布。

    5.3 PUB + replay¶

    引擎 ZmqEventPublisher
  ├─ PUB  endpoint          ──► Conductor SUB（养 PrefixCacheTable）
  └─ ROUTER replay_endpoint ──► 订户发现 seq 缺口 → 要"从 N 起重放"
     （内存留最近 buffer_steps 批）
    • 推：ZMQ events 养索引；拉：HTTP /query 查索引。
    • PUB/SUB 不保证**永不丢（HWM 满可丢）→ 靠 **seq + replay 增量补拉，不用全量重建。这也是 Conductor 重启后能快速恢复、Motor 注册时要同时上报 replay_endpoint 的原因。

    5.4 per-DP publisher¶

    DP 部署下**每个 DP rank 起独立 Publisher**（端口按 rank 偏移），避免重复事件、保证事件归属——与 Motor"每个 endpoint 按 基础端口 + endpoint.id 单独注册"严格对齐，是 DP rank 粒度亲和成立的物理基础。TP 多 worker 场景另有 KVEventAggregator 去重：只有所有 TP worker 都上报的事件才算数（KV 在 TP 组内是分片的同一份逻辑缓存）。

    5.5 请求路径 6 步¶

    1. 引擎写满 block → BlockStored → ZMQ PUB
2. Conductor 更新 PrefixCacheTable
3. 新请求 → Motor tokenize（同源）
4. POST /query(tokens) → 各 DP longest_matched
5. unified / load_gated 打分选机
6. 落点再写新 block → 回到 1；驱逐则 BlockRemoved

    六、PrefixCache 引擎内核（与跨实例亲和两层正交）¶

    6.1 链式 block hash¶

    vLLM 用 PagedAttention 把 KV 切成固定 block。Prefix Cache/APC 在满块上算链式 hash：

    block_hash_i = H(parent_hash_{i-1}, tokens_in_block_i, extra_keys)
extra_keys   ≈ LoRA / multimodal / cache_salt   # 防串缓存

    Indexer/事件侧常等价写成滚动指纹：local_hash[i]=H(tokens of block i)；seq_hash[i]=H(seq_hash[i-1] ‖ local_hash[i])。BlockStored 常带 block_hashes + parent，与引擎链一致供 Conductor 加边。

    要点：满块才入库（短于 1 block → 本机 0 命中）；遇 miss 必须停（因果，后面即使同 hash 也不可用）；故意不去重 block ID 保 append-only 稳定；block_size 引擎常见 16、Motor/Conductor 常配 128——索引与引擎必须一致，不一致则精确亲和静默全 miss、退化 LB。

    6.2 APC 定义与"手动"意象对照¶

    APC = 已算完的 KV block 自动留在本实例 GPU；新请求 token-block 前缀相同则自动跳过对应 prefill——不解决多 replica 打散。

    APC（vLLM 内核） "手动/插件"意象
    谁决定可缓存 满块自动 hash 入表 业务/插件显式截断、get/put
    输入是否改写 Scheduler 隐式跳过已算 token MindIE 等：Python 手动截断 input_ids
    跨机 无原生；靠事件+路由或 Connector 可有 MemPool/远程块

    三者别混：近似 cache_aware 是路由器**猜谁有**；APC 是引擎内**真复用**；Conductor 是**元数据真值**。

    6.3 驱逐与引用计数¶

    请求命中块 → ref_cnt++
请求释放/结束 → ref_cnt--
LRU 候选：仅 ref_cnt == 0 可驱逐
→ 公共前缀 ref 高 → 后缀先死、前缀后死

    全命中也至少留 1 token 算 logits，尾块可能对齐重算。抢占（v1 常见）：整段 PREEMPTED → num_computed_tokens=0 recompute（无 SWAPPED）。

    三层驱逐别混成同一个 LRU：引擎 BlockPool 逐 GPU 物理 KV block；Conductor 索引删元数据边；Mooncake Store 逐共享池对象/lease。

    6.4 两层正交（必背）¶

    ┌─ 层 A：引擎 APC / PrefixCache ──────────────────┐
│  同机链式 hash → 共享物理块 → 跳过 prefill        │
│  事实源：BlockPool；发 BlockStored / Removed      │
└────────────────────────┬─────────────────────────┘
                         │ ZMQ 通知面（元数据，不搬张量）
                         ▼
┌─ 层 B：跨实例亲和 ──────────────────────────────┐
│  Conductor PrefixCacheTable；Motor /query + 打分 │
│  目的：把请求送到"层 A 存货最长"的实例            │
└─────────────────────────────────────────────────┘

    金句：机内 prefix cache vs 跨实例亲和 = 同机复用 vs 路由共址；正交。APC 不解决 N 机 RR 稀释；Conductor 聚全局 hash 视图，Motor 送回最长前缀机——命中后少算多少仍由该机 APC 决定。亲和优化命中概率与负载，不替代引擎内核。

    A/B 红线（测亲和净收益）：两组都开 Prefix Cache，只切 LB ↔ KVA，否则测到 APC + 亲和叠加。

    6.5 vLLM 是 hash 链，不是 Radix¶

    vLLM = hash map + 链；SGLang = Radix Tree。勿背成"vLLM=RadixTree"。SGLang 引擎树：边存 token 段、value=GPU KV 物理索引；match_prefix 做 LCP，边中命中就 _split_node；lock_ref 沿父链保护，只驱逐 lock_ref==0 的叶子。调度用 Cache-Aware LPM 把共享前缀请求批近。两者同一目标（少重复 prefill），vLLM 偏 block 内容哈希整块复用，SGLang 树形变长边 LCP 更细但分裂/级联锁更重。

    七、Mooncake 三层 + Store 驱逐副本 + Connector 对照 + 传输成本直觉¶

    7.1 三层协作（Conductor / Store / TE）¶

    Mooncake 不是单体，是三层共享同一底层 Transfer Engine：

    层 一句
    Conductor 调度层兼全局 KV 索引——订阅引擎 kv-events 维护 PrefixCacheTable，按前缀命中 + 负载 + 迁移代价选实例，查"谁有这段前缀"+选路，不负责搬 KV
    Store 把闲置 DRAM/SSD 组成分布式 KV 池，Master 管元数据与淘汰，数据面点对点直传；管"KV 放哪、何时逐"，Get/Put 走 TE
    Transfer Engine (TE) 零拷贝传输地基（RDMA/TCP/NVMe-oF/NVLink/CXL/ascend 等后端，拓扑感知选路）；管"怎么搬"

    Motor 只对接 Conductor 的查询做路由决策；真正的存/传由引擎侧 Connector 调 Store/TE。此外 Mooncake 还有 EP/PG（面向大规模 MoE 容错专家并行与分布式通信）、P2P Store（权重同步，演进为 checkpoint-engine）、TENT（TE 下一代实现），与当前 KV 亲和场景**正交**。

    与 Motor 边界（红线）：Motor = 决策层；D 实例不注册 Conductor；TTFT−70% 是代表性测算，不是 Mooncake 论文数字也不是客户 raw log。

    7.2 Mooncake Store 驱逐与副本（抽查级）¶

    Master 管账本，Client 搬砖：Master 只做元数据——key→副本列表、分配、放置、后台驱逐、Segment 上下线、HA；真正 GB 级数据由 Client 经 TE 点对点直传，不经 Master。

    Put:  Client → PutStart(分配) → TE 写入副本 → PutEnd(COMPLETE)
Get:  Client → GetReplicaList → TE 从 COMPLETE 副本读
    概念 一句
    Replica 可读 仅 COMPLETE；INITIALIZED/PROCESSING 半成品不可读
    busy refcnt>0 → 驱逐跳过
    soft / hard pin soft=最后逐；hard=禁止逐（如系统 prompt）
    写入模式 单副本 / DRAM+SSD 双副本 / 多副本可靠

    驱逐是可插拔策略 + 后台 BatchEvict：内置 near-LRU（访问 UpdateKey 移头，尊重 lease/soft pin 优先级），跳过 busy 与 hard pin，soft pin 最后才逐。热点靠 ReplicateConfig 加 DRAM/SSD 副本分散读；冷数据降副本甚至逐出。Segment 优雅下线（扩缩容）：OK → DRAINING（停新分配、仍可读）→ DRAINED → UNMOUNTING，给在用副本自然读完/迁完的窗口。

    生产配置例（llm-d 部署对照，非自家压测）：高水位约 0.95、每轮逐约 0.05、soft pin TTL 约 30min。

    边界：Conductor 管"去哪"（收 Removed 改索引，不直接逐），Store 管"在哪/何时逐"（BatchEvict 真正释放空间），Motor 只查 Conductor。禁语："Conductor 负责把冷 KV 删掉"——删空间是 Store，Conductor 最多丢索引条目。

    7.3 Connector 对照（vLLM vs SGLang）¶

    Connector 不是 Mooncake 官方组件，是引擎为"消费"Mooncake 能力自己写的适配层（调 mooncake.engine.TransferEngine / mooncake.store.MooncakeDistributedStore）。两边底层都调同一 mooncake wheel；差别在**产品化深度**。

    维度 vLLM SGLang 谁更深
    PD 传输粒度 Block 级；Prefill 整段算完再批量传；wait_for_layer_load/save_kv_layer 空实现 → 无分层流式 Page/chunk 级；enable_overlap 时中间 chunk 边算边传 SGLang
    握手/建连 Proxy round-robin 配对 + 请求体 kv_transfer_params HTTP Bootstrap Server 拓扑注册 + ZMQ 传元数据，两阶段更规范 SGLang
    跨实例前缀 MooncakeStoreConnector 原生；可与 PD Connector 经 MultiConnector 拼接 HiCache L1/L2/L3 三级树，Store=L3；PD 与 HiCache 共享同一 TE 实例 SGLang
    路由 cache-aware 示例 proxy ≈ itertools.cycle 轮询，无前缀感知 Rust Gateway 有 CacheAware（近似前缀树+负载），不直查 Mooncake 元数据 SGLang

    归属金句：vLLM main 已把 MooncakeConnector/MooncakeStoreConnector 原生注册进 factory，不再靠 kv_connector_module_path 外挂——代码归属追平 ≠ 传输策略追平（策略仍是整段算完批传）。禁语："vLLM 还要外挂 mooncake-wheel"、"代码进主仓 = 已流式分层传"、"SGLang CacheAware = 查 Mooncake 元数据"、"Connector 是 Mooncake 三层之一"。

    7.4 传输值不值得（成本直觉）¶

    能不能抓：能。调度失手落到冷 Prefill 节点，仍可走 Store Get()——KV 是集群共享对象，查 GetReplicaList 再经 TE RDMA 拉过来，只对未命中尾部做增量 prefill。这是"缓存追计算"；Conductor 路由那条是"计算追缓存"。两条正交别混。

    该不该传：重算 FLOPs 随前缀 p 平方**涨（attention），传输量只随 p **线性**涨 → **p 越大越该传。论文带宽阈值：LLaMA3-70B、前缀 8192，8×A800 约 6 GB/s、8×H800 约 19 GB/s；现代 RDMA 聚合常 100+ GB/s，比阈值高一到两个数量级——长上下文几乎总是传更划算。实测建议下限：集群总带宽 <100 Gbps 时 TTFT 易恶化。GPU 更快阈值反而变高：阈值 B/G 中 G↑ 需更大 B 才喂饱，符合"算力涨、网络门槛跟着涨"。

    何时不该传：前缀太短（握手/查表固定开销盖过省下的算力）、网太差、单副本热点拥塞。

    谁在算账：

    层 "该不该传"怎么做 有无 Ttransfer
    Mooncake Conductor Algorithm 1 估 TTFT=Ttransfer+Tqueue+Tprefill，超 kvcache_balancing_threshold 才跨机传 有
    Motor KVA score=scale×max(0,isl−credit×matched)+w×load 无；credit 默认 1.0 隐式近乎免费
    vLLM StoreConnector lookup 有更长命中就拉 无；二元存在性

    一句话：Motor 简化，Conductor 精细。分层合理——贵决策只在调度层算一次，执行层老实拉。Motor 侧用 credit 近似"命中≈省算力"，依赖集群 RDMA 远高于论文阈值这一物理前提；真要精细账那是 Conductor Algorithm 1 的活。风险：Solo 挂 StoreConnector、没有上层成本模型 → 退化"有命中就传"，短前缀可能亏。分层流式传输与增量 prefill pipeline 重叠时，墙钟上的"额外等传"可趋近于零。

    八、Conductor 超时与回退¶

    8.1 亲和是优化不是依赖¶

    Motor 选中心化 Conductor 查真索引，就要付"一跳 HTTP + 组件可用性"——设计原则是**亲和是优化不是依赖**。查询客户端超时 0.2s（200ms）是配置，不是已测 P99。超时/无数据/tokenize fail-closed → 整条亲和返回 None，调度回退 LoadBalance，请求主路径不堵。

    数字 含义 标注
    0.2s / 200ms query_conductor 客户端超时 配置
    注册超时 2s /register 预算更宽 配置
    P50 <5ms / P99 <20ms 常态查询目标 目标·非实测
    回退率目标 <0.1% 超时/无索引占比 目标·非实测

    上场话术：200ms 是客户端预算天花板，健康时应远小于它——P50/P99 还没补齐曲线，只报目标与回退语义，不把 200ms 说成已测 P99。

    8.2 回退链（不拖垮路由）¶

    触发（任一）: Conductor 超时/挂/无匹配数据 · tokenize fail-closed → []
       ↓
亲和路径返回 None
       ↓
LoadBalance（全局最小负载，非 RR）
       ↓（再异常）
round_robin 兜底

    8.3 LoadBalance 是什么（被追"回退到什么"）¶

    1. 不是 RR：摊平所有 endpoint，heapq.nsmallest 取负载分最低者。
    2. 复合分：endpoint_score + weight × (instance_score / endpoint_count)，endpoint_instance_score_weight 默认 0.05——感知"同机其他 DP 已满"的整机压力，除以 endpoint 数是为不惩罚 DP 多的大实例。
    3. 与 unified 连续：LB = unified 去掉亲和项，负载分与第二项同源，量纲仍是 token。
    4. 平局旋转：start_index 按 worker 编号旋转遍历起点，多 worker 冷启动/全空闲时不齐刷（herding 低配防护）。
    5. 同受 Scheduler 仲裁：LB 候选也走"worker 提案 + scheduler 权威重选"通道。

    公式为什么能相加。 unified 中的 prefill_cost=max(0, isl-credit×matched) 表示还要计算的 token 数，load_cost=active_tokens+0.3×active_kv_cache 表示已排队的 token 等价工作量；两者统一为 token 量纲，才可用 scale×prefill_cost + weight×load_cost 取最小。默认三个系数都为 1，是“命中 1 token 省 1 token Prefill、排队 1 token 等于待算 1 token”的中性假设；0.3 和实例压力权重 0.05 都是可校准的启发式常数，并非理论常数。A 命中 m、B 零命中时，B 获胜阈值是 load_A-load_B > scale×credit×m/weight。完整推导、LB 复合分和校准方法见专题 12 的 §2.6.1。

    Conductor 重启：/services 对账重注册 + kv-events replay 补索引。D/E 非 KVA 角色本来就走 LB。

    8.4 追问要点¶

    • "200ms 是不是太大会拖 TTFT？"→ 健康路径应远小于 200ms；200ms 是故障快失败预算，宁可偶发放弃亲和也不让慢 Conductor 把调度打成秒级。真拖 TTFT 的是超时后走满量 prefill，不是超时值本身——要盯回退率与命中率，而不是把超时调成"假装很低的 P99"。
    • "回退 LB 会不会制造 herding？"→ LB 有最小负载 + 平局旋转 + Scheduler 权威，比"假阳灌空壳机"安全；亲和 herding 另用 overlay→top-k→unified 修，超时回退是可用性层不是打分层。

    九、PD 分离 / 混部选型与 handoff¶

    9.1 三条第一性 + 传输税¶

    PD 混部（ROLE_U/union）：同一实例又做 Prefill 又做 Decode——零 KV 跨机传输、工程简单；代价是 P compute-bound 饿死 D、无法按 ISL/OSL 独立扩缩，长 ISL/高并发下 TPOT 长尾易炸。

    PD 分离（P 池 + D 池）：干扰消除 + 资源异构 + 独立扩缩**三条第一性；付 P→D 的 KV handoff 税。**主收益是隔离 + 扩缩，不是"传一定比算快"。

    9.2 何时混部/何时分离¶

    信号 倾向混部 U 倾向分离 P+D
    ISL/OSL 短 prompt、OSL 不极端 长 ISL、短 OSL、P/D 比例需弹性
    卡数/拓扑 少卡、同池够用 多卡可拆池、按负载比扩 P 或 D
    网络 无 RDMA/带宽紧 RDMA 或有效 BW 够扛 |KV|
    干扰 可接受同机争用、可先 chunked P 饿死 D 已成痛点、要稳 TPOT
    运维税 少组件、无 transfer connector 愿付 connector + handoff/reschedule

    一句选型：能忍同机干扰、传不起或不值得传 → 混部/chunked；干扰与扩缩已成瓶颈且有传输栈 → 分离；两者都不是"传比算快"才上分离。 与 chunked prefill 是**同题不同解**——chunked = 混部低成本切块；PD 分离 = 彻底隔离 + 独立扩缩、付传输税。

    9.3 handoff vs concurrent¶

    维度 handoff concurrent / layerwise
    切换点 P 完成 + KV 就绪再调 D P/D 同时启动，逐层 sync
    D 能否提前跑 不能，等远端 KV 可启动 forward，每层 wait_for_layer_load
    TTFT 直觉 T_p + T_tx ≈ max(T_p, T_tx)
    D 侧典型状态 WAITING_FOR_REMOTE_KVS 按层等待 load
    Motor capability prefill_handoff_decode（Mooncake/NIXL） concurrent_engine_sync（MoRIIO/Layerwise）
    落地注意 实现简单、链路清晰 vLLM MooncakeConnector 的 layer hook 空实现时，"逐层流式"会打成 request 结束后 batch 传；真正逐层看 MoRIIO

    链路简单/兼容优先 → handoff；传输能与 P 重叠且 connector 真支持逐层 → concurrent。选型别把空 hook 的 MooncakeConnector 说成已逐层。

    9.4 handoff 成本与临界点¶

    |KV| ≈ 2 × L × N_layers × H × dtype_bytes
T_tx = |KV| / BW + T_handshake
T_recompute ≈ L × t_prefill_per_token
传优于"重算这份 KV"当：T_tx < T_recompute

    一句成本：handoff 税 = 等 P 算完再搬 KV，T_tx = |KV|/BW + T_handshake，TTFT ≈ T_p + T_tx——常与 prefill 同量级甚至更大，主买单的是 D 不被饿死 + 独立扩缩。

    数量级例（文档数量级例，非客户实测）：70B、L=8K、80 层、H=8192、FP16 → |KV|≈21GB；有效 BW~10GB/s → 传 ~2s；同配置 prefill 或仅数百 ms——此例传更慢，但 D 池不被 P 阻塞仍可能值得。改善临界点：RDMA、只传 delta（KVA 最长前缀命中后的未命中 suffix）、真 layerwise overlap。

    9.5 Motor 对 P / U / D 的注册差异¶

    角色 注册 Conductor？ KVA 选机？ instance_id 形态
    P（分离 Prefill） ✅ ✅ role==ROLE_P vllm-prefill-{id}
    U（混部 union） ✅ ✅ _KVA_SELECT_ROLES={P,U} vllm-union-{id}
    D（分离 Decode） ❌ 不注册 ❌ 走 LoadBalance / CPCD 另选 —

    为什么 D 不注册：prefix 索引只在写入侧有意义；D 只消费远端 KV，注册上去会污染"谁写了可复用前缀"的语义。 配套工程口径：注册粒度 = 每个 DP endpoint（基础端口 + endpoint.id）+ replay_endpoint；PD 下 prefill 完成可即时释放 P 负载（不等整请求结束），CPCD 可 prefill 后再选 D、基于更新鲜的负载；P/D capability 无交集 → fail-closed 503。Prefix × 分离叠加：KVA 选最长前缀 P → 只算 suffix → 传 delta。

    9.6 调度器如何处理"等远程 KV"的请求¶

    vLLM V1 调度器每 step 先调度 RUNNING 队列、用完 token_budget 才轮到 WAITING；处于阻塞态（WAITING_FOR_REMOTE_KVS/等 grammar/等流式）的请求隔离到 skipped_waiting 子队列，靠 Worker 侧 KVConnector 汇报 finished_recving（如 MooncakeConnector 判断 RDMA 传完）事件驱动状态迁移（WAITING_FOR_REMOTE_KVS → WAITING），下一循环自然捡回——事件驱动 + 每步一次低成本轮询检查，不是忙等。

    十一、语义路由与强弱模型¶

    先分两层，避免和 Motor 实例级 KV 亲和答串：

    层次 项目直觉 决策依据 解决什么
    实例级 production-stack vllm-router / Motor 前缀哈希、KV 元数据、QPS、session 选哪个**副本**
    模型级 vllm-project/semantic-router 意图/领域/难度/安全等信号 选哪个**模型**（强/弱/预算）

    面试官问的"语义/强弱分发"是**模型级**（第二层）；Motor KV 亲和是**实例级**（第一层）。

    难度信号（强弱分发关键）：

    预置：hard 例句集 embedding / easy 例句集 embedding
请求：q → embedding(q)
δ = max sim(q, hard) − max sim(q, easy)
→ 阈值切 easy / medium / hard

    决策引擎用布尔规则组合信号，例（官方部署口径级，勿背成自家指标）：domain: math AND complexity: hard → 强推理模型 + 高预算；complexity: easy/medium → 中/弱模型 + 低预算；安全/jailbreak 信号 → 拦截或走专用策略。之后在候选模型集里做质量-成本选择（rating / 级联 AutoMix / 延迟感知）。

    与 KV 亲和正交：亲和优化"同模型副本间的缓存命中与负载"；语义路由优化"跨模型的质量-成本"。可叠：先语义选模型（或档），再在该档实例池内做 prefix/KV-aware。别说成"Motor 已做强弱分发"（未做）、别说成"语义路由替代亲和"（层不同）。Motor 目前只做同模型多实例 KV 亲和；强弱分发是值得补的一层，最小切入是"规则 + 轻量难度/意图分类"粗分到 2~3 档，强档池内继续跑现有亲和，先可观测再谈 RL 选型。

    十二、HCCL 边界（抽查级）¶

    原则：边界 + 正交 + 诚实"通信非主战场"。协议/线程模型/hccn.conf/transportMem* API 细节指深文，不上场背。

    昇腾上 KV 还是两件事：亲和**决定"去哪台"（Motor 查 Conductor），**传输**决定"块怎么搬"（引擎 Connector 调 Mooncake TE）。**HCCL 别混两层：训练/推理里 backend=hccl 的 TP/EP 集合通信（AllReduce 等）≠ PD/池化里 TE 封装的**点对点 KV 搬块**。同叫 HCCL 生态、语义不同。

    元数据面：KV Events → Conductor → Motor /query   （谁有前缀）
数据面：  Connector → Mooncake TE (ascend/…)     （怎么搬块）
并行面：  ProcessGroup HCCL / TP AllReduce        （组内同步）

    与 TE 关系（只背一句）：Mooncake TE = 传输地基；HCCL 在此是 ascend 数据面后端之一，不是 Conductor，也不是"我写的亲和算法"。诚实边界：主战场是调度亲和与结构化约束交付，通信栈是协作归因语言——能分清层、能跟 Profiling 对齐，不装生产写过 HCCL transport。

    抽查追问： - Profiling 见 HCCL/NCCL 长 → 先分 TP 集合 vs PD 传 KV，再调并行度/overlap 或 PD 路径，勿"加卡"一句了事。 - 910B Prefill + 别的卡 Decode → 异构常不能 HBM 直达，叙事是聚合 → Host DRAM → RDMA → 对端，拷贝与传输可流水线；讲模式不报自制带宽数。 - MindIE 一定走 Mooncake TE 吗？→ 不一定。vLLM-Ascend 路径常接 Mooncake Connector+TE；MindIE 另有 LLMDataDist 一类原生 PD 数据面。别把所有昇腾 PD 都说成 Mooncake。

    金句：亲和正交于传输；集合通信 ≠ KV Transfer；通信非主战场、边界要清。

    十四、第三层追问（严格面试官）¶

    ① "亲和 + PD，TTFT 公式怎么写？70% 从哪来？" 先降调：70% 是 Qwen3-32B、TP4、ISL=8K、共享前缀约 6.5K、高复用下的**代表性测算**。公式（亲和收益，与 PD 传输税分开讲）：TTFT ≈ c₀ + T_prefill_full × (1−h)。取 T_prefill_full≈1230ms（单 P ~6.5k tok/s）、c₀≈80ms；命中率 h 从 0.10→0.78 → TTFT 约 1187ms→351ms（−70.5%），E2E 约 1712→876ms（−48.9%）。机制可用 cached_tokens 与调度日志证；没有留存可复核客户 A/B 包，所以不说成实测。复测会两组都开 Prefix Cache、只切 LB↔KVA，报 P50/P95/P99、cached ratio、负载倾斜、回退率。勿说：把 Prefill 少算说成"算子优化了 70%"、把示例拓扑说成"已跑通客户压测"。

    ② "假阳性一次 TTFT 到底差多少？" 拆两段：少命中（满算 vs 1/N 期望）约多 ~250ms prefill（ISL=8K、6.5k tok/s 假设）；热点排队在同前缀 burst 下可到数秒量级上界（burst=8 全灌一台均摊 ~4.4s 上界，非期望），后者常是主伤。全部标测算/机制推导。

    ③ "有 Removed 是否就零假命中？" 否。仍有丢包无 replay、AllBlocksCleared 未实现、决策–Stored 空窗、Scheduler 陈旧 cost。Motor 路由错只伤性能（引擎 miss 重算），不伤正确性；要稳还需 utilization 门控 + 可选 speculative TTL。

    ④ "为何不能用 load_weight 调出 load_gated？" 字典序（先负载、再亲和）无法用有限加权和表达；weight→∞ 会压死亲和变纯 LB。必须单开模式。

    ⑤ "为何 gated 不升 V3 全局重排？" 软 combined 会松绑"不出最闲集合"的硬上界；故意 prefill_cost=None、钉 V2。

    ⑥ "传 2s、prefill 0.5s 还做 PD？" 承认单请求墙钟传可能更贵；买单的是 D 池稳态与独立扩缩、长尾 TPOT。短 L 改口混部/chunked；改善：RDMA、传 delta、layerwise overlap。

    ⑦ "混部也开亲和，和分离比收益差在哪？" 机制上 U 同样查 Conductor；但混部 P/D 争用 + 驱逐更凶，前缀命中更不稳；分离 P 池缓存空间通常更稳、却多 T_tx。TTFT−70% 是高复用测算场景，勿说成"分离一定多 70%"。

    ⑧ "你们和 SGLang 那套近似树 / SGLang router 谁准？" 先框层：引擎 RadixCache（真 KV indices）/ Gateway cache_aware（文本历史近似）/ 实验 sgl-router 的 cache_aware_zmq（吃 ZMQ）/ HiCache（worker 内分层，≠ Gateway 精确路由）/ Motor Conductor（查真索引、双 tokenize）。字符树在 tools/模板注入处易从错位点起整链 hash 分叉 → 级联满 miss，且无 Removed → 假阳无界；Motor 付 Conductor 税换真值与安全错向。

    ⑨ "你们 PD 是 handoff 还是 layerwise？证据？" 先说部署 capability（prefill_handoff_decode vs concurrent_engine_sync）；再承认 connector 实现决定"纸面 layerwise"是否真逐层——vLLM MooncakeConnector 的 layer hook 空实现时会打成 request 结束后 batch 传，真正逐层看 MoRIIO。

    ⑩ "TransportMem / hccn.conf 讲一下？" 边界级：身份要绑到具体 NPU、设备网 IP 表给 TE 用；细节按深文补，不上场背 API 名。交付口径是分层正交，不是 transport 作者。

    ⑪ "难度分类错了会怎样（语义路由）？" easy 判 hard → 成本虚高；hard 判 easy → 质量掉点。靠阈值/人工规则兜底、级联（先小后大）、生产反馈校准；不报未验证准确率数字。

    ⑫ "PyMotor 为何双 tokenize？错了会改输出吗？" Coordinator tokenize 只查 Conductor/估负载，首请求**不**注 input_ids；不一致 → 亲和 miss，**不**改模型输入（engine 自己 tokenize）。所以 tokenize 分叉只伤命中率、不伤正确性。
