    Q1 · Prefill vs Decode：算力还是访存？¶

    栏 内容
    问 Prefill 和 Decode 在算子层瓶颈差在哪？
    45 秒答 看 OI=FLOPs/Bytes。Prefill：一份 KV 服务很长 Q，OI 高→计算密集，典型用 PFA（切 Q），Cube 易打满。Decode：每步 Q≈1，搬权重+KV 只算一行，OI 低→访存密集，典型用 IFA（切 KV），Cube 常吃不满。优化口令：Prefill 打满算力；Decode 打满带宽 + 降 launch。
    易踩坑 说「Decode 算力不够」；把 Prefill/Decode 说成两种算法而不是同一套 attention、tiling 方向相反。
    深挖 suanzi/06 A2/C3；suanzi/08 §5；07-10/02 §2；术语 Q11

    Q2 · FlashAttention / online softmax 直觉¶

    栏 内容
    问 FA 为什么快？online softmax 为何等价？
    45 秒答 朴素路径把 O(N²) 的 S/P 反复进出 HBM。FA：tiling（块驻片上）+ 块内融合（QK→softmax→×V）+ online softmax，主要写回 O。online：维护 running m,l,O；新块 m'=max(m,块max)，scale=exp(m-m')，旧 l/O 乘 scale 再累加块内项，最后 O/l——与整行 safe softmax 数学等价。
    易踩坑 背「FA 就是更快 softmax」；说 online 是近似；把 FA 与 PagedAttention 说成二选一（FA=怎么算，Paged=KV 怎么存）。
    深挖 suanzi/06 C1/C2；suanzi/08 §4（两块手算）；07-10/02 §1；术语 Q4/Q5

    Q3 · PagedAttention 与 block_size¶

    栏 内容
    问 PagedAttention 算子层谁做？block_size 怎么说？
    45 秒答 KV 按页分配，靠 block_table 逻辑→物理。写：常 scatter_pa_kv_cache（可融进 rope_cache）。读：多数路径 FA/IFA 内核内按表间接寻址，不一定先 gather。block_size=每页 token 数（常见量级如 16/128）：太小索引开销大，太大碎片浪费。和 Continuous Batching 配套。
    易踩坑 说「每次 Decode 都 gather→FA→scatter」；把 block_size 说成「越大越好」。
    深挖 suanzi/06 D1；suanzi/08 §8；suanzi/03 §2.1；术语「Paged」

    Q4 · MLA absorb 与 M=128¶

    栏 内容
    问 MLA 为何改善 Decode？M=128 满载等于计算密集吗？
    45 秒答 MLA 把 KV 压到约 576 维 latent，砍显存与读带宽；absorb 下多 head 共享 latent，BMM 的 M 可拼到 head 维（如 128），Cube fractal 不再只填 1 行。但：fractal 满 ≠ 整步计算密集——短上下文 W_absorb 仍可主导搬运，整步仍偏访存；长上下文 + MTP 才更可能翻转。
    易踩坑 背「MLA 一律计算密集」；把 mla_preprocess（Prefill 写 latent）和 mla_prolog（Decode 准备）说成同一个。
    深挖 suanzi/06 F1/F2；suanzi/08 对照表 MLA 行；suanzi/03 §2.4–2.6；suanzi/19

    Q5 · 大 batch：FFN vs Attention¶

    栏 内容
    问 大 batch 对 FFN 和 Attention 一样吗？
    45 秒答 不一样。 FFN 共享权重：多请求拼大 M，权重搬运被摊薄（粗算 OI 近似随 M 涨），单核 Cube 变饱。Attention：各请求 KV 不同，不能跨请求拼 KV/M，主要靠多核并行打满；要拉 Attention 的 M 靠 GQA/MLA（head） 或 MTP（token）。
    易踩坑 「continuous batching 把 Attention 也变成计算密集」；把 TP 说成切 M（TP 不切序列维，切序列是 SP）。
    深挖 suanzi/06 A3；suanzi/08 §3 + 对照表；术语 Q12

    Q6 · GE / aclgraph / CUDA Graph（昇腾语境）¶

    栏 内容
    问 aclgraph、GE、CUDA Graph 边界？怎么选？
    45 秒答 aclgraph：运行期 Capture&Replay，只省 Host 逐 kernel 下发；不融合，Device 负载基本不变。GE：编译期整图优化（融合/内存复用/多流），真降 Device 负载。二者可叠加。NVIDIA CUDA Graph 思想同为录制重放；Paged 动态元数据都要特殊处理（piecewise / update metadata / update_attn_params）。选型：Host-bound（小 shape、Decode、层多）→ 先 Graph；Device 忙且可融合/显存紧 → GE/融合。
    易踩坑 「aclgraph 会融合」；「aclgraph 替代 GE」；已 Device-bound 还指望只开 Graph 大涨吞吐。
    深挖 suanzi/06 B1–B4；suanzi/08 对照表；07-10/02 §3/§6；术语 Q8–Q10

    Q7 · MoE / MC2（一句能站住）¶

    栏 内容
    问 MoE 推理算子链？MC2 解决什么？你做过 HCCL 吗？
    45 秒答 MoE：gating → 按专家重排 → Grouped MatMul → 还原加权；跨卡再加 dispatch/combine。MC2（如 matmul_all_reduce）：计算与通信重叠，藏 AllReduce 暴露。诚实边界：未独立交付 HCCL；理解动机与调用位置，协议细节协作边界。
    易踩坑 把自己说成通信库作者；把 GMM 说成「一个大 dense FFN」。
    深挖 suanzi/06 G1–G5；suanzi/04；术语索引 04

    Q8 · 量化算子交界（一句）¶

    栏 内容
    问 量化在算子层卡在哪？W 和 KV 怎么分？
    45 秒答 Decode 访存密→降位宽直接减搬运/显存。分清：**W 量化**救 Linear 权重读；**KV 量化**救 cache 显存与长上下文读带宽——不是一回事。W8A8：权重常静态 INT8，激活常 per-token 动态；层间常 dequant 或融进下一算子。选型看 profiling + 精度验收。
    易踩坑 把 KV INT8 和 W8A8 混成一件事；只谈「量化更快」不谈精度/动态量化开销。
    深挖 suanzi/06 H1–H3；suanzi/05；suanzi/08 §10；01 C 节

    Q9 · Host-bound vs memory-bound（口令）¶

    栏 内容
    问 Decode 变慢，先查什么？
    45 秒答 先分界：NPU 闲 + 下发间隙大 → Host-bound → Graph；NPU 忙 + HBM 打满 → memory-bound → 量化/融合/压 KV；算力满带宽有余 → compute → TP/减算。再按链：Linear 权重？Attn 读 KV？通信可 MC2？小算子碎片？用 profiling 验证，忌无数据调参。
    易踩坑 一上来调 batch/量化；把 Host-bound 和 memory-bound 混用同一套药。
    深挖 suanzi/06 I2/J0；suanzi/08 §6；07-10/02 §0；01 D

    Q10 · PFA / IFA 切分与 reduce¶

    栏 内容
    问 Prefill/Decode 为何不同算子？IFA 为何要 reduce？
    45 秒答 同一套 online softmax，多核切分反向：PFA 切 Q 行块，各核写自己的 O，通常**不用合并 softmax 状态**；IFA 的 Q≈1，切 KV，各核只有局部 (m,l,O)，必须再按 online 规则 跨核 reduce 才得到全局正确输出。
    易踩坑 说 IFA「不需要 softmax」；不知道为何只有 Decode 路径强调 reduce。
    深挖 suanzi/06 C3；suanzi/08 §7；术语 Q6

    Q11 · Graph × Paged 动态元数据¶

    栏 内容
    问 Graph 要静态 shape，Attention 的 block_table 怎么办？
    45 秒答 捕获冻结 tiling/地址；block_table/seq_lens 每步变。解法：CUDA 侧 FULL+更新 metadata / PIECEWISE / Breakable + padding；昇腾侧 update_attn_params / TaskUpdate 一类 hook。结构化 mask/sample 动态多，常**留图外**，中间层可分段捕获。
    易踩坑 「开了 Graph 就不能用 Paged」；把 Schema 编译缓存和 aclgraph 混为一谈（前者省 CPU 编译，后者省 kernel 下发）。
    深挖 suanzi/06 B2/D2；07-10/02 §3；suanzi/08 编译缓存 vs aclgraph；05 G6

    Q12 · 「我没手写 AscendC」诚实边界¶

    栏 内容
    问 没写过 AscendC/HCCL，岗位怎么匹配？
    45 秒答 主战场是 框架/调度（结构化输出、KV 亲和、Tool Call、Server）。算子层：用 Roofline + 读源码 补齐 FA/online softmax、PFA/IFA/MLA 选型、GE/aclgraph、MoE/MC2 与量化动机，能把调度对齐到「算力还是带宽、该融合还是该捕获」，并与算子团队用同一套归因语言协作。不谎称独立交付生产 kernel / HCCL；bitmask 是框架编排现有 Vector 能力，非自研 AscendC FA/FFN。
    易踩坑 夸大成算子作者；被追问就空白——应主动划界并展示选型/归因能力。
    深挖 suanzi/06 I3/J；07-10/02 §6；suanzi/21 #8；术语 Q15

    自检清单¶

    • [ ] Prefill 计算密 / Decode 访存密，能挂上 PFA/IFA
    • [ ] online softmax 能说 m/l/O + scale，不说「近似」
    • [ ] Paged：scatter 写 + kernel 内读；非必须先 gather
    • [ ] MLA：~576 latent；M=128≠整步计算密集
    • [ ] 大 batch：救 FFN，不跨请求救 Attn
    • [ ] aclgraph 不融合；Host→Graph，Device→GE/融合
    • [ ] MoE/MC2/量化各一句 + AscendC 诚实边界 )
