    1 · 自我介绍 180s · 目标 2:40–3:00¶

    我目前在华为昇腾做 MindIE LLM **推理服务**开发，入职约一年，主线是服务侧系统问题：延迟、吞吐、缓存命中、正确性。三条可核交付：

    结构化输出 0→1：从 Schema 编译到采样硬约束整条链路我扛过——xgrammar 做字节级 PDA，再落到 bitmask 约束 logits。编译缓存按内容哈希，策略是 FIFO、默认 100，命中不调序。异步调度场景里出现过 mask 与 token 游标错位导致非法 JSON，我按线程/游标/顺序三因修过，验收是正确性，不拿未测 TPOT 百分比圆场。bitmask 在 NPU 上是 torch 组合 masked_fill_；fused 是可优化点，我不上场吹自研 kernel。线上落地后端是 xgrammar；guidance 多后端是预留抽象，不是「线上有 guidance」。MTP 与结构化在入口 硬互斥，不能一起开。个人累计交付量级简历写 5000+，钉在这条线，不和 Tool/Server 行数混算——author cloc 待补。

    Motor KV 亲和：问题是多实例下「猜缓存 vs 查缓存」。我们做 token 级精确亲和——先同源 tokenize（含 tools 透传、fail-closed），再查 Mooncake Conductor 全局索引，打分用 unified/load_gated，并演进过防 herding。相对 SGLang/vLLM Router 本地字符树，我们错向偏向安全假阴。PD 下 D 实例不注册 Conductor，只 P/U 上报。TTFT −70% 是按命中少 Prefill 的 代表性测算（例 1187→351 ms，场景绑定 Qwen3-32B、~8K、共享前缀~6.5K），公式与假设可讲；不是客户原始日志实测。Mooncake 我用边界讲：Conductor 查、Store 放、TE 传；通信/HCCL 细节 非主战场。

    Tool Call + Server：Tool Call 侧重协议适配与流式状态机解析，和结构化「生成期硬约束」互补；Server 侧 Handler/Interface 拆分，「约 1 万行」以 PR diff 为准，今晚不编 before/after。

    相对最熟是 KV 亲和 + 结构化 0→1。希望今天往调度、约束解码或 PD 任一方向深挖。

    禁踩：LRU/128；客户 raw log −70%；自研 fused；「线上有 guidance」；D 也注册。

    2 · 结构化输出 STAR · 目标 65–75s¶

    S：要在 MindIE 上把 Schema 约束做到采样硬保证；异步路径还出过 mask 错位。
    T：我扛 0→1：编译 → PDA → bitmask → 采样，验收钉正确性。
    A：落地 xgrammar；缓存 FIFO/100；修三因错位；bitmask 用 torch masked_fill_，不上场吹 fused kernel。
    R：这条线累计 5000+（待 cloc）；不是**客户测出 TPOT−x%。guidance 没接；MTP 和结构化**不能一起开。热路径 bitmask 占 TPOT <1%–3% 只是经验量级，A/B 待补。

    3 · Motor KV 亲和 STAR · 目标 75–90s¶

    S：多副本时本地字符树是猜；我们要在高重复场景少做 Prefill。
    T：做成 token 级精确亲和：查索引、可回退、可防扎堆。
    A：同源 tokenize → Conductor 最长前缀 → unified/load_gated；驱逐看 Removed；D 不注册。tokenize 4K 约 6 ms 是固定税；Conductor timeout 0.2 s 是**配置预算**，不是已测 P99。
    R：−70%/−50% 是 代表性测算（1187→351 类），**不是**甩客户日志。重复率不够则收益缩水——最脆是命中率 \(h\)。

    4 · Mooncake 三层 · 目标 55–65s¶

    Mooncake 不是单体，是三层协作：
    Transfer Engine 做零拷贝传输地基（RDMA/TCP 等，拓扑感知选路）；
    Store 在上面把闲置 DRAM/SSD 组成分布式 KV 池，Master 管元数据与淘汰，数据面点对点直传；
    Conductor 是调度层兼全局 KV 索引——订阅引擎 kv-events，按前缀命中 + 负载选实例。
    我们 Motor 只对接 Conductor 的查询做路由决策；真正的存/传由引擎侧 Connector 调 Store/TE。Connector 不是 Mooncake 官方组件，是 vLLM/SGLang 各自适配层。D 实例 不注册 Conductor；TTFT−70% 是亲和测算，不是 Mooncake 论文数字。

    5 · 投机演进线 · 目标 2:40–3:00¶

    Decode 小 batch 是 memory-bound：权重贵、算力闲。投机 = 便宜 draft 猜 k → target 一次前向并行验。单 token 延迟约 (T_draft+T_verify)/τ；大 batch 变 compute-bound，投机常不划算。

    Medusa：多头挂 target 顶层，省独立 draft；头间无块内依赖。
    EAGLE-1：特征层自回归 + 真实 token 反馈。
    EAGLE-2：置信度动态长草稿树，免重训抬 τ。
    EAGLE-3：直接预测 token + 多层融合，破 scaling；draft 仍串行。
    MTP：预训练联合草稿头，对齐好——前提模型自带权重。
    DFlash：block diffusion 一次并行出块，解耦 T_draft 与 k；易 suffix decay。
    DSpark：并行主干 + 低秩 Markov 补连贯 + 按 GPU 负载裁 verify 长度。

    MindIE 三插件：MTP / Lookahead / Memory Decoding——候选来源不同，verify 多走**贪心逐位比对**，≠ vLLM 默认概率拒绝采样。MTP×结构化：Serving 入口硬互斥。无自测曲线，**不报**自制加速比。

    6 · vLLM 按目标选配置 · 目标 55–65s¶

    我不背十类清单，按**三个目标**点旋钮：
    提吞吐：拧大 max_num_batched_tokens / max_num_seqs；KV 池水位；需要时 kv_cache_dtype fp8；chunked 混批抬利用率；async scheduling 消 GPU 空档。
    降 TTFT：enable_prefix_caching 吃重复前缀；长 prompt 用 chunked 防堵队列（注意 TTFT 本身可能略增）；多实例再加 KV 亲和；规模上 PD。
    降延迟尾（TPOT/P99）：cudagraph_mode 消 launch；speculative_config 只在低并发开；权重量化减访存；chunked + 控预算防长 prefill 饿死 decode。
    先 profiling 看 GPU 空还是满，再动配置——无曲线不报加速比。

    7 · 跳槽动机 · 目标 60–75s¶

    不是随大流看机会，是方向想清楚了：我要深耕**推理服务系统**。
    拉力：希望去把推理加速当核心竞争力、有真实大规模部署的团队——和我做过的多实例调度、TTFT 优化、结构化约束解码互补；我能带来 0→1 特性交付，想获得更深的系统优化与更大流量场景。
    推力：只基于业务判断——团队定位是配套昇腾硬件生态，自研引擎转向拥抱 vLLM，我预期未来一两年**框架层深度工作空间会收窄**；与其等方向进一步变窄，不如在刚完成结构化输出和 KV 亲和两坨交付、势能最好的时候换赛道。
    稳定性：方向对，我会沉下来长期做；这次看机会是锚定方向，**不是**嫌加班或跟风涨薪。待遇是结果不是动机，数字到 offer 再对齐。

    8 · 红线 8 条串读 · 目标 25–35s¶

    一、编译缓存默认 100，FIFO，命中不调序——不是 LRU、不是 128。
    二、tokenizer 在 Coordinator 本地 model_path 同源加载，不是运行时从引擎拉。
    三、bitmask 是 torch NPU 算子组合 masked_fill_，不是自研 fused kernel。
    四、xgrammar 是 字节级 PDA，仓内别随口说 FSM。
    五、MTP 与结构化 入口硬互斥，不能一起开。
    六、PD 下 D 不注册 Conductor，只 P/U。
    七、TTFT−70% 是 代表性测算，不是客户 raw log——除非拿出日志。
    八、线上唯一落地后端是 xgrammar；guidance 是预留，不是「线上有」。
