    1 · 60 秒电梯稿（可直接背）¶

    亲和假命中分两类：类型 A 驱逐滞后（假阳性）——索引还说「W 有前缀 S」，引擎已 BlockRemoved；类型 B 事件空窗——路由已决策、BlockStored 未进索引，多为假阴性，speculative TTL 则是可控假阳。[文档已有·kv/11]

    假阳性比 round-robin 更糟：本地近似树无驱逐感知时，会**确定性**把同前缀 burst 全灌到「它以为有、实已 LRU 掉」的实例——既满量重算，又制造热点排队；RR 至少按 1/N 摊开，还有偶然命中。Motor 走 Mooncake Conductor：引擎 ZMQ PUB BlockStored/Removed，索引实时摘块，/query 查真值——从根上砍假阳。[文档已有·13 / 09]

    代价是中心索引一跳 HTTP + Conductor 运维；残缺窗口错向是**假阴性**（安全退化 LB），不像近似树假阳**无界且不收敛**。Motor 缺口：利用率与 speculative 未进 unified 分——高 churn 稳定性还需补。[文档已有·kv/11 §3.7]

    2 · 假阳性亲和为何比 RR 更糟（白板可抄）¶

    场景锚（对练设定，非实测）：Qwen3-32B、ISL=8192、共享前缀≈6.5K、单 P ~6.5k tok/s、N=4。[测算·13]

    RR：期望命中 ≈ 1/N → 仍可能偶然打到有前缀的机
假阳性（近似树）：命中视为「有」→ 实际 0；且同前缀 burst 全灌热机 A
    项 RR（N=4） 假阳性打到已驱逐机 标注
    有效前缀命中 期望 ~1/N 0（整段重算） [机制推导]
    Prefill 量级 期望重算约 6567 tok ≈ 1.01s 满量 8192 ≈ 1.26s（多 ~250ms） [测算·13]
    热点排队（burst B=8） 摊到 4 台，均摊排队量级 ~0.5s 全落 A，均摊 ~(B−1)/2×1.26 ≈ ~4.4s（上界） [测算·13]
    主伤害 稀释 热点排队 ≫ 少命中那 250ms [机制推导]

    口述一句：

    「假阳性不是『少赚一点命中』，是『确定性灌爆一台空壳机』——比没有亲和的随机摊开更差。」

    盈亏阈值（选型论证，[机制推导·13]）：中心 /query 常态目标 P50 <5ms、P99 <20ms（0.2s 是超时上界非常态）；本地树一次级联 miss 可到 ~1s。令中心多付 5ms > p×1000ms → 仅当误判率 p<0.5% 省一跳才划算；tools/chat-template 字符错位是常态，故 精度比省一跳值钱。

    3 · BlockRemoved / ZMQ 驱逐感知¶

    环节 做什么 锚点
    引擎 块写入/驱逐 → 发事件 vLLM ZmqEventPublisher 等
    传输 ZMQ PUB/SUB；可选 replay_endpoint 补丢包 kv knowledge/09
    事件 BlockStored / BlockRemoved / AllBlocksCleared 规范 + Conductor
    Conductor 订事件 → PrefixCacheTable；HTTP /query 返最长匹配 Mooncake Conductor
    Motor 无本地 prefix 表；只查 Conductor；Removed 经索引生效 kv_cache_affinity.py 
    真驱逐 ──BlockRemoved──► 索引删 hash ──► /query 的 M↓ ──► 不再打到空壳机
无 Removed（近似树）──► 路由仍高 M ──► 假阳性 + 热点

    两类别混：

    类型 A 假阳 类型 B 空窗
    主因 未订 Removed / 丢事件 / approx 决策领先 Stored
    解药 Removed + Cleared + replay speculative / predicted 短 TTL
    Motor 经 Conductor 有 Removed 路径 无 speculative

    利用率耦合一句：亲和灌热 → 利用率↑ → 驱逐风暴 → 假阳更猛；故 utilization 是稳定性约束，不是装饰项。[文档已有·kv/11]

    4 · Motor vs 本地近似树（对照表）¶

    维度 Motor + Conductor SGLang / vLLM Router cache_aware
    索引 中心真值（token/block 级） 调度侧**本地近似**字符 radix
    驱逐感知 ✅ 订 BlockRemoved（+ replay） ❌ 系统性不知引擎 LRU
    匹配误差 tokenize 一致性风险（部署约定） 字符≠token，tools 易级联 miss
    假命中主风险 事件延迟 / Conductor 残缺 → 假阴为主 假阳系统性 + 字符误差
    负载融合 unified / load_gated + SHM 多看 load() / 树 LRU，不进 utilization
    失败错向 残缺→少认命中→退化 LB（安全） 假阳→灌空壳机（危险）
    运维税 Conductor + ZMQ + replay 无中心组件，但正确性靠时间淘汰

    一句话选型：

    「猜缓存（近似树）省一跳、假阳无界；查缓存（Conductor）付毫秒级查询，错向偏向安全假阴。」

    5 · 快问 8 题（10–20s / 题）¶

    1. 两类假命中？ → A 驱逐滞后假阳；B 空窗（多为假阴 / TTL 可控假阳）。
    2. 假阳为何比 RR 糟？ → 命中 0 + burst 热点；RR 有 1/N 与摊开。
    3. 主伤害数字？ → 少命中 ~250ms 量级；burst=8 热点均摊上界 ~4s 量级——测算。
    4. BlockRemoved 干什么？ → 索引删块，防按陈旧 M 打空壳。
    5. ZMQ 角色？ → 通知面，不是搬 KV 数据面。
    6. Motor 有本地树吗？ → 无；只 HTTP /query。
    7. Conductor 重启窗口？ → 残缺索引 → 假阴安全；靠 replay+新 Stored 收敛。
    8. Motor 还缺啥？ → utilization / speculative 未进分（稳定性闭环）。

    6 · 追问 3 连（严格面试官）¶

    连 1 ·「假阳性一次，TTFT 到底差多少？」
    → 先拆两段：少命中（满算 vs 1/N 期望）约多 ~250ms prefill（ISL=8K、6.5k tok/s 假设）；热点排队**在同前缀 burst 下可到数秒量级上界——后者常是主伤。全部标 **[测算/机制推导]，并声明 4.4s 是「全 miss+全落一台」上界非期望。[13 Round1–2]

    连 2 ·「那你们为什么不直接用 SGLang 那套近似树？」
    → 字符树在 tools/模板注入处易从错位点起整链 hash 分叉 → 级联满 miss；且无 Removed → 假阳无界。盈亏：误判率 p 须 <~0.5% 才抵得过中心 ~5ms；生产 tools 场景做不到。Motor 付 Conductor 税换真值与安全错向。[13 Round1]

    连 3 ·「有了 Removed 是否就不会假命中？」
    → 否。仍有：丢包无 replay、Cleared 未实现（如 AIBrix 空实现）、决策–Stored 空窗、Scheduler 用陈旧 prefill_cost 的短窗高估。Motor 路由错只伤性能（引擎 miss 重算），不伤正确性；要稳还需 utilization 门控 + 可选 speculative TTL。[kv/11 + 13 Round4]

    7 · TTFT 相关数字诚实卡（上场先过脑）¶

    数字 含义 标注

    ~4.4s burst=8 全灌一台的均摊排队上界 [测算·上界]
    ~1.0s 字符错位级联 miss 重算 ~6.5K [机制推导]
    P50<5ms / P99<20ms Conductor /query **目标**延时 [目标·非实测]

    8 · 30 秒自检¶

    1. 假阳 vs RR？→ 更糟（0 命中 + 热点）。
    2. 主伤？→ 排队 常大于少命中。
    3. Motor 感知驱逐？→ Conductor + BlockRemoved。
    4. 70% / 4.4s？→ 测算，非客户 log。
