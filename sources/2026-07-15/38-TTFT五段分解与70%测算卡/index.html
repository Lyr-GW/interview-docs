    38 · TTFT 五段分解与 70% 测算卡（可背）¶

    本夜续批（2026-07-15 · 03:15 双 tick）
    用途：把简历 TTFT−70% / E2E−50% 收成**可背测算卡**——五段白板 + 1187→351 假设清单 +「客户日志」红线话术。
    全程数字标测算；无 A/B 原始包不上场说「客户实测」。

    1 · 60 秒电梯稿（可直接背）¶

    Motor KV 亲和主收益是 少做 Prefill 重算，不是算子 magically 快 70%。[机制]
    TTFT 我按五段拆：tokenize + Conductor + queue + prefill + delivery；亲和直接砍的是 prefill，queue 可能间接降。[文档·05§6]
    简历 −70% 是 代表性测算[测算]：Qwen3-32B、ISL≈8K、共享前缀≈6.5K；\(h\) 从 0.10→0.78 时，\(80+1230\times(1-h)\) 给出约 1187→351ms（−70.5%）[测算]。
    不是客户 raw log 实测[红线]。机制证据用 cached_tokens、Scheduler 选点日志、Conductor /query；正确 A/B 是两组都开 Prefix Cache，只切 load_balance↔kv_cache_affinity。

    金句：少 Prefill，非 FA 加速；测算非日志；主项在五段的 prefill。

    2 · 五段分解白板（必默）¶

    TTFT = T_tokenize + T_conductor + T_queue + T_prefill + T_delivery
         │            │           │          │            │
         │            │           │          │            └─ 首 token 回传 [经验·ms]
         │            │           │          └─ 主项；命中可省 [机制]
         │            │           └─ 峰时可为数百 ms [经验]
         │            └─ ms 级；超时 0.2s 回退 LB [代码]
         └─ 4K≈6ms [经验量级·墙钟]
    段 量级 / 锚点 KVA 怎么动它 标注
    tokenize 4K≈6ms；TokenizerManager 固定税（前置到 Coordinator） [经验量级]
    Conductor ms 级；timeout 0.2s 固定税；P99≪0.2s 才划算 [代码] / 目标 [待补]
    queue 峰时数百 ms 间接（更准路由→少堵） [经验]
    prefill 全量可 秒级 直接砍（少算 suffix） [机制]
    delivery ms 级 几乎不动 [经验]

    压缩公式（亲和收益打在 prefill）[测算 · ir/12 §3.1]：

    TTFT ≈ c₀ + T_prefill_full × (1 − h)
    符号 含义 本卡取值 [测算]
    \(c_0\) 调度/Conductor/P→D/首步开销 80ms
    \(T_{prefill}^{full}\) 全量 Prefill 1230ms（≈6.5k tok/s × 需算量级）
    \(h\) 有效命中率 Baseline 0.10 → Affinity 0.78

    TTFT ≠ 仅 prefill_time：含前端 tokenize/排队；对 metrics 时别混桶。[文档·01D/05]

    3 · 1187→351 类测算：假设清单（被追问时逐条报）¶

    Baseline:  80 + 1230 × 0.90 = 1187 ms   [测算]
Affinity:  80 + 1230 × 0.22 =  351 ms   [测算]
降幅:      (1187 − 351) / 1187 ≈ 70.5%  [测算]

E2E ≈ TTFT + (OSL−1)×TPOT
  Decode 假设：TPOT≈35ms × 15 ≈ 525ms     [测算]
  E2E: 1712 → 876 ms ≈ −48.9%             [测算]

    3.1 场景假设（开口必带）¶

    # 假设项 取值 [测算] 脆度
    1 模型 Qwen3-32B Dense ~32.8B，BF16 中——换模型要重算
    2 并行 / 拓扑 例：P×TP4；PD 或 U 混部叙事可讲，勿冒充实测拓扑 中
    3 请求形态 ISL=8192、OSL≈16；共享前缀≈6.5K 高——前缀不够就不成立
    4 Prefill 速率 单 P 有效 ≈6.5k tok/s → 全量 ~1.23s 中
    5 \(c_0\) 80ms 相对稳；tokenize 有 6ms/4K 经验锚
    6 \(h\) 抬升 0.10 → 0.78（≤6.5K/8K 上限） 最脆
    7 Decode TPOT≈35ms；短输出才让 E2E 接近 −50% 高（长输出稀释）
    8 A/B 公平 两组都开 PC，只切调度策略 方法论红线

    3.2 边界 + 复测（背两句）¶

    重复率 80%+ 才可能逼近 −70%[经验·05§6]；20% 缩水；最坏 ≈ LB + tokenize。TPOT 理论上基本不变[测算]（亲和砍 Prefill）。
    复测要报：TTFT 分位+原始 ms、cached_ratio、回退率、偏斜；禁「关 PC vs 开 PC+亲和」混叠、编客户名/截图数。Conductor P99≪0.2s 目标 [待补]。

    4 · 「客户日志」红线话术（必背）¶

    应答序：降调测算 → 给公式/假设 → 转机制（cached_tokens/选点日志）→ A/B 方法论 → 简历过满则承认改口（不编日志文件名）。

    正确 3 句模板：

    「我没有留存可复核的客户 A/B 原始包，所以**不把 70% 说成客户实测**。[待补/诚实]
    用 Qwen3-32B、8K、约 6.5K 共享前缀做代表性测算：\(h\) 从 0.10 到 0.78 时，TTFT 约 1187→351ms，约 −70.5%。[测算]
    机制上亲和砍的是 Prefill；要复验我会只切调度策略并报分位与 cached_ratio。」

    ❌ 禁止 ✅ 改口
    「客户 raw log 实测 −70%」 「代表性测算；公式可白板」
    「FA/算子优化了 70%」 「调度命中 → 少算 Prefill」
    「任意负载都能 −70%」 「高前缀重复；\(h\) 最脆」
    「关 PC vs 开 PC+亲和」刷% 「两组都开 PC，只切亲和」
    编拓扑/客户名/截图 ms 「机制可证；数字标测算」

    极限诚实一句：若把简历「客户 −70%」理解成有可下载日志——过满；准确=场景级测算 + 机制可证。

    6 · 追问 3 连（严格面试官）¶

    连 1 ·「把 70% 白板推一遍，每步标假设。」
    → 写五段 → 压缩式 → 填 \(c_0=80\)、\(T=1230\)、\(h\) 两档 → 算出 1187/351/70.5% → 口头标注**全部为测算** → 补一句「复测用引擎指标替换 1230/80」。[测算·ir/12]

    连 2 ·「客户日志呢？拿不出来怎么证明你没吹？」
    → 走 §4 模板：降调测算 → 机制证据链 → A/B 方法论 → 不编文件。可指：cached_tokens 透传、Scheduler 亲和分日志、Conductor API。[红线+代码]

    连 3 ·「命中率高但 TTFT 不降，你查什么？」
    → 五段对账：命中在**别的实例**？Conductor 超时回退？queue 把收益吃掉？block 不足仍满量 prefill？先 L1 分位+hit+queue，再下 profiler。[文档·05/30]
