    47 · 语义路由与强弱模型口述卡（可背）¶

    本夜续批（2026-07-15 · 03:55）
    用途：收口复盘失分题 Q10——「vLLM Router 按语义/难度分发强弱模型怎么做？」；深文 interview-review/06。
    先框两层路由再答机制，避免和 Motor KV 亲和（实例级）答串。

    交叉引用¶

    文档 用途
    interview-review/06 实例级 vs 模型级；semantic-router 信号/规则/选型
    interview-review/01 Q10 失分记录
    旁链 12 / Motor 亲和；26；15 ir 实例级深挖

    数字标注：[文档·06]。不编自测成本节省百分比。

    1 · 60 秒电梯稿（可直接背）¶

    先分两层：实例级路由——同一模型多副本，选哪个副本（前缀/KV/负载）；模型级路由——不同能力/成本的模型池，选强还是弱。面试官问的「语义/强弱分发」是第二层；我做的 Motor KV 亲和是第一层。[文档·06]

    vLLM 社区对应 semantic-router：对请求抽信号——领域/意图、难度（complexity） 等。难度用对比式 embedding：预置 hard/easy 两组例句 embedding，算查询与两组最大余弦相似度之差，阈值分成 easy/medium/hard；再加布尔规则（如 math ∧ hard → 强模型+高推理预算，easy → 弱模型+低预算），候选集里再按质量-成本选 endpoint。[文档·06]

    与 KV 亲和正交一句：亲和决定「同模型下进哪台、前缀能否命中」；语义路由决定「该不该升级到更贵模型」——可先选模型，再在该模型池内做亲和。[文档·06]

    Motor 目前只做了同模型多实例的 KV 亲和；强弱分发是值得补的一层，我知道它在栈里的位置。

    金句：实例级选副本；模型级选强弱；难度信号驱动；与 KV 亲和正交可叠。

    2 · 两层路由（先框定再答）¶

    层次 项目直觉 决策依据 解决什么
    实例级 production-stack vllm-router / Motor 前缀哈希、KV 元数据、QPS、session 选哪个**副本**
    模型级 vllm-project/semantic-router 意图/领域/难度/安全等信号 选哪个**模型**（强/弱/预算）

    易混对照（实例级策略，勿答成强弱）：

    策略 一句
    prefixaware 字符/chunk 哈希树，最长前缀选实例
    kvaware 查 LMCache 等元数据，最长前缀 KV；低阈值回退 QPS
    session / RR 粘性 / 轮询

    3 · 难度信号（强弱分发关键）¶

    预置：hard 例句集 embedding / easy 例句集 embedding
请求：q → embedding(q)
δ = max sim(q, hard) − max sim(q, easy)
→ 阈值切 easy / medium / hard（文档量级：数十～百 ms 级分类）

    决策引擎：布尔规则组合信号，例（官方部署口径级，勿背成自家指标）：

    • domain: math AND complexity: hard → 强推理模型 + 高预算
    • complexity: easy/medium 通用问答 → 中/弱模型 + 低预算
    • 安全/jailbreak 信号 → 拦截或走专用策略

    之后在候选模型集里做质量-成本选择（rating / 级联 AutoMix / 延迟感知等）；前沿有 RADAR/IRT 难度建模方向——知道名字即可。[文档·06]

    4 · 与 KV 亲和正交（一句 + 叠法）¶

    正交：KV 亲和优化「同模型副本间的缓存命中与负载」；语义路由优化「跨模型的质量-成本」。
    可叠：先语义选模型（或模型档），再在该档的实例池里做 prefix/KV-aware。

    别说成「Motor 已经做了强弱分发」——没有；别说成「语义路由替代亲和」——层不同。

    5 · 理想收口（可替换 60s 尾句）¶

    「这是模型级语义路由，和我在 Motor 做的实例级 KV 亲和是两层。semantic-router 用难度等信号 + 规则把难题给强模型、简单题给弱模型控成本。我们目前只做了同模型亲和；这层分发我清楚位置，是值得引入的能力。」

    6 · 快问 8 题（10–20s / 题）¶

    1. 两层路由？ → 实例级选副本；模型级选强弱模型。
    2. Motor 属于哪层？ → 实例级（KV 亲和）。
    3. 强弱分发属哪层？ → 模型级（semantic-router）。
    4. 难度信号怎么来？ → 对比 hard/easy 例句 embedding，看相似度差。
    5. 决策怎么合？ → 信号 + 可配置布尔规则 → 模型/预算。
    6. 和 prefixaware 区别？ → prefix 仍是同模型选实例；不是选模型档。
    7. 与 KV 亲和？ → 正交；可先选模型再亲和。
    8. 你们做了没？ → Motor **未做**强弱分发；懂生态位置，不装已交付。

    7 · 追问 3 连（严格面试官）¶

    连 1 ·「难度分类错了会怎样？」
    → easy 判 hard → 成本虚高；hard 判 easy → 质量掉点。工程上靠阈值/人工规则兜底、级联（先小后大）、以及生产反馈校准（文档提 IRT/RADAR 方向）。我不报未验证的准确率数字。[机制·06]

    连 2 ·「和 Dynamo cost / Motor 打分为何不是一回事？」
    → Dynamo/Motor 成本函数多在**同模型池**里权衡命中 vs 负载/迁移；语义路由在**跨模型**维做能力-成本。可同台编排，目标函数不同层。[文档·06 + 旁链竞品]

    连 3 ·「你要是补这一层，最小切入？」
    → 先上「规则 + 轻量难度/意图分类」把流量粗分到 2～3 档模型；强档池内继续跑现有 KV 亲和。别一上来上完整 RL 选型；先可观测（各档命中率/成本/质量）。[系统视角·06]

    8 · 30 秒自检¶

    1. 两层？→ 副本 vs 模型。
    2. 难度？→ hard/easy embedding 差。
    3. ×亲和？→ 正交可叠。
    4. 交付？→ 未做强弱；懂位置。

    白板一页（与 60s 对齐）¶

    请求
  ├─ 模型级（semantic-router）
  │    信号：领域 / 难度 τcpx / 安全…
  │    规则 → 强/弱模型 + 预算
  └─ 实例级（vllm-router / Motor）
       前缀 / KV / QPS / session → 某副本
