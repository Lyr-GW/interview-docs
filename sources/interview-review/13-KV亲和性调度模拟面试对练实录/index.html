    专题 13：KV 亲和性调度 · 模拟面试对练实录（严格面试官 × 面试者）¶

    形式：双 Agent 对练。面试官（严格、细致，专攻「技术选型为什么这样、要数据、要手动分析、为什么用 A 不用 B」）持续追问；面试者（本项目作者视角）基于 MindIE-PyMotor/ 真实代码与专题 12 已核实事实作答，区分「实测 / 代表性测算 / 机制推导」。

    记录原则：不编造实测数据；所有数字标注来源性质。对练直到面试官明确判定「完整」为止，最后附总结复盘。

    关联：专题 12（特性全解与证据边界）、专题 04/10/11（Mooncake 与竞品）、~/wiki/repos/mindie-pymotor/kv-affinity.md（代表性测算来源）。

    对练设定¶

    • 岗位设定：大模型推理服务 / AI Infra 高级工程师。
    • 考察重点：① 关键技术选型的第一性原理；② 用数据和手动量纲分析支撑决策；③ A vs B 的取舍边界；④ 对「实测 vs 测算」的诚实度。
    • 代表模型场景（用于量化推演，非实测）：Qwen3-32B Dense（32.8B、64 层、GQA 64Q/8KV、head_dim=128），PD 分离 4P4D、单实例 TP4，ISL≈8K、共享前缀≈6.5K、OSL≈16。

    Round 1 · KV 前缀索引选型：中心化 Conductor vs 本地近似 radix tree¶

    面试官：

    你选了中心化真实索引（Conductor 订阅 kv-events），放弃了 SGLang router 那套调度层本地近似 radix tree。你在设计文档里写"精度比省一跳 HTTP 值钱"——请你先把这句话量化成可比较的数：在 Qwen3-32B、ISL=8192、block_size=16 的场景下，本地字符级近似树一次错判前缀边界的期望代价是多少 token 的重复 prefill、折算多少毫秒？而中心索引每请求多付的那一跳 HTTP（0.2s 超时）P50/P99 各是多少毫秒？两个量级摆出来，你凭什么说"值"？ - 追问一：本地近似树"无驱逐感知会假阳性"到底怎么伤害你？一次假阳性把请求路由到一个前缀已被 LRU 驱逐的实例，和 round-robin 随机落点相比，TTFT 到底差多少、差在哪一步——是不是反而比没有亲和更糟？ - 追问二：Conductor 重启丢状态那段窗口，kv-events replay + /services 对账要多久才能把全局前缀表重建到"可信"？这段窗口里 /query 返回的是真实还是残缺索引、你怎么知道该不该信它？本地树没有这个"半可信"窗口，你怎么反驳？

    面试者：

    先说明量纲基线：Qwen3-32B、TP4、ISL=8192、共享前缀≈6.5K，block_size 用**引擎默认 16**（文档提到的 128 只是把每块调粗、抬高 fast-path 阈值，不改结论）。以下标注｜实测 / 代表性测算 / 机制推导｜。

    ① 一跳 HTTP（中心索引，每请求必付）｜机制推导+目标值｜：/query 是小 POST，body ≈8192 个 int（JSON ~40KB），Conductor 把 token_ids 按块切成 8192/16=512 个链式哈希**沿表查，O(512)。目标 P50 <5ms、P99 <20ms；0.2s 是超时**硬上界（超了回退 LoadBalance），不是常态延时。这块无实测毫秒值，是缺口。

    ② 一次错判前缀边界（本地字符树）｜机制推导｜：关键不是"错一块=16 token"。block hash 是 hash(parent, token_ids) 链式的，字符级在 tools/chat-template 注入处一旦对不齐 token 边界，从错位点起后续每块哈希全变→整段 6.5K 前缀 miss。按单 P ~6.5k token/s：6500/6500≈1.0s 重复 prefill；就算只错尾块=16 token 也要 ~2.5ms，但字符树误差恰恰高发在前缀前部，是**级联**而非优雅退化。

    ③ 凭什么"值"：设误判率 p。中心固定多付 ~5ms/请求，本地树期望多付 p×1000ms。令 5ms > p×1000ms → 仅当 p<0.5% 省一跳才划算。字符级在 tools 场景对不齐是常态，p 远超 0.5%，故精度值钱——量级差是"确定 5ms vs 期望几十～上千 ms"。

    追问一｜假阳性 vs round-robin｜机制推导｜：本地树无驱逐感知，会**确定性**把请求送到"它以为有、实际已被 LRU 驱逐"的实例 A。RR 随机落点命中前缀概率≈1/N（N=4 时期望重算约 6567 token≈1.01s）；假阳性命中=0（重算满 8192≈1.26s），且把同前缀 burst 全灌向 A→A 排队拉长。两处都更糟：prefill 多算 ~250ms + 制造热点——确实比没有亲和还差。Motor 靠 BlockRemoved 事件实时把驱逐块从持有者集合摘掉，从根上不产生假阳性。

    追问二｜Conductor 重启窗口：恢复两条——(a) 拓扑对账 re_register_kv_instances 拉 /services 比对补注册，秒级（2s 超时/次）；(b) 内容恢复靠 kv-events replay（PUB 端 ROUTER socket 缓存最近 buffer_steps 批，按序列号增量补拉），若停机超过缓冲深度，老块只能靠新 BlockStored 逐步重建。诚实坦白：这段窗口 /query 返回**残缺索引**，且当前响应没有覆盖度/序列号水位，我无法从 payload 判断该不该信——真实缺口。但错向是**安全**的：残缺→假阴性（少算命中）→退化 LoadBalance，不会像本地树假阳性打到驱逐实例。反驳"本地树没有半可信窗口"：本地树重启后是空树、且**永远不知道真实驱逐**，等于长期处于可能假阳性的不可信态，误差**无界且不收敛**；中心索引窗口**有界、可自愈**（replay+新事件收敛真值），窗口内错向还安全。

    补测方案：给 /query 响应加 per-endpoint last_applied_seq+replay 完成标志，量出"重启→追平"真实耗时；窗口内对 matched 打折或强制偏 LoadBalance；用 cached_tokens 反算真实命中率校验。

    面试官点评（4/5）： 亮点——把"盈亏平衡 p<0.5%"当阈值论证、区分假阳性(命中0+热点)与假阴性(退化 LB)的方向安全性、主动补 last_applied_seq。不满意——P50<5ms 全是目标非实测，"坦白缺口"不能替代数（至少能按 512 次哈希+一次 RTT 手推量级）；"比 round-robin 更糟"只算了命中差(0 vs 1/N)，没量化热点排队（burst 全灌已驱逐实例，队列深度×TPOT 的尾延迟才是真伤害）；级联 miss 说"6.5K 全 miss"偏理想化，字符树未必错在最前端。

    Round 2 · tokenize 前置的选型：进程内嵌 vs 引擎 render vs 独立 tokenizer 服务¶

    面试官：

    主问题：你把 tokenize 放进 Coordinator 进程内（HF apply_chat_template），而不是复用引擎的 render、也不是起独立 tokenizer 服务。为什么？请按"一致性、延迟、可用性、扩展性"四维给出你选内嵌的第一性理由，并说清你放弃独立 tokenizer 服务（llm-d 旧 gRPC-UDS 路线）具体丢了什么、又躲了什么。 - 追问一：你说 token 序列与引擎"逐字节一致"。可你是靠"同 model_path + 同默认模板"的**部署约定**，没有 revision/template-hash 握手。给我一个会让这个约定悄悄失效、导致亲和全 0 的具体场景，并量化：一致性一旦破，TTFT 从测算的 0.35s 退回到多少？ - 追问二：给成本数：4K/8K token 一次 apply_chat_template 在你的 CPU 上大概几毫秒？QPS 到 500 时要吃多少核、会不会成为 Coordinator 瓶颈？对比"引擎二次 tokenize"你到底净省了什么——还是只是把这份 CPU 从 GPU 机搬到了调度机？

    面试者：

    补：假阳性热点排队量级｜机制推导｜：设同前缀 burst B=8。假阳性全落 A，单 P 串行 prefill≈1.26s，第 k 个等 (k−1)×1.26s，均摊排队≈(B−1)/2×1.26≈4.4s；RR 摊到 N=4（每台 2 个）均摊≈(2−1)/2×1.01≈0.5s。热点排队差≈4s，比"少命中"那 250ms 大一个量级——这才是假阳性真正贵的地方。

    主问题｜四维第一性理由：

    • 一致性：进程内 HF apply_chat_template(+tools) 对文本 Chat 能与引擎逐字节一致；引擎 render 是**同代码路径**（最强，但当时 vLLM 未稳定暴露 /render）；独立 tokenizer 服务要**自己复刻模板**，llm-d 正因盖不住 Harmony/tools/MM 才 deprecate。进程内 HF 是"比字符猜准、比自建服务省心"的务实中档。
    • 延迟：进程内=零网络跳、零序列化；render 多一跳 HTTP、独立服务多一跳 gRPC/UDS+序列化。热路径预算毫秒级，进程内最省。
    • 可用性：进程内不引入新故障域，tokenize 失败走 _safe_fallback_encode→[]→fail-closed→LoadBalance；独立服务=新 SPOF，render 把路由可用性绑到引擎端点。
    • 扩展性：进程内**最弱**——tokenize CPU 焊在 Coordinator，不能独立扩，高 QPS 成瓶颈。这是明确认的取舍（路线图抽 TokenProducer 接口）。

    放弃独立 tokenizer 服务：丢了独立横向扩容与干净解耦；躲了新增组件、SPOF、IPC 延时、以及"独立实现追不上引擎语义"的漂移（llm-d 已证此路难走）。

    追问一｜约定失效场景 + TTFT 退回｜机制推导+测算｜：运维在**引擎侧**改了 --chat-template（或加 reasoning 模板 / tokenizer 升 revision / 热更权重换了 tokenizer_config.json），但 Coordinator 缓存的 tokenizer 未重载。调度端渲染 T_old、引擎按 T_new 哈希，哪怕差一个 token，block hash 链从第 0 块就分叉→/query 全 endpoint matched=0→亲和**静默退化为 LoadBalance**。TTFT：全 miss≡Baseline，测算 0.35s→≈1.19s（+840ms，约 3.4×）。最阴险的是不报错（tokenize 成功只是"对错了"，fail-closed 不触发），唯一信号是 cached_tokens 掉到 ~0——这正说明"无 revision/template-hash 握手"是真实缺口。

    追问二｜成本数｜tokenizer 实测（非端到端 TTFT 实测）+外推｜：文档给 4K≈6ms，HF fast tokenizer 是 Rust 后端、encode 时**释放 GIL**可真并行。线性外推 8K≈12ms。QPS=500、8K：总量=500×12ms=6000ms/s=6 核**满载（4K 则 3 核）。数十核调度节点扛得住，但热路径同步执行、尾延迟敏感，QPS×ISL 再涨即瓶颈。**对比引擎二次 tokenize：引擎无论如何要 tokenize 一次，Coordinator 这次是**额外第二次**，当前**并非净省 CPU**，而是"搬到调度机 + 多做一次"；真正收益是**保护 GPU 机**（超长/非法请求入口按真实 token 拒掉，PR #349）+ 路由精度。要变净省需 token-in/token-out 让引擎吃 token_ids——Motor 还没这条链路，不粉饰。

    面试官点评（4/5）： 亮点——热点排队终于量化（burst=8→均摊≈4.4s）、一致性破裂给了 +840ms 具体退化、扩展性主动认最弱并给路线图、坦白无握手/非净省 CPU。可压——4.4s 是"全 miss+全落一台"双重最坏（上界非期望，未考虑假阳性也可能命中部分前缀）；6ms 没说什么 CPU/单核还是多核、哪个 tokenizer（数字悬空）；"保护 GPU 机"略夸大，省的到底是排队还是算力要说清。

    Round 3 · 双算法选型：unified（软融合）vs load_gated（硬门控）¶

    面试官：

    主问题：为什么必须两种模式，而不是让 unified 靠调 load_weight 覆盖 load_gated 的行为？请从第一性上说清：加权软融合能不能表达"硬负载上界"这个语义？若能，何必单开一个模式；若不能，差在哪。 - 追问一：unified 公式 prefill_load_scale×max(0,isl−overlap_credit×matched)+load_weight×load 两项相加——为什么量纲可比？overlap_credit=1.0、load_weight=1.0 默认值凭什么定，物理含义是什么？给一个具体数值例子：两个 endpoint，一个高命中但已排队、一个零命中但空闲，代入公式算出 unified 把请求让给空闲实例的临界负载差是多少 token。 - 追问二：load_gated 的 topn 默认 2——为什么是 2 不是 3 或 5？topn 取多大退化成 unified、取多小退化成纯负载均衡？为什么 load_gated **刻意不**升级到 PR#304 的全局重排——若升级了，它的硬约束语义会被破坏成什么样？

    面试者：

    核对代码，默认值与文档一致：kv_affinity_load_weight=1.0、overlap_credit=1.0、prefill_load_scale=1.0（coordinator.py）。一处点明：kv_affinity_load_gate_topn 代码**字面默认 0**（哨兵），运行时 0→2（_DEFAULT_LOAD_GATE_TOPN）——语义默认 2，与文档一致但字面不是 2。

    （补上轮遗留：4.4s 是"全 burst 串行落 A"的**上界非期望**；6ms 是**单核 Rust encode 的单请求墙钟**；"保护 GPU 机"省的是**排队**不是算力——那次 prefill 算力该花还得花。）

    主问题｜为何必须两种模式｜推导｜：unified 是加权和 scale*prefill_cost + weight*load，单调可交易——命中足够大时任何**有限** load_weight 都能被亲和翻盘。load_gated 要的是**字典序**（先负载、再亲和），负载是硬天花板。字典序无法用任何有限权重的加权和表达：要"负载绝对优先"得 load_weight→∞，那亲和被彻底压死、退化纯 LB。所以"软融合覆盖硬上界"数学上做不到，必须两种模式。

    追问一｜量纲与临界值｜代码确认｜：两项都在 token 量纲。prefill 项 isl − overlap_credit×matched（token）；load 走 calculate_workload_score("prefill")=active_tokens+0.3×active_kv_cache，ROLE_P 分配时用 len(token_ids) 真实 token 数 → load 也是 token。overlap_credit=1.0：命中 1 token 恰省 1 token prefill（物理精确）；load_weight=1.0：1 token 排队负载 = 1 token 待算 prefill，是**中性基线**。

    数值例（isl=8192、三系数=1）： - epA matched=6000 但已排队：分 =(8192−6000)+load_A = 2192+load_A - epB 零命中但空闲：分 =8192+load_B - epB 胜：8192+load_B < 2192+load_A → load_A−load_B > 6000 token

    即"空闲实例要赢，负载差得超过命中收益 6000 token"。一个在飞的 8K prefill 给 epA 加约 1.3×8192≈10.6K 负载，已远超 6000——仅一个大请求排在前面，亲和就该让位。方向已核对 _select_with_load（取最小分）。

    追问二｜topn=2 与不升级全局重排｜推导+代码｜： - topn=全部：不筛，stage2 按最长前缀排 → "最闲集合内纯亲和"（非 unified，因 stage2 是最长前缀非融合分）； - topn=1：只剩最闲那台，亲和无从作用 → 纯 LB； - 2 是"既保硬负载上界、又给亲和一个平局空间"的最小集，3/5 只是放宽上界更偏亲和。 - 为何不升级全局重排：load_gated 的存在意义就是硬上界。Scheduler 全局重排用**软融合分**，一旦对 load_gated 候选重排=把硬约束松绑成软权衡，模式**塌回 unified**。代码坐实：_select_load_gated 调 _stash_affinity_debug(raw) 不带 with_prefill（默认 False），prefill_cost 存 None → Scheduler 结构上拿不到重算所需量，**无法**对它全局重排；unified 路径才 with_prefill=True。有意的不对称。

    面试官点评（5/5）： 几乎无懈可击——字典序不可被有限权重加权和表达是本质回答；数值例把临界 6000 token 和在飞 8K 折算 10.6K 负载算通；with_prefill=None 的"有意不对称"是读进源码才答得出的。可挑两处（不扣分）：load=active_tokens+0.3×active_kv_cache 里 0.3 的来历没讲；临界 6000 其实 =overlap_credit×matched，随 overlap_credit 线性缩放，没点明。

    Round 4 · 多进程防 herding：in-flight overlay → top-k 候选 → 全局重排¶

    面试官：

    主问题：V1 的 in-flight overlay 为什么是**方向错误**、必须整体移除，而不是把 TTL 调好？请从"作用域"第一性上论证：本地 overlay 再怎么调 TTL，也解决不了什么？ - 追问一：V2 已经是"top-k 候选 + Scheduler 权威重选"，为什么还不够、非要 V3 全量上报每 endpoint 的 prefill_cost 让 Scheduler 全局重排？给一个 k=3 会漏掉全局最优的具体并发场景。 - 追问二：V3 的分工前提是"prefill_cost 时不变、只有 load 易腐"。这个前提真成立吗？给一个并发时序：worker 在 T0 查 Conductor 得到 matched，Scheduler 在 T1 决策——这中间别的请求可能已改写该 endpoint 的缓存（存入/驱逐），使 T1 真实 matched≠T0。此时 Scheduler 拿 T0 的 prefill_cost 重排，会错到什么程度？你怎么兜？

    面试者：

    核对代码：worker 提案 top-k=3（_AFFINITY_CANDIDATE_TOPK=3），unified 全量上报 prefill_cost+两标量，Scheduler combined = pscale*prefill_cost + lweight*fresh_load 全局取 min、平局偏低 prefill_cost（scheduler_server.py）；load_gated 因 prefill_cost=None 结构上进不了全局重排（scheduler_client.py）。

    （补上轮：① active_tokens+0.3×active_kv_cache 的 **0.3 是启发式权重**非第一性——active_tokens 抓 prefill 计算队列压力，KV 占用是次要显存压力项折 0.3，是调出来的常数；② 临界 6000=overlap_credit×matched，overlap_credit=0.5 时门槛降到 3000，调它就是调"亲和敢认多少命中收益"。）

    主问题｜V1 为何方向错｜第一性·作用域｜：herding 是**跨 worker** 现象——N 个 worker 各自独立算出同一全局最小 endpoint 全灌过去。overlay 只改**本 worker** 视图，其余 N−1 个 worker 仍堆同一台，跨 worker 碰撞没解。TTL 两难印证方向错：太短→请求没入真实账本 overlay 就清，等于没有；太长→该 endpoint 被**长期虚高**，命中好的实例反而长期接不到请求；且窗口内与真实账本**双写**、同一负载算两次。正解是把仲裁收敛到唯一权威点（Scheduler），故 #210 整体移除。

    追问一｜V2 不够、V3 全量｜代码+场景｜：V2 报 best-first top-3，Scheduler 在这 3 个里重选——权威跨进程，但 k=3 是截断。场景：20 个同前缀 burst，所有 worker 算出**同一 top-3**，Scheduler 把 burst 摊到这 3 台却量大全打热；命中略低（5800 vs 6000）但**完全空闲**的第 4 名才是全局最优，却不在候选集→选不到。k 调大→消息体+扫描成本涨；k=全部即 V3。V3 利用 unified 可分解：prefill_cost（时不变）+2 标量全量上报，Scheduler 用新鲜 load 对**每个** endpoint 重算取全局 min，无截断，第 4 名场景自然覆盖，且不需 prompt/tokenizer/Conductor，O(endpoints) 线性扫描。

    追问二｜prefill_cost 真时不变吗｜诚实推导｜：不严格成立。T0→T1 间别的请求可能存入/驱逐，真实 matched(T1)≠T0： - 存入（burst 同前缀常见）：matched 只增→T0 prefill_cost 被高估→少认命中、少赚不亏，路由仍对； - 驱逐：matched 高估→送到"以为有、实已驱逐"的实例→引擎本地 miss 正常重算——路由错只伤性能不伤正确性。

    错多少（测算）：最坏整段 6.5K 前缀在窗内被驱逐→prefill_cost 低估约 6000 token→该请求 TTFT 多约 6000/6500≈0.92s；但仅此一条，下个 kv-event 更新后**自愈**，且窗口被 0.2s 超时+亚毫秒决策夹住，全驱逐落窗内概率低。兜底（代码）：① 0.2s 超时缩窗；② matched=min(matched,isl) 钳制防过认；③ cached_tokens 事后校验系统性漂移；④ 正确性从不依赖它——miss 必重算。Scheduler 端**故意不重查 Conductor**，用"有界陈旧 prefill_cost 换廉价权威 load"是刻意取舍。

    面试官点评（5/5）： "路由错只伤性能不伤正确性"（引擎 miss 重算兜底）抓到了本质——亲和是优化不是正确性依赖；驱逐窗 +0.92s 且自愈也量化到位。可挑（不扣分）：20-burst 打热第 4 名的场景没算"20 个摊到 top-3 各接约 6~7 个、队列多深才叫热、第 4 名此时省多少"；"自愈靠下个 event"没量化 kv-events 批量发布间隔（几十 ms 还是几百 ms）。

    Round 5 · 性能收益 70% 的诚实归因与 A/B 方法论（关键轮）¶

    面试官：

    主问题："TTFT 降约 70%"（1187ms→351ms）是**实测还是测算**？把它拆开，逐个为假设找依据：单 P 有效 6.5k token/s、c0=80ms、命中率 h 从 0.10 升到 0.78，这三个数哪个最站不住脚、你凭什么填？ - 追问一：baseline 既然引擎自己开了 prefix-caching，为什么 h 只有 0.10 而不是更高？0.10 是 1/N 稀释推算还是拍的？换成 8 实例、重复率 90% 时该是多少？ - 追问二：给一个能隔离"亲和路由"净收益的 A/B：两组 prefix-cache/模型/硬件/回放序列怎么设，为什么**不能**用"关 cache vs 开 cache+亲和"对照？冷启动几轮怎么分桶？报 P50 还是 P99、为什么？最后——明确说这套数里哪些你**从没实测过**？

    面试者：

    先给结论，不绕：1187→351、约降 70.5% 是代表性测算，不是实测。 早前面试里确实说过"某客户约 4K、TTFT 降约 70%"，但现在翻仓库找不到那次实测的拓扑、请求构造、基线毫秒和调度日志，无法自证；所以现在只当**历史口述 + 理论测算**，不作为客户实测结论。代码和可观测链路能证明**机制成立**（cached_tokens 透传、Scheduler 权威日志），但机制成立 ≠ 我量到了 70%。

    主问题｜三个假设谁最脆： - h 从 0.10→0.78：最站不住脚。 这是**假设的路由改善**，0.78 只是"6.5K/8192≈0.79 可复用上限"打折，0.10 是稀释推算。真实值须用 cached_tokens 反算，我没有这组数。70% 几乎全押在这一项，它一动结论就动。 - 单 P 6.5k token/s：量级粗估。 从 8B 级 NPU 外推到 32B，32B 更慢、6.5k 很可能高估，高估会**放大**收益，应以引擎真实吞吐替换。 - c0=80ms： 调度+Conductor 查询+P→D KV 传输+首步 decode 的合计估值，是加性常数，对**百分比降幅影响小**，三者里最稳。

    一句话：最脆是 h，其次 6.5k，c0 最稳。

    追问一｜baseline 开了 PC 为何 h 只 0.10：0.10 不是实测，是 1/N 稀释推算。h_baseline ≈ h_reuse × P_route × P_pool——round-robin 下 P_route≈1/N。N=4→约 0.10 量级；8 实例→P_route≈1/8→baseline 更低（≈0.05 量级）；亲和把 P_route 拉到≈1，故实例越多亲和相对收益越大；重复率 90% 只抬高 h_reuse 上限。0.10 与 0.78 都是建模参数，非测量值。

    追问二｜隔离净收益的 A/B： - 两组都开 prefix-caching：Baseline=load_balance+PC on，Experiment=kv_cache_affinity+PC on；同模型/硬件/拓扑、同一批请求回放、同到达率。 - 为什么不能"关 cache vs 开 cache+亲和"：那测的是"PC + 路由"叠加，把引擎前缀缓存的功劳算给路由，严重高估。变量必须只剩调度策略。 - 冷启动：前几轮建缓存**单独分桶剔除**，只在缓存稳定后衡量。 - 报 P50 和 P95/P99：亲和降 prefill 拉低 P50 与尾部；但若造热点 P99 可能**反被抬高**，只报均值会掩盖副作用。另需报 cached_ratio（分冷/热）、负载倾斜（P99/P50）、回退率、Conductor 查询延迟。

    从没实测过的（不含糊）： 实测模型、NPU 型号、P/D/U 数与 TP/DP、QPS/并发、请求总数、输出长度与预热轮次、共享前缀精确 token 数、TTFT P50/P95/P99、KVA 开/关**原始毫秒**、PD 分离与混部两种形态各自收益；以及 6.5k token/s、c0=80ms、h=0.10/0.78 这些参数本身——全是测算/推导，无一实测。 真要复测，按上面 A/B 出原始毫秒和分位数再谈数字。

    面试官点评（5/5）： 把"1187→351 是测算、客户 70% 只是口述且仓库无佐证"讲死，三假设按脆弱度排序（h > 6.5k > c0）并说清 c0 是加性常数、8B 外推 32B 可能高估，还给了 h_baseline≈h_reuse×P_route×P_pool 稀释模型，最后逐条列"从没实测"清单——实测/测算的硬区分正是这类岗位最稀缺的加分项。

    Round 6 · 收尾：DP rank 粒度 + Mooncake 边界 + 回退链¶

    面试官：

    主问题：亲和粒度为什么必须到 DP rank 而非实例级？vLLM DP 下每 rank 独立 KV 池——若只做实例级，一次"实例命中但落错 rank"的请求会退化成什么？给个数值：DP=4、matched=6000 时，实例级路由的期望有效命中是多少（提示：1/4 命中满、3/4 全 miss）？ - 追问一：Mooncake 你到底用了哪几块、没用哪几块？为什么用 Conductor 做索引、却自研调度打分与 Scheduler 仲裁，而不直接上 Mooncake 全套？边界在哪、自研那部分 Mooncake 为什么给不了你？ - 追问二：回退链 kv_cache_affinity→load_balance→round_robin——为什么 load_balance 是全局最小负载而不是轮询？两者在 8 endpoint、负载不均时尾延迟差多少？再说 0.2s 查询超时：这个数怎么定的，设成 50ms 或 1s 各会付什么代价、和你测算里省下的 ~836ms prefill 怎么权衡？

    面试者：

    核对代码：LoadBalance 用 heapq.nsmallest（load_balance.py）取全局负载最小、复合分 endpoint_score + 0.05×(instance_score/endpoint_count)（DEFAULT_ENDPOINT_INSTANCE_SCORE_WEIGHT=0.05）、start_index 只影响平局顺序；Conductor 注册端口=基址+endpoint.id，/query 返回 {instance:{"DP":{rank:matched}}}，打分侧 dp_map.get(f"{ep.id}") 逐 rank 取（kv_cache_affinity.py）。

    主问题｜为何必须到 DP rank｜推导｜：vLLM DP 下每 rank 独立 KV 池，前缀在 rank A 不等于 rank B 有。实例级路由只知"实例命中 6000"却不知在哪个 rank，落点等于实例内**随机选 rank**。数值（DP=4、matched=6000、isl=8192）：实例级期望有效命中 = 1/4×6000 + 3/4×0 = 1500 token——把 6000 稀释到 1500，有效命中率从 6000/8192≈0.73 掉到 0.18；错落的 3/4 请求要在目标 rank 重算整段 prefill。所以必须 per-rank——粒度错一层，收益直接打到 1/DP。

    追问一｜Mooncake 边界｜代码+推导｜：用了两块——① Conductor（全局前缀索引，订阅 kv-events 维护 PrefixCacheTable，HTTP register/query）；② Transfer Engine/Store 系（昇腾 ascend_transport+AscendStoreConnector 做 KV 传输与池化）。自研**整个决策层：unified/load_gated 打分、worker 提案+Scheduler 权威仲裁、DP rank 路由、fail-closed 回退。不上全套的原因：**Mooncake 给的是"缓存在哪"的事实层，给不了"结合本集群实时负载+多 worker 一致性该去哪"的决策层——Conductor 不知道 Motor 的 endpoint 负载账本、instance 版本、PD 角色，这些和本地 SHM 负载/多进程架构强耦合。边界：事实（Mooncake）vs 决策（Motor）。

    追问二｜回退与超时｜代码+推导｜：LoadBalance 不是轮询，是全局最小负载（nsmallest+复合分+旋转防冷启动 herding）。量级差：8 台里 1 台堆 10 个长请求、其余空闲——轮询照样轮到它，P99≈10×单请求；全局最小负载绕开它，P99≈1–2×，尾延迟差数倍。0.2s 超时定法：约束是"远小于省下的 prefill 收益（测算 ~836ms），又给正常查询（目标 P99<20ms）留冗余"——0.2s≈正常的 10 倍冗余、仅占潜在收益 ~24%。两端代价：50ms→正常够用但抖动易误触发回退、白丢命中；1s→Conductor 真挂时每请求白等 1s > 省下的 836ms，亲和反成**净负**。故卡在亚秒级、远小于 1s。

    诚实边界：本轮 1500/0.18、10×vs1-2×、836ms/0.2s 权衡属**测算/推导**；per-rank 结构、nsmallest/复合分/旋转、超时 0.2s、Mooncake 用哪几块均为**代码/配置事实**。全链回退：kv_cache_affinity（无索引/超时/tokenize 失败）→load_balance（异常）→round_robin——亲和自始至终是优化不是依赖。

    面试官点评（5/5）： 实例级期望命中 1500/6000（有效命中率 0.73→0.18、收益打到 1/DP）算得干净；Mooncake"事实层 vs 决策层"边界清晰；0.2s 超时权衡有数。

    完整性判定：考察完整，可以结束。 六轮已覆盖索引选型、tokenize 前置、双算法、防 herding 三版、性能归因与 A/B 方法论、DP 粒度/Mooncake 边界/回退链，从技术选型第一性、量纲/数值推导到证据边界均被逼到底且守得住。

    总结复盘（我方视角）¶

    一、关键技术选型 A vs B 一览（面试主线）¶

    决策点 选了 A 放弃 B 第一性理由 关键量化
    前缀索引 中心化 Conductor（订阅 kv-events 的真实索引） 调度层本地近似 radix tree（SGLang router） 目标场景前缀长，错一处级联 miss 整段前缀；中心索引含驱逐感知 盈亏平衡 p<0.5% 才值得省一跳；错判级联≈1.0s 重算 vs 一跳目标 <5ms
    tokenize 位置 Coordinator 进程内嵌（HF apply_chat_template） 独立 tokenizer 服务 / 依赖引擎 render 零网络跳、不引新故障域、fail-closed 回退；避免独立实现语义漂移 8K≈12ms/核，QPS500≈6 核；一致性破→亲和全 0→TTFT 退回 ≈1.19s
    打分算法 unified（软融合）+ load_gated（硬门控）双模式 单一模式靠 load_weight 覆盖 字典序（硬负载上界）无法用有限权重加权和表达 空闲实例翻盘临界：load_A−load_B > overlap_credit×matched（例 6000 token）
    防 herding worker 提案 + Scheduler 全量重排（V3） 本地 in-flight overlay（V1）/ 固定 top-k（V2） herding 是跨 worker 现象，须收敛到唯一权威点；top-k 会漏全局最优 20 burst 打热 top-3、第 4 名（空闲）选不到；V3 O(endpoints) 线性扫描
    亲和粒度 DP rank 级 实例级 DP 下每 rank 独立 KV 池 实例级期望命中 6000→1500 token，有效命中率 0.73→0.18
    Mooncake 用量 只用 Conductor（索引）+ Transfer/Store（传输池化） 上 Mooncake 全套决策 事实层（缓存在哪）vs 决策层（结合本地负载/一致性该去哪）强绑本地状态 —
    回退兜底 全局最小负载 LoadBalance → round_robin 纯轮询 轮询无视负载，长请求堆单台 8 台 1 台堆 10 请求：轮询 P99≈10×，最小负载≈1–2×

    二、代表性测算口径（务必标注：非实测）¶

    Qwen3-32B、TP4、ISL≈8192、共享前缀≈6.5K、TTFT=c0+T_prefill·(1−h)：

    • Baseline：80 + 1230×0.90 = 1187ms；Affinity：80 + 1230×0.22 = 351ms → TTFT −70.5%
    • E2E（+525ms decode）：1712ms → 876ms → −48.9%
    • 假设脆弱度排序：h(0.10→0.78) 最脆 > 6.5k token/s（8B 外推 32B，可能高估）> c0=80ms（加性常数，最稳）

    三、暴露的真实缺口（面试守住但要补）¶

    1. 核心收益无一手实测：模型/硬件/QPS/请求数/前缀 token/分位数/原始毫秒/两形态收益全缺；6.5k、80ms、h 均为测算。
    2. 无 tokenizer revision/template-hash 握手：引擎侧改模板/升级会静默使亲和全 0，仅靠 cached_tokens 掉 0 才发现。
    3. Conductor 重启窗口：/query 返回残缺索引，payload 无覆盖度/序列号水位，无法自判可信度（补 last_applied_seq）。
    4. 部分启发式权重无第一性依据：负载分里 0.3×active_kv_cache 的 0.3 是调参常数。
    5. tokenize CPU 焊在 Coordinator：高 QPS 瓶颈，路线图需抽 TokenProducer 接口。

    四、复测行动项（把测算升级为可验证结论）¶

    1. A/B：两组**都开 prefix-caching**，只切 load_balance ↔ kv_cache_affinity；同模型/硬件/拓扑、同一批请求回放、同到达率。禁用"关 cache vs 开 cache+亲和"对照。
    2. 冷启动前几轮单独分桶剔除；报 P50 + P95/P99（亲和降 P50/尾部，但热点可能抬 P99）。
    3. 观测：cached_ratio（分冷/热）、endpoint 负载倾斜（P99/P50）、回退率、Conductor 查询延迟（目标 P50<5ms、P99<20ms）。
    4. PD 分离与混部**分别**出数，不合并声称同一降幅。

    五、一句话结论¶

    机制侧（选型第一性、分布式一致性、DP 粒度、回退安全）已能扛住严格追问且大量以源码坐实；唯一硬门槛是补一组端到端 A/B 实测，让"70%"从代表性测算变成可复核的实测。

    参与代理：严格面试官 × 面试者。
