    专题 16：结构化输出独立复习专题¶

    对应简历条目："在 MindIE 从 0 到 1 交付结构化输出（xgrammar 约束解码）"（cvs/林炜-推理框架方向.pdf）。 源材料：/Users/lvv/wiki/repos/mindie-pyserver/function-call.md + 本工作区 MindIE-LLM/、vllm/ 源码核实。 定位：本文是结构化输出的**复习视角**——功能点、技术点、通用场景与面试考点，方便快速过一遍。xgrammar 的**原理深潜**（Schema→PDA→bitmask、vLLM 架构深挖、开销数值、副作用完整版）已在 03-结构化输出与约束解码专题.md 系统覆盖，本文只做概览并在需要处交叉引用 03，不重复造轮子。与 Function Call 的交叉关系见 17-FunctionCall与结构化输出综合专题.md。

    1. 功能点：结构化输出是什么¶

    结构化输出（structured output / guided decoding）：让 LLM 的输出**保证**符合某种形式规范——JSON Schema、正则、EBNF 语法、工具调用格式。仅靠 prompt 无法保证 100% 合法，约束解码在**采样阶段**硬性屏蔽非法 token：每步解码前算出"当前状态下合法的 token 集合"，把非法 token 的 logit 置 −inf。

    核心价值：把"模型大概率会输出合法 JSON"升级为"模型必然输出合法 JSON"——这是任何需要机器消费模型输出的下游系统（Agent、数据抽取、API 编排）的刚需。

    它支持的约束类型（面试可作为"通用场景"列举，不止 tool call）：

    约束类型 典型入参 场景举例
    JSON Schema response_format / guided_json 数据抽取、API 响应固定结构
    正则（regex） guided_regex 电话号、日期、枚举串等短格式
    EBNF 语法 guided_grammar 领域 DSL、SQL 子集、自定义语法
    工具调用格式 structural_tag / tools Function Call（结构化输出的特化子集，见综合专题）

    2. 技术点：xgrammar 约束解码机制概览（简要，细节引用 03）¶

    xgrammar（CMU Catalyst / MLC，MLSys 2025）的处理链路一句话概括：

    JSON Schema ──转换──> EBNF 上下文无关文法（CFG）
    ──编译──> 字节级下推自动机（byte-level PDA）
    ──预计算──> adaptive token mask cache
运行时：PDA 栈状态 ──> token bitmask ──> apply 到 logits ──> 采样

    复习只需记住三个"为什么"，深潜见 03-结构化输出与约束解码专题.md 第 2 节：

    • 为什么是 PDA 不是 FSM：JSON 是递归结构（对象套对象、数组套数组），嵌套深度无界，FSM 表达不了，需要带栈的下推自动机（简单正则约束可退化为 FSM）。
    • 为什么快（核心优化一）：token 二分类——>99% 的 context-independent token 仅凭 PDA 栈顶位置即可判合法，编译期预计算进 mask cache；<1% 的 context-dependent token 运行时用持久化执行栈现场检查。mask 生成从"全词表模拟"降到微秒级。
    • 为什么不占关键路径（核心优化二）：mask 生成在 CPU、与 GPU 前向 overlap；bitmask 以 int32 压缩位图传 GPU，一次 masked_fill_(-inf)。

    后端对比（背一句）：Outlines 是"正则→FSM 查表"，快但表达受限、编译慢；Guidance/llguidance 是"运行时解析"，灵活但每步要算；xgrammar 走中间路线——PDA 支持完整 CFG 又把 99% 判定预计算掉，既通用又快。 完整对比表见 03 第 4 节。

    3. 技术点：代码佐证与四层结构¶

    MindIE 交付的结构化输出是 vLLM 架构的迁移，四层结构（manager/grammar/bitmask/logits handler）：

    • MindIE-LLM/mindie_llm/text_generator/plugins/structured_output/structured_output_manager.py——总控：延迟导入 xgrammar、编译 grammar、matcher 缓存、生成 bitmask；
    • .../structured_output_grammar.py——XgrammarGrammar 封装 GrammarMatcher：逐 token accept_token、fill_next_token_bitmask、终止检测；
    • .../structured_output_bitmask.py——bitmask 应用到 logits（CPU/NPU 路径 masked_fill_ 置 −inf）；
    • MindIE-LLM/mindie_llm/text_generator/samplers/logits_handlers/pta_handlers.py——GuidedDecodingLogitsHandler（@register_class("guided_decoding")）在采样前挂载。

    vLLM 侧对照（vllm/vllm/v1/structured_output/）：StructuredOutputManager 单例把编译丢线程池、用 Future 非阻塞轮询把"编译未完成"变成调度状态；bitmask 在 scheduler（CPU）侧生成、worker（GPU）侧只 apply。进程边界、后端 auto 优先级（xgrammar > guidance > outlines）、投机解码 rollback、reasoning 跳过约束等实现细节见 03 第 3.5 节。

    4. 技术点：编译缓存（SHA-256 + FIFO/100）¶

    约束解码的开销分两段：

    1. 编译期（per-schema）：Schema→EBNF→PDA→mask cache 预计算，跑在 CPU，简单 schema 约 5–15ms、复杂 schema 约 100–200ms，直接加在 TTFT 上。
    2. 运行期（per-token）：查掩码缓存 + 检查少量 context-dependent token + apply bitmask，xgrammar 下通常 <1% 每步开销，且可与 GPU overlap。

    缓存设计（代码真相 vs vLLM）：

    ⚠ 代码默认 100 条 / FIFO（_DEFAULT_GRAMMAR_CACHE_SIZE=100，超限 next(iter) 删最早插入项，命中不调序）。简历若写 LRU/128，上场改口对齐代码。

    • MindIE（现行实现）：规范化 schema 串 SHA-256 为 key；默认容量 100；淘汰 FIFO → 相同 schema 二次请求零编译。业务 schema 近全热时 FIFO≈LRU，故选实现最简。
    • vLLM：缓存下沉给 xgr.GrammarCompiler(cache_enabled=True)，字节上限（默认 512MB）控制内存——自我迭代点：条数 + 字节双门限更稳。
    • 进一步：多实例 per-instance 缓存；**schema 亲和路由**可提高命中率——与 KV 亲和同构（见综合专题）。

    命中率经验值：tool-calling 场景默认 100 条约 85%–95%；长尾自定义 schema 可掉到 20% 以下。内存账见 03 第 5.5 节。

    5. 面试高频考点（快问快答）¶

    问题 核心答点
    结构化输出解决什么问题？ 把"prompt 大概率合法"升级为"采样阶段硬性保证合法"，屏蔽非法 token 的 logit 置 −inf
    支持哪些约束类型？ JSON Schema、正则、EBNF 语法、工具调用格式；tool call 是它的特化子集
    为什么用 PDA 不用 FSM？ JSON 递归结构、嵌套深度无界，FSM 表达不了，需带栈下推自动机；简单正则可退化为 FSM
    xgrammar 为什么快？ token 二分类：>99% context-independent 编译期预计算进 mask cache，<1% context-dependent 运行时持久化栈现场检查
    副作用有哪些？ TTFT 增加（编译耗时）、每步 mask 开销、强约束可能损害输出质量、投机/异步调度组合的正确性成本、batch 内干扰、内存占用（详见 03 第 6 节）
    编译缓存怎么做？ SHA-256 哈希规范化 schema 为 key + FIFO（默认 100 条）；命中不调序。vLLM 下沉给 xgrammar 按字节数（512MB）控制，更稳
    编译缓存和 KV 亲和什么关系？ 同构问题——schema 亲和路由（同 schema 进同实例）同时提高编译缓存与 KV prefix 命中率
    为什么开约束还要 tool parser？ 约束管 token 合法性（采样阶段），parser 管字段抽取与流式增量（解码阶段），职责正交（详见综合专题）
