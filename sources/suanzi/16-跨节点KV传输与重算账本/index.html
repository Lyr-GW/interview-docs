    16 · 跨节点 KV 传输 vs 重算（PD / 亲和账本）¶

    亲和命中后「少算 Prefill」见 14。
    本篇补另一半面试必问：PD 分离时，KV 是传过去还是在 Decode 节点重 Prefill？带宽账怎么粗算？
    简历锚点：Mooncake Conductor 全局索引、token 级 LPM、PD 分离/混部、tokenize 前置 Coordinator。

    诚实边界：调度与索引设计；不谎称写过 HCCL/Mooncake 传输 kernel。

    L0 · 三种「让 Decode 节点有 KV」的路径¶

    路径 做什么 何时划算
    A. 本地命中 同实例已有前缀 KV 混部 + 亲和路由成功（你的主故事）
    B. 跨节点传输 P→D 或实例间搬 KV blocks PD 分离；前缀在别的节点
    C. 重算 Prefill Decode 侧对 prompt 再跑一遍 PFA+FFN 传输极慢/不可用、前缀很短、或正确性回退

    金句：亲和调度优化的是「尽量走 A」；PD 分离引入「A 不够时在 B/C 之间权衡」。

    L1 · 粗算账：传 KV 还是重算¶

    记：

    • S = 需复用的前缀 token 数
    • L = 层数
    • D_kv = 每层每 token KV 字节（含 K+V，或 MLA latent）
    • B_net = 有效跨节点带宽
    • T_prefill(S) = 满量 Prefill 该前缀的墙钟时间

    则：

    T_xfer ≈ (S × L × D_kv) / B_net   (+ 序列化/排队)
T_recompute ≈ T_prefill(S)

    选 传输 若 T_xfer + T_setup << T_recompute；
    选 重算 若网络差、S 很小（Prefill 本来就快）、或 KV 布局不兼容必须重算。

    量级直觉（口算用，非标称 SLA）¶

    假设 Dense，D_kv 量级每层每 token 数百字节～数 KB（视头数/精度/量化）；L=60，S=4096：

    • 总 KV 体积可到 百 MB～数 GB 量级；
    • 若有效带宽仅数百 MB/s，传输可到 秒级 → 可能不如本地短 Prefill；
    • 若机内/RDMA GB/s 级，传输常优于重算长 Prefill。

    MLA：latent 更小 → B 路径更香（这是 MLA 对 PD 的隐性红利，可面试加分）。

    KV 量化：体积 ↓ → 传输更划算；与 05 联动。

    L1 · 和你简历故事的对齐¶

    简历点 账本位置
    tokenize 前置 Coordinator（~6ms） 降 Host 路径延迟；让 **token 级**匹配尽早发生，提高走 A 的概率
    Mooncake 全局索引 + token LPM 精确知道「哪台有哪段 KV」→ 选 A 或定向 B，避免字符级假命中导致 C
    规避 system/tools 注入 miss 假命中 → 错误 KV → 只能 C 或出坏结果
    PD 分离/混部双形态 混部偏 A；分离必须认真做 B 的索引与传输策略
    load_gated 保护 Decode 池不被「为亲和强行塞 Prefill」打爆 → 间接影响是否值得等 B

    TTFT −70% 主叙事仍是：高前缀重复场景下用 A 砍掉 compute-bound Prefill（13），不是「传输协议快了 70%」。

    L2 · 算子脚印：B 路径发生了什么¶

    P 节点: 已有 paged KV (block_table)
   → 按 block 打包 / 可能压缩
   → 网络发送
D 节点: 收包 → 写入本地 KV pool → 更新 block_table
   → 后续 Decode 走 IFA（读本地 cache）
    步骤 算子/组件 你是否主场
    读/写 paged KV scatter/gather、block_table 懂概念即可
    传输 Mooncake/RDMA/HCCL 等 调度侧选型与正确性，非协议作者
    Decode Attn IFA / FIA 消费方

    面试一句话：

    传输省的是 PFA + 大 M FFN；Decode 侧 IFA 仍要逐步跑。传错 block = Attention 数学输入错，和假前缀命中同类事故。

    L2 · PD 混部 vs 分离：对 A/B/C 的偏好¶

    混部 分离
    A 本地命中 易（同池） 难（KV 在 P 池）
    B 传输 少 主路径之一
    C 重算 回退 回退 / 短前缀
    算子争用 PFA 与 IFA 抢带宽 池内更纯，但多一跳

    挂简历：原生支持双形态 → 能说清「调度必须按部署形态改路由目标函数」，不是只调一个阈值。

    L3 · 追问与反模式¶

    Q：为什么不全部重算，省得传 KV？

    长前缀 Prefill 是计算密集、TTFT 大头；客户 4K 高重复场景，重算会吃掉亲和收益。短前缀或网络极差时重算可接受。

    Q：传 KV 能否 overlay 和下一个 Prefill？

    系统上可以 pipeline；算子层仍是「带宽 vs 算力」两本账。面试点到「重叠」即可，勿编造未做的实现。

    反模式（见 13）：

    • 「我优化了 FA，所以 TTFT −70%」
    • 「Mooncake 让 IFA 快了 50%」
    • 把字符级命中率当成功指标
