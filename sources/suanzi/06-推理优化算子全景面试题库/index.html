    06 · 推理优化算子全景面试题库¶

    覆盖「推理优化岗」对算子/硬件/图模式的常见追问。
    答题策略：框架视角切入 → 点到算子/硬件机制 → 诚实边界。
    详解见同目录其它文档；本题库以可背诵口述为主。

    使用说明¶

    • ★ = 高频必会；☆ = 加分；无标 = 应会。
    • 每题给「参考口述」；括号内指向深挖文档。

    A. 硬件与 Roofline¶

    A1 ★ Cube 和 Vector 有什么区别？¶

    Cube（AIC）专跑矩阵乘，最小粒度 fractal 常为 16×16×16；Vector（AIV）跑逐元/归约（softmax、norm、激活）。二者可流水并行。Decode 时若 M=1，Cube 有效利用率可低至约 1/16。

    A2 ★ 什么是算术强度？怎么判断 Prefill/Decode 瓶颈？¶

    OI = FLOPs / 从 HBM 搬的 Bytes。高于屋脊点（910B 大约二百出头）偏算力，低于偏带宽。Prefill 一份 KV 服务很长 Q，OI 高→计算密集；Decode 每步 Q≈1，OI 低→访存密集。（ops-Q&A.md）

    A3 ★ 大 batch 对 FFN 和 Attention 一样吗？¶

    不一样。FFN 共享权重，拼 M 能让单核 Cube 变饱。Attention 各请求 KV 不同，不能跨请求拼 M，主要靠多核并行；要拉 Attention 的 M 靠 GQA/MLA/MTP。（01-*.md）

    A4 屋脊点怎么算？¶

    峰值算力 ÷ 峰值带宽。用来判断算子落在 Roofline 的哪一段。

    B. 图模式与下发¶

    B1 ★ aclgraph 和 GE 区别？¶

    aclgraph：运行期 Capture&Replay，只省 Host 逐 kernel 下发；不融合、Device 负载基本不变。GE：编译期整图优化，融合/内存复用/多流，真降 Device 负载。可叠加。（算子与图编译学习笔记.md）

    B2 ★ 为什么 aclgraph 要静态 shape？Attention 怎么办？¶

    捕获时 tiling/地址冻结，重放不再跑 Host tiling。Attention 依赖每步 seq_lens/block_table，需 update_attn_params 一类 hook 打补丁。

    B3 ★ 什么场景选谁？¶

    Host-bound（小 shape、Decode、层多）→ aclgraph；Device 侧算力/访存/可融合多/显存紧 → GE。

    B4 CUDA Graph 和 ACL Graph？¶

    思想同为录制重放。约束与 API 不同；Paged 动态元数据都要特殊处理（piecewise / update）。

    C. FlashAttention / Softmax¶

    C1 ★ FA 为什么快？¶

    避免 O(N²) 的 S/P 进出 HBM：tiling + 块内融合 + online softmax，主要写回 O。

    C2 ★ Online Softmax 为何等价？（白板）¶

    维护 m,l,O；新块：m'=max(m,块max)，scale=exp(m-m')，l'=scale·l+块sum，O'=scale·O+块P·V，最后 O/l。

    C3 ★ Prefill/Decode 为何不同算子？¶

    PFA 切 Q、Cube 满；IFA 切 KV、可能 Vector 降级，并要跨核 reduce。

    C4 FA 反向为什么 recompute？¶

    前向不存 S/P 省显存；反向用统计量重算，算力换显存。

    C5 ☆ FA2/3/4？¶

    算法内核相近；硬件世代与 paged/FP8/调度元数据支持不同。昇腾对应 FAS/PFA/IFA/FIA 栈。

    D. PagedAttention / KV¶

    D1 ★ PagedAttention 解决什么？算子层谁做？¶

    KV 按 block 分页，提高显存利用率。写：scatter_pa_kv_cache；聚集：gather_pa_kv_cache。

    D2 ★ 和 Graph 冲突怎么解？¶

    动态 block_table vs 静态图 → piecewise / 更新 metadata / TaskUpdate。

    D3 KV 亲和调度和算子有何关系？（简历）¶

    亲和命中省的是 Prefill；Prefill 计算密集、贵，所以 TTFT 收益大。我做框架路由与全局索引，不是手写 FA。

    E. Linear / FFN / 融合¶

    E1 ★ Decode 为何像 GEMV？¶

    M≈1，N/K 很大，时间花在搬权重。

    E2 ★ QKV 合并、SwiGLU 融合图什么？¶

    少启动、少权重重复读、中间结果少落 HBM。

    E3 ★ 小算子为何融 Norm+RoPE+Cache？¶

    单独做税高；融合后中间留片上。本仓见 posembedding/qkv_rms_norm_rope_cache 等。（02-*.md）

    E4 ★ 图融合 / UB 融合 / SuperKernel？¶

    图融合改拓扑合并计算；UB 融合中间留 UB 省往返；SuperKernel 多子 kernel 一次启动省启动开销。

    F. MLA / GQA¶

    F1 ★ MLA 为何改善 Decode？¶

    KV 压到约 576 维 latent，显存与读带宽大降；absorb 下多 head 可共享 latent，BMM 的 M 可拼到 head 维。

    F2 ★ M=128 满载为何短上下文仍可能访存密集？¶

    还有 W_absorb 等大块常数搬运；要分清「fractal 利用率」和「整步 OI」。

    F3 GQA 相对 MHA？¶

    多 Q head 共享一组 KV，减 KV 体积，Decode 略抬 OI，实现简单。

    G. MoE / 并行 / MC2¶

    G1 ★ MoE 推理算子链路？¶

    gating/topk → routing/permute → grouped_matmul(+swiglu) → unpermute/finalize；跨卡再加 dispatch/combine。（04-*.md）

    G2 ★ Grouped MatMul 是什么？¶

    一次处理多组不等长 GEMM，适配「每专家 token 数不同」。

    G3 ★ matmul_all_reduce 图什么？¶

    TP Row-Parallel 的计算与 AllReduce 重叠，藏通信。

    G4 Prefill/Decode 并行策略差异？¶

    Prefill 适度 TP 降 TTFT；Decode 偏 DP/EP 扩吞吐，TP 过大通信占比高。

    G5 ☆ 你做过 HCCL 吗？¶

    未独立交付。理解 MC2 融合动机与调用位置；协议细节协作边界。

    H. 量化¶

    H1 ★ 为何 Decode 爱量化？¶

    访存密集，降位宽直接减搬运与显存，间接抬高 batch。

    H2 ★ per-token vs per-tensor？¶

    per-token 精度更好、开销更大，动态激活常用；per-tensor 更粗更便宜。

    H3 W 量化 vs KV 量化怎么选？¶

    看 profiling 瓶颈与上下文长度；用质量+吞吐验收。

    I. 工程与排查¶

    I1 ★ AscendC 算子目录结构？¶

    op_host（tiling）+ op_kernel + op_api(aclnn)；可选 op_graph。

    I2 ★ Decode 变慢怎么查？¶

    ① Host-bound？→ 图捕获。② Linear 搬权重？→ batch/量化/融合。③ Attn 读 KV？→ 长度/量化/MLA。④ 通信？→ MC2/并行策略。⑤ 小算子碎片？→ 融合/GE。用 profiling 验证。

    I3 ★ 没写过 AscendC，岗位匹配度？¶

    主战场框架/调度（结构化输出、KV 亲和、Tool Call）。算子层：Roofline + 读 ops-transformer/runtime，能选型与对齐调度。手写 kernel/HCCL 是补齐项，不伪装成已交付。

    I4 结构化输出 bitmask 算算子吗？（简历）¶

    是 NPU 上的 element-wise/向量操作组合；我打通 Schema→xgrammar→matcher→bitmask 全链路，底层调用现有能力。

    J. 综合场景题¶

    J0 ★ 先分清 Host-bound / memory-bound / compute-bound¶

    NPU 闲 + 下发间隙大 → Host → Graph；NPU 忙 + HBM 满 → memory → 量化/融/压 KV；算力满带宽有余 → compute → TP/减算。对照 08 §6。

    J1 ★ 给客户降 TPOT，你怎么系统性做？¶

    先定指标与 profiling 分层（Host / Kernel / 通信 / 显存）。Decode：攒 batch、CUDA/ACL Graph、融合、W/KV 量化、检查通信是否可 MC2、长上下文看 Attn。Prefill：chunk 大小、TP、FA 是否生效。避免无数据调参。

    J2 PD 分离的算子层依据？¶

    Prefill/Decode OI 与最优并行不同，混部易互相拖累；分离后各自选 TP/DP/EP 与量化策略。传 KV vs 重算见 16。

    J3 ☆ SpecDec/MTP 和算子的关系？¶

    一步多 token → 临时拉大 M，改善 Cube/OI；验证步像短 Prefill，拒绝要回滚 KV/seq_lens。框架与草稿质量是主战场。深挖 17。

    J4 LM Head / bitmask 插在哪？¶

    LM Head 出 logits（18）→ 可选 bitmask（15）→ sample。常图外；与中间层 aclgraph 可分段。

    K. 30 秒自我介绍（算子向收尾）¶

    我在昇腾 MindIE 做推理框架：结构化输出、KV 亲和调度、Tool Call 与 Server 重构。算子层我补齐了硬件 Roofline、FA/online softmax、PFA/IFA/MLA 选型、GE/aclgraph、MoE/MC2 与量化动机，能把调度决策对齐到「算力还是带宽、该融合还是该捕获」。AscendC/HCCL 手写不是我已交付范围，但我具备与算子团队协作并做性能归因的能力。

    L. 简历向题目（摘要）¶

    完整专项题库见 10-简历向追问题库.md。此处只留 5 道必背：

    1. ★ TTFT -70% 的算子解释？→ 少跑计算密集 Prefill，不是改 PFA kernel。
    2. ★ bitmask 在哪、是否主瓶颈？→ logits 后 Vector；通常不是。
    3. ★ 为何 token 级？→ 防假命中导致错 KV / 满量 Prefill。
    4. ★ load_gated？→ 保护 Decode 侧 IFA/FFN 不被挤爆。
    5. ★ 没写算子如何做优化？→ L5/L4 交付 + L3–L1 选型归因（见 12 分层）。
