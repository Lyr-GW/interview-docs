    09 · 简历 × 算子：挂钩地图与面试口述¶

    对象：林炜（MindIE 推理框架 / 结构化输出 / KV 亲和 / Tool Call）。
    目的：把简历每一条「框架交付」翻译成面试官能听懂的**算子/硬件语言**，并标清诚实边界。
    简历源：cvs/林炜-推理框架方向.pdf。
    答辩口径：编译缓存以代码为准——SHA256 + FIFO≈100、命中不调序；若 PDF 仍写 LRU，以 FIFO 为准（见 interview-review/08）。

    0. 30 秒定位（开场可背）¶

    我在昇腾 MindIE 做推理**框架与调度**：独立交付结构化输出全链路，做过多实例 KV 亲和路由（客户场景 TTFT -70%），以及 Tool Call 与 Server 重构。
    算子层不是我已交付的 AscendC/HCCL 手写范围，但我用 Roofline + 读 ops-transformer/runtime，能把调度决策对齐到「算力还是带宽、该融合还是该 Graph、Prefill/Decode 该用哪类算子」。
    下面可以按你关心的点展开：bitmask 在 NPU 上怎么跑、为什么前缀命中能降 TTFT、PD 分离的算子依据等。

    1. 简历条目 → 算子/硬件挂钩总表¶

    简历点 框架层你做了什么 算子/硬件层「为什么有效」 你应主动说的边界
    结构化输出 Schema→xgrammar→Matcher→bitmask 全链路设计、编译缓存、异步错位 bug bitmask 是 Vector 侧 element-wise；每步改 logits；宜与采样同侧、少 HBM 往返；常在 Graph 外（动态） 未手写 bitmask AscendC kernel，调用/组合现有能力
    编译缓存 SHA256+FIFO（默认容量约 100，命中不调序；勿说 LRU/128） 重复 Schema 编译开销→0 类比 Host 侧「常量折叠/缓存」：省的是 CPU 编译，不是 Device FLOPs；和 aclgraph 冻 tiling 是不同层 这是 Host/框架优化；口径对齐 interview-review/08
    mask/采样步错位 异步调度下修正时序 Decode 一步 = 多 kernel 流水；mask 必须对齐「本步 logits」；错位=用了过期合法集 根因在调度时序，不是 FA 算错
    KV 亲和 tokenize 前置 + token 级前缀匹配 Coordinator 路由到持 cache 实例 省的是 Prefill；Prefill 是 计算密集（OI∝S_q），重算贵 → TTFT 收益大 未写 FA/Paged 算子；懂 PFA/scatter 职责即可
    4K tokenize ≈6ms vs 省数百 ms Prefill 用数据证明前置划算 数量级对比：Host 轻量 vs Device Prefill（PFA+Linear） 用 Roofline 解释「为什么 Prefill 贵」
    vs 字符级 Router token 级+全局索引防 miss 字符/token 不一致 → 假前缀 → 仍走满量 Prefill 算子 属于正确性→性能
    unified / load_gated 前缀收益与负载权衡 纯追前缀可能把请求打到过热实例 → Decode 侧 **batch/带宽**变差，吞吐掉 调度要同时看 Prefill 节省与 Decode 拥塞
    PD 分离/混部 双形态原生支持 Prefill 算力密集 vs Decode 访存密集 → 算子与并行策略不同 能讲 OI 依据即可
    TTFT -70% / E2E -50% 客户场景验证 指标拆解：TTFT≈Prefill；E2E 还含 Decode；高前缀率时 Prefill 占比大 避免夸成「算子优化了 70%」
    Tool Call / Reasoning 解析多模型族 输出侧后处理；与 structured output 同属「约束/解析」；一般不进 FA 主路径 非算子交付
    Server 重构 -1 万行 抽象请求链路 降低后续接融合/Graph/采样扩展的成本 可维护性为主，非直接加速
    实习 Benchmark 可视化 多维指标对比 对应 TTFT/TPOT/吞吐/算子占比的观测习惯 铺垫 profiling 口述

    2. 结构化输出 × 算子（深挖）¶

    2.1 一步 Decode 里 bitmask 插在哪¶

    … → Transformer 层（Linear/Attn/FFN 算子）→ LM Head
    → logits[V]
    → 【bitmask 屏蔽非法 token】← 你的特性插入点（Vector）
    → Sampler（temperature/top-p/…）
    → 新 token → GrammarMatcher 更新合法集

    要点：

    1. 主计算**仍是 MatMul/FA；bitmask 相对便宜，但**每步都有，且必须正确。
    2. 非法位置置 -inf（或极小值）再 softmax/采样，保证语法约束。
    3. 与 aclgraph：Sampler + mask 常因动态控制流留在图外；整图捕获时要分清「可捕获段」与「不可捕获段」。
    4. 与 异步调度：mask 必须对应「即将采样的那一步 logits」。你修的错位 bug = 调度层时序与算子步对齐问题。

    2.2 编译缓存 vs Device 优化（别混层）¶

    优化 省什么 类比
    Schema→grammar 编译缓存 Host CPU 时间 类似「常量折叠结果缓存」
    aclgraph Host 下发 Capture&Replay
    FA/融合 Device 访存/计算 算子层

    面试官若问「你做的算不算算子优化」：

    算**推理正确性与 Host 路径优化**；Device 侧是正确调用 bitmask/采样算子组合。整体 TTFT/吞吐还依赖 Prefill/Decode 主算子，我用调度（亲和）去减 Prefill。

    2.3 追问树（结构化输出）¶

    Q: bitmask 在 GPU/NPU 怎么实现？
A: element-wise；昇腾走 Vector；可与采样融合减往返。我侧是框架编排。

Q: 词表很大时 bitmask 会不会成瓶颈？
A: 通常远小于 Linear/Attn；若 Host 生成 mask 慢，要看 mask 下发与异步重叠。应用 profiling。

Q: 和 xgrammar/guidance 关系？
A: 后端可切换；编译产物喂 Matcher；缓存 key=SHA256(schema)。

Q: 为什么异步会错位？
A: 连续批处理下一请求多步流水，mask 生命周期必须绑「batch 内该 slot 的当前步」。

    3. KV 亲和 × 算子（深挖）¶

    3.1 收益链路（必须讲清因果）¶

    前缀命中
  → 跳过/缩短 Prefill（少跑 PFA + 多层 Linear）
  → TTFT 下降
  → 若前缀率高，E2E 也降

    算子层依据：

    • Prefill：OI ∝ S_q，计算密集，重算一整段 prompt 很贵；
    • 你测的 tokenize 6ms ≪ Prefill 数百 ms → Host 前置完全划算；
    • 命中后实例上仍要跑 Decode（IFA + FFN GEMV…），所以要 load_gated，避免把 Decode 打爆。

    3.2 和 PagedAttention / 前缀缓存的分工¶

    机制 层 作用
    PagedAttention 单实例内 KV 分页 显存利用率；scatter_pa_*
    实例内 prefix cache（如 Radix） 单实例 同机复用前缀
    你的 KV 亲和 多实例路由 把请求送到「已有前缀」的实例

    三者正交：亲和解决「去哪台」；Paged/prefix 解决「机内怎么存/复用」。

    3.3 为什么坚持 token 级匹配（算子后果）¶

    字符级匹配在 tools/system prompt 注入后可能 假命中 → 路由到「看似有前缀、实际 token 对不上」的实例 → 仍满量 Prefill 或错误复用。
    错误复用 KV 比 miss 更糟（正确性）。token 级 + 全局索引是为「Prefill 算子输入正确」。

    3.4 追问树（KV 亲和）¶

    Q: TTFT -70% 是不是算子写快了？
A: 不是。是少做 Prefill。Prefill 本身是 PFA+Linear 的计算密集段。

Q: 若 Decode 已是瓶颈，亲和还有用吗？
A: 对 TTFT 仍有用；对吞吐取决于 Prefill 占比。要看阶段拆分。

Q: PD 分离下亲和怎么做？
A: Prefill 池与 Decode 池索引与路由策略不同；我设计支持两种部署。算子上 P 侧吃 PFA，D 侧吃 IFA。

Q: 和 Mooncake 关系？
A: Conductor 提供全局 KV 索引；我做匹配与调度策略。传输/存储细节可指向 Mooncake 专题，非我 AscendC 交付。

    4. Tool Call / Server 重构 × 算子¶

    点 挂钩说法
    Tool Call 解析 输出侧；可能与 structured output 组合（JSON 工具参数）；不占 FA 主路径算力
    DeepSeek V3 模型侧有 MLA/MoE → 面试可转「我熟悉服务侧解析；模型算子侧我了解 MLA/GMM/MC2 选型」
    Server 重构 请求链路清晰后，更容易安全地接 Graph 分段、采样钩子、约束解码

    5. 科研背景怎么轻量挂钩（可选）¶

    硕士 CNN-Transformer / ViT：说明懂 Attention 结构与计算量直觉，**不要**说成写过昇腾 FA。
    一句话：训练/视觉里的 Attention 与推理 FA 同构，推理额外有 KV cache、Paged、Decode M=1 等约束。

    7. 禁止说法 / 推荐说法¶

    禁止 推荐

    「我写了 NPU bitmask 算子」 「我打通到 NPU 侧 bitmask 屏蔽，kernel 用现有能力」
    「TTFT -70% 因为算子更快」 「高前缀场景少做计算密集的 Prefill」
